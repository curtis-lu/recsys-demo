  ├────────────────┼────────────────────────────┼───────────────────────────┤
  │ 執行方式       │ bash scenarios/run_all.sh  │ pytest tests/scenarios/   │
  │                │                            │ -v                        │
  └────────────────┴────────────────────────────┴───────────────────────────┘

  建議用 pytest：
  - 自動化 assertions（檢查行數、欄位、snap_dates 等）確保 pass/fail 明確
  - 同時把 pipeline 產出（parquet 檔）和驗證報告寫到一個固定目錄，供人工事後檢視
  - 和現有 tests/ 架構一致，用 pytest tests/scenarios/ 單獨執行

  獨立性

  你說得對，情境測試的設定和資料應該獨立，不要混進 conf/ 和 data/。修正方向：

  tests/
    scenarios/
      conftest.py                     # 共用 fixture（產生資料、建立工作目錄）
      data_generator.py               # 擴展合成資料產生器
      test_scenario_1_new_inference.py
      test_scenario_2_shift_window.py
      test_scenario_3_new_features.py
      test_scenario_4_new_products.py
      output/                         # 人工檢視用產出（.gitignore）
        scenario_1/
          report.txt
          ranked_predictions.parquet
        ...

  每個 test 透過 pytest fixture：
  1. 在 tmp_path 或 tests/scenarios/output/ 建立隔離的工作目錄
  2. 用 fixture 產生情境資料（feature_table / label_table）
  3. 用 fixture 在工作目錄下組裝 conf/base/ + 情境覆蓋設定（不碰專案的 conf/）
  4. 呼叫 pipeline 函式（或 subprocess 呼叫 CLI）
  5. Assert + 產出報告

  這樣的好處：
  - 專案 conf/ 保持乾淨，不會出現 scenario_1/ 等測試用設定
  - data/ 不受影響，所有測試資料在 fixture 中產生
  - 可重複執行，fixture 處理 setup/teardown
  - 人工檢視仍可行，把關鍵產出複製到 tests/scenarios/output/

  我會更新計畫反映這個方向。
