指標說明：mAP / NDCG / per-item

每個指標程式實際怎麼算（對照 src/recsys_tfb/evaluation/metrics.py 與 metrics_spark.py 的公式）、輸出格式、報表怎麼分段、常見疑問。排序概念的數學自學見手冊 gbdt_learning_to_rank.md。

評估永遠在 query group 內進行：query group ＝ 同一個 (time, entity) 下所有候選 item。模型分數只在組內比大小、指標也是每組各算、再平均。這和「逐筆算 AUC」的二元分類根本不同（見 README §0）。

0. 三層計算（一張圖看懂資料怎麼被縮約）

原始預測（每筆 = 一個 entity×item 的 score） │ rank_within_query：組內按 score 由高到低給名次 pos │ add_query_total_rel：組內正例數 total_rel = Σ label ▼ 逐列貢獻（add_row_contributions） cum_rel = 到此名次累計正例 prec_at_pos = cum_rel / pos dcg_term = label / log2(pos+1) top_k@K = 1 if pos≤K else 0 ▼ groupBy(query) 每組指標（compute_per_query_metrics）：map@K · ndcg@K · precision@K · recall@K ▼ mean over groups 總表指標（summary） ▼ attribute 到 item，macro over items per-item 指標：hit_rate@K · map_attr@K · ndcg_attr@K · mean_pos

1. 主指標：mAP

每個 query group 先算 AP（Average Precision），再對所有「至少有一個正例」的 group 取平均，就是 mAP。這是頭號指標。

AP（單一 query group）— compute_ap

order = argsort(-score) # 組內按 score 由高到低 prec_at_pos = cumsum(label) / pos # 到此名次的精確率 AP = Σ( prec_at_pos × label ) / Σ label

直覺：只在「命中正例的名次」上取精確率、再平均。正例都排在前面 → AP 接近 1；排在後面 → AP 低。

zero-positive group 回傳 None、不計入 mAP：一組裡完全沒有正例時 AP 沒有意義（分母 0）。這也是 dataset 的 filter_groups_with_positives 只對 val/test 丟掉這種組的原因（見 dataset.md）。

mAP（跨 group）— compute_mean_ap

mAP = mean( AP_g for g in groups if AP_g is not None )

同分（score 相同）依輸入穩定順序決定名次（np.lexsort）。空輸入或全無正例 → 0.0。

2. @K 指標（top-K 截斷）

把名次截在前 K，算 top-K 版本。每組先算、再平均。逐組公式（compute_per_query_metrics）：

指標 @K每組公式意思
map@KΣ(prec_at_pos · label · [pos≤K]) / total_rel只看前 K 名的 AP
ndcg@KΣ( dcg_term · [pos≤K] ) / iDCG@K有位置折扣的排序品質；已被理想排序正規化到 [0,1]
precision@Khits@K / K前 K 名裡有多少比例是正例
recall@Khits@K / total_rel該組正例有多少比例被排進前 K

其中 dcg_term = label / log2(pos+1)；iDCG@K = Σ_{i=1..min(total_rel,K)} 1/log2(i+1)（理想排序的 DCG，把正例全擺最前面）；hits@K = Σ(label·[pos≤K])。總表指標是這些每組值的平均。

K 的退化情況：當 K ≥ 每組候選 item 數 時，前 K 名涵蓋全組 → precision@K 趨近「正例佔比」、recall@K 恆為 1，兩者失去鑑別力。看 @K 指標時要對照每組候選數。

3. per-item 指標（macro over items）

把每個正例的貢獻歸因到它的 item，再對 item 做 macro 平均（每個 item 等權）。讓你看「哪些 item 排得好 / 差」，不被熱門 item 的數量蓋過。

指標意思
map_attr@Kper-item 歸因的 mAP@K（該 item 的正例貢獻平均）
ndcg_attr@Kper-item 歸因的 NDCG@K
hit_rate@K該 item 的邊際 recall：它的正例有多少比例被排進前 K
mean_pos該 item 在 P-positive 列上的平均名次

聚合採「row-equal-weight：對 P-positive 列取等權平均」（aggregate_per_item）。

HPO 用哪個指標選超參？ training 的 tune_hyperparameters 在 val 上算的分數由 hpo_objective 決定。本 repo 的 config（parameters_training.yaml）設成 macro per-item mAP（compute_macro_per_item_map，即這節的 per-item 巨觀平均，避免熱門 item 主導調參）；config 沒設時程式碼預設回退到一般 mean_ap。見 training.md。

4. baseline 對照

compute_baseline_metrics 算一個 popularity baseline：用「近期最常被選的 item」（逐 snap_date 在回溯窗內統計人氣）當不看個人特徵的基準排序，套同一組指標。模型要有價值，得明顯贏過它。報表會把模型與 baseline 並列。

5. 報表怎麼分段

分段內容
dataset overview評估資料規模（group 數、正例率等，compute_dataset_overview）
summary總表 mAP ＋ 各 K 的 map@K / ndcg@K / precision@K / recall@K，與 baseline 並列
per-item各 item 的 map_attr@K / ndcg_attr@K / hit_rate@K / mean_pos
per-segment依 segment_columns 分群的指標（需 segment_sources 提供欄；否則該段不出現，見 README §4）
comparison（--compare）與另一 model_version / 外部表在共同客戶上的對比（見 evaluation.md）

可選：product_categories 設定可把 item 收合成類別後再算指標（collapse_to_categories）。

6. 常見疑問

疑問解答
為什麼不看 AUC / logloss？那是逐筆的分類指標；這裡要的是「組內相對名次排得好不好」，所以用 per-group 排序指標。
mAP 和 map@K 差在哪？mAP 不截斷（看完整排序）；map@K 只看前 K 名。報表兩者都有。
同分怎麼排？依輸入穩定順序（np.lexsort / Spark window），不是隨機。
某 item 指標是 0 或缺？該 item 在評估窗沒有正例 → 不貢獻 per-item 平均（同 zero-positive group 的道理）。
