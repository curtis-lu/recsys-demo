# 《Spark 優化參考手冊》Round-2 全書重審 — 設計（spec）

> **定位**：這是手冊**初版完成後的第二輪整體重審**設計文件，不是新手冊。原始設計 spec＝`docs/superpowers/specs/2026-06-14-spark-tuning-handbook-design.md`（仍為各章內容的基準）；本 spec 只規範「**這一輪要怎麼重審、依哪些決策重構**」，是 round-2 修改計畫（`docs/superpowers/plans/2026-06-22-spark-handbook-round2-revision-plan.md`）的單一真實來源。**結構方向若再調整，先改本 spec。**

**Goal**：對已成稿的 8 章（01–06、08、09）＋index 做一次**從整體架構出發、非單點式**的重審與修訂，沿五個面向（架構邏輯／易讀性／技術正確性／實用性／知識深度）拉齊，補上被識別出的能力缺口，並把暫緩/未寫的三章（PySpark、場景索引、速查）依新骨架補完。

**本輪授權**：完整重構——可增/刪/合併/拆分/重排章節。結構變更一律先落本 spec，再改章。

## 1. 讀者模型（雙職涯兩條主線）— 本輪所有「實用性/深度」判準的基準

> 取代原 spec 較籠統的「SQL-first 分析師」單一設定。兩條線**共用 01–06 優化基礎**，在營運線分流。

- **主線①　初階資料分析師**：用 **Hue/Impala 做 ad-hoc 分析**；用 **Spark 做定期資料排程「出行銷名單給 PM」**；**建模時產出特徵表**。
- **主線②　進階 analytics engineer**：營運**共用特徵庫（feature store）**等共用資料集，**供應給模型訓練**；以及 **reverse ETL 把資料送回業務端的「客戶行銷及管理系統」**。

**終極能力目標**（沿用原 spec §11）：讀完整本能**長期穩定營運資料排程與特徵庫**。**兩條主線的共同終點＝把算好的資料送出去**（初階→名單給 PM；進階→reverse ETL 回業務系統）。

## 2. 審稿框架：5 種 subagent 角色

> 五個面向 → 角色對應。**技術正確性與 persona 無關**（事實對錯不分讀者）；易讀性/實用性走雙 persona；架構與教學是兩個獨立的全書視角。

| # | 角色 | 面向 | 視角與限制 | DoD（要產出） |
|---|---|---|---|---|
| R1 | **Technical reviewer** | 技術正確性 | 版本釘死 **Spark 3.3.2 / Hive 3.1.3 / CDP 7.1.9**；**只認官方來源**（Spark 官方文件、Cloudera CDP 官方文件、Airflow/dbt 官方文件、指定三本書）；不評易讀性。 | 逐條主張 ✅已驗證(附出處)/❌錯誤(正確值+出處)/⚠️無法查證；點「建議被寫成硬限制」「因果正負號」「引用錯」；三級彙整。 |
| R2 | **Reader-初階** | 易讀性＋實用性〔初階線〕 | persona＝會 SQL、無 DE/分散式背景；做 Hue/Impala ad-hoc＋Spark 排程出名單＋產特徵表。問「**讀得懂嗎＋我這份工作做得了嗎**」。術語第一次出現沒當場定義就標。 | 逐節卡關點＋類型；「我會這樣想/我會問什麼」；**初階 end-to-end（探索→排程→出名單）能不能照著走**；三級彙整。 |
| R3 | **Reader-進階 AE** | 知識深度＋實用性〔進階線〕 | persona＝analytics engineer；營運特徵庫供訓練＋reverse ETL 回業務系統。問「**深度夠不夠營運＋我的進階工作有沒有覆蓋**」。 | 逐章深度是否撐得起營運；進階能力缺口（training–serving 一致、出口推送、多租戶、版本/發佈）；三級彙整。 |
| C  | **Architecture / Completeness** | 架構邏輯性＋覆蓋度＋深度（全書）**＋跨章一致性** | 跨整本；不查單點技術對錯、不挑逐句易讀性。**吸收跨章一致性**：術語表一致、交叉引用對得上、notation（欄式 not 列式）、config 預設值跨章不打架、footer 體例。 | 能力地圖（兩主線→哪章支撐→缺口）；章序/依賴；逐章深度；結構提案；一致性掃描；三級彙整。 |
| P  | **Pedagogy（教學設計）** | 教學法 | 把手冊當教學作品；對標 **三本名著編排 ＋ Diátaxis 四象限 ＋ Databricks 漸進式**；不查技術對錯、不評覆蓋度。**僅用於教學法對標，不放寬技術來源規則**。 | 每章 Diátaxis 分類＋模式混淆；概念弧（直覺→機制→取捨→落地數字）；worked-example 密度；章首/章末/footer 一致性；跨章敘事弧；三級彙整。 |

**socket-death 對策**（既有教訓：subagent 長跑 >13 分易斷線）：每次 dispatch **窄範圍**（單章×單角色，或全書但只走結構/抽樣），**邊做邊即時 append live log** 到 `.reviews/`，斷線可讀 partial log 接續或自審補。

### 4.1 使用者追加方向（2026-06-23）— ch02 升級為「操作核心」

> 與 §4 的 Phase1 自動發現區隔：這是使用者主動指定的**深化**（非缺陷修補）。Phase1 就現狀評 ch02「剛好、教學乾淨」，使用者選擇**主動加大深度**，把 ch02 做成全書「先量再調」的手把手操作樞紐。

- **D-UI（深化·必做）— ch02 改寫為手把手診斷指南**：大量增補細節，做到「**畫面顯示什麼 → 對應什麼問題 → 該查哪一格 → 去哪一章修**」。要件：
  1. **逐頁籤 screen-by-screen 導覽**（Jobs / Stages / SQL / Executors / Storage / Environment），每頁籤用**標註過的 mock 面板**示意，講「這格數字代表什麼、異常長什麼樣」。
  2. **症狀 → 面板 → 章** 路由表升級（shuffle 過大/skew/spill/掃太多/小檔/廣播失敗…→ 在哪個面板哪一格看出來 → 翻 §03/§04/§05 哪節修）。
  3. **三張 checklist 檢核表**，對齊兩條主線的工作型態：**讀 UI 改 SQL**（→ §03）、**讀 UI 改 Spark 設定**（→ §04）、**讀 UI 改寫表/儲存邏輯**（→ §05）。
  4. **邊界**：ch02 只**診斷與路由**，**不重教** §03/§04/§05 的修法（避免重複、守住「不可單點式」的整體性）。
- **代價/取捨**：ch02 會明顯變長 → 用清楚的 `##`（每頁籤一節）＋章末總 checklist 控制密度，避免變成單節做太多事（呼應 Pedagogy f-5 對 §5.5 的同款提醒）。HTML 階段 mock 面板可換公司真實 History Server 截圖。

## 6. 來源與限制

- **技術事實**：只認官方（Spark 3.3 官方文件、Cloudera CDP 官方文件、Apache Airflow / dbt / Iceberg 官方文件、《Learning Spark 2nd》《Spark: The Definitive Guide》《High Performance Spark》）。對齊 3.3.2 / Hive 3.1.3 / CDP 7.1.9，不套別版預設值。連結用可達頁（version-exact 404 → 用 `/docs/latest/` 或 `docs-archive` 並註明）。
- **教學法對標**（僅 Pedagogy 角色）：三本名著編排 ＋ Diátaxis（diataxis.fr）＋ Databricks 學習資源。**不放寬技術來源規則**。
- **reverse ETL 章（新 09）的 Step A 要特別查證**：CDP 上把資料推出 Hadoop 的官方途徑與限制（JDBC/connector、檔案交換、Sqoop 在 CDP 的狀態、對接外部 API），**不臆測**；查不到標「無法查證」並用保守說法。
