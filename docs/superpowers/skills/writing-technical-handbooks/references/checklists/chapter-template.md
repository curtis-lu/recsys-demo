# 章節骨架模板

每章依此順序組裝，不得跳過、不得亂序。每個結構件後附一句說明其作用。

---

## 骨架順序

### 1. 本章前提（Prerequisites）

列出「讀這章之前讀者必須已知的概念」，一條一條，不寫廢話。  
作用：讓讀者知道自己能不能直接讀、還是得先補什麼。缺這塊讀者會讀到一半才發現卡關。

範例格式：
```
**前提**：讀者應已理解 X 的基本定義、知道 Y 的運作方式；無需事先了解 Z（本章會從頭帶）。
```

---

### 2. 本章目錄（帶 GitHub 錨點連結）

條列本章所有小節，用 Markdown 錨點連結（`[§x.y 小節名稱](#xy-小節名稱-slug)`）。  
作用：讓讀者一眼掃完結構、可直接跳到感興趣的段落；也作為作者的完整性自查清單。

> 錨點規則：標題全小寫、空格換 `-`、去掉特殊符號；跨 repo 或尚未撰寫的章節用「軟指標」（見骨架末尾說明），**不要下 `[連結文字](url)` 形式的硬連結**，因為它一旦失效或章節未完稿就會 404。

---

### 3. 內文（§x.y 小節）

一節一個概念。先給直覺（動機），再給形式化（公式、定義），再解讀。  
作用：把「為什麼需要這個量」放在式子之前，讓讀者有接收框架再看細節。某節過長代表它在做兩件事，需要拆開。

節標題格式：`## §x.y 小節名稱`

---

### 4. Capstone 貫穿範例

用一組具體數字把本章核心機制跑一遍。這組數字應與前一章（或後一章）共用，形成全書的貫穿主軸。  
作用：讓讀者親眼看到因果——同一組輸入，進不同機制，出不同結果。  
格式要求：
- 數字要可手算、可複算，關鍵步驟標 `✓`。
- 若換了新場景，明確說「這裡沿用第 X 章的同一組數字」或「這裡重建一個新範例，角色延續」。

---

### 5. 取捨（誠實講限制與代價）

列出本章介紹的方法在哪些情況下不適用、有哪些成本或上限。  
作用：防止讀者把一個適用範圍有限的方法當成萬靈丹。結論要對得起前面的診斷，不能輕描淡寫成「調一下就好」。

格式：
- **適用前提**：A、B、C 條件同時成立時效果最好。
- **限制**：當 D 時此方法失效，原因是……
- **代價**：多出 O(N) 計算量 / 需要額外標注資料 / 引入額外旋鈕。

---

### 6. 一句話帶走

一句話總結本章讀者應帶走的核心結論。必須對得起前面所有內容，不可過度簡化。  
作用：幫讀者建立長期記憶；也是作者的自檢——若這句話和內文矛盾就要回去修。

格式：`> **帶走**：[一句話結論]`

---

### 7. 上下章導覽

固定格式，放在章末：
```
---
← 上一章：[第 X 章：章名](../chX.md)　｜　下一章：[第 X+2 章：章名](../chX2.md) →
```
作用：讓讀者在閱讀流程中保持方向感；也讓作者在調整章序時快速找到需要更新的連結。

> 尚未撰寫的鄰章用**軟指標**（見下方說明），不下硬連結。

---

### 8. 來源 Footer（📚）

每個小節或每章末尾，標出引用的論文、書目、工具文件：
```
📚 來源：作者姓名（年份）〈論文或書名〉；工具名稱官方文件（版本號或存取日期）。
```
作用：讓讀者知道哪些主張有文獻支撐、哪裡可以深入。作者名稱與年份必須核對原始來源，不可靠記憶填寫。

---

## 軟指標說明（缺章的 forward-reference 作法）

當引用的章節**尚未撰寫**或**位於外部文件**時，用一句白話說明概念所在，不下 Markdown 連結。

正確作法（軟指標）：
```
關於 Y 機制的完整推導，見後續討論校準誤差的章節。
```

錯誤作法（硬連結，會 404 或因章序調整失效）：
```
關於 Y 機制，見 [第 9 章](../ch9.md#section-y)。
```

規則：只要目標章節還沒定稿、或 URL 可能因重構改變，一律用軟指標。等那章確定存在後再改成硬連結。
