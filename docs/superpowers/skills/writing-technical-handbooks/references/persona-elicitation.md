# 讀者 Persona 引導（persona-elicitation）

> **這是最高槓桿的一步。** persona 一模糊，整本手冊的深度、該 gloss 什麼術語、capstone 的真實度全跟著偏。
> 產出物（一段 persona spec）會**注入每個 reviewer prompt**，R2／R3／Follower 尤其依此扮演。寧可在這裡多花十分鐘，不要在五輪審稿後才發現「假設的讀者程度根本錯了」。

引導目標只有一個：把「讀者是誰」從**形容詞**（中階、有點底子、不算新手）逼成**具體錨點**（他現在已經能完成的一件具體事）。形容詞不可審；具體錨點才能讓 reviewer 判斷某句話讀不讀得懂。

---

## 1. 六維具體錨點表

對每位讀者，沿以下六維各釘一個具體錨點。**判準：每一格都要能被一個「不熟這位讀者的第三人」讀懂並據以判斷。** 出現「中階／有基礎／熟悉」這類形容詞，就是還沒到位，繼續逼。

| 維度 | 不要這樣寫（形容詞，不可審） | 要這樣寫（具體錨點，可審） |
|---|---|---|
| **背景知識（地板／天花板）** | 「SQL 中階」 | 「**能**寫多表 JOIN、看得懂 `EXPLAIN`，**但沒聽過** shuffle」 |
| **閱讀目的 / job-to-be-done** | 「想學 Spark」 | 「讓每週批次別再 OOM」 |
| **深度期待（承重水位）** | 「想懂得深一點」 | 「要會調**也**要懂為什麼，但不需讀原始碼」 |
| **工具 / 操作介面** | 「會用一些工具」 | 「在 Hue 寫 SQL、偶爾 `spark-submit`；不寫 Scala」 |
| **環境 / 約束** | 「在公司叢集上跑」 | 「CDP 7.1.9、禁 UDF、無外網」 |
| **反例（不是寫給誰）** | （常被略過） | 「不是寫給已在 prod 跑 Spark 的資料工程師」 |

幾個維度的細節：

- **背景知識同時要地板與天花板。** 地板 = 他已經穩穩會的（手冊可直接站在上面、不必重教）；天花板 = 他明確還不會的（手冊要從這裡開始補）。只有兩端都釘住，reviewer 才知道哪句該 gloss、哪句是廢話。範例的力道全在那個轉折：「能 JOIN、看得懂 EXPLAIN」**但**「沒聽過 shuffle」——這一句就同時劃出了地板與天花板。
- **閱讀目的要寫成他想完成的任務，不是他想學的學科。** 「讓批次別再 OOM」會驅動完全不同的章節取捨與範例選擇；「學 Spark」什麼都涵蓋＝什麼都不聚焦。
- **深度期待＝承重水位**：手冊的解釋要承載到多深才算夠、再深就是過載。「要懂為什麼但不需讀原始碼」清楚劃出了上界。
- **反例（不是寫給誰）最常被跳過，但它消除歧義最有效。** 「不是寫給已在 prod 跑 Spark 的人」一句，就把一整類「對老手太淺、對新手剛好」的取捨爭議直接判決掉。**務必逼出至少一個反例。**

## 2. 環境／約束維度的歸宿（重要原則）

環境限制（如 `No UDF`、無外網、CDP 7.1.9、CPU-only、不可裝額外套件）的**正當歸宿是 persona，不是手冊的普世規則**。

- ✅ 寫在 persona：「這位讀者的環境禁 UDF」→ 手冊在這位讀者的脈絡下避開 UDF，理由清楚、可被 reviewer 對照。
- ❌ 寫成普世規則：手冊正文出現「Spark 不應使用 UDF」這種無條件斷言 → 這是**脈絡洩漏**：把某一個環境的約束偽裝成普遍真理，誤導其他環境的讀者，也讓技術正確性審稿（R1）抓到把柄。

換句話說：約束是「**這位讀者**面對的限制」，不是「**這個技術**的本質」。把它放進 persona，正文才能用「在你的環境（禁 UDF）下，因此……」這種有條件、有出處的語氣表達。

## 3. 引導腳本（複用 brainstorming，一次一問）

沿用 brainstorming 的節奏：**一次只問一個問題**，等使用者答完再進下一題，不要拋一張六維問卷讓他一次填完（會退回形容詞）。

### 步驟 ① 先要一個真實 proxy 人物

不要問「你的讀者程度如何」。問：

> 「想一個**真實存在的人**當這本手冊的代表讀者——某位同事、某位你帶過的人，或者**N 個月前還不會這主題的你自己**。是誰？」

真人比抽象 persona 好推理太多：使用者對真人有完整、具體、可回憶的印象，逼錨點時不會憑空捏。**這個 proxy 人物法本身就是第一個減壓閥**（見 §5）。

### 步驟 ② 就那個真人逐維逼出具體錨點

抓住那個真人，逐維提問。關鍵句型——**問「他能完成的一個任務是？」，不要問「他程度如何」**：

| 維度 | 這樣問（逼出任務／錨點） | 不要這樣問 |
|---|---|---|
| 背景知識 | 「他**昨天就能獨力完成**的一件相關任務是什麼？反過來，哪個詞講出來他會愣住？」 | 「他底子如何？」 |
| 閱讀目的 | 「他翻開這本手冊，**想解決的具體麻煩**是什麼？」 | 「他想學什麼？」 |
| 深度期待 | 「他會滿足於『照做就好』，還是會追問『為什麼』？追到原始碼嗎？」 | 「他想學多深？」 |
| 工具介面 | 「他平常**在哪個介面**敲指令？哪些語言／工具他不碰？」 | 「他會用哪些工具？」 |
| 環境約束 | 「他的環境有哪些**硬限制**（版本、禁用功能、隔離網路）？」 | 「他在什麼環境？」 |
| 反例 | 「哪種人讀這本會嫌太淺或太囉嗦？**那種人我們不寫。**」 | （別略過此題） |

逐題把答案收斂成一句具體錨點，對齊 §1 右欄的寫法。任何一格還是形容詞，就回頭追問「舉一個他能／不能做的具體例子」。

### 步驟 ③ 問有無第二讀者

> 「除了這位，還有沒有一類**明顯更進階或更初階**的人也會讀？」

- 有 → 收 **2 個** persona：一個**地板**（P1，最沒底子、決定 gloss 下界與前置補課量）＋一個 **stretch**（P2，最進階、決定哪裡可以再深、避免對老手過度囉嗦）。
- **上限 2–3 個（YAGNI）**。超過就是在替不存在的讀者寫書，徒增審稿維度、稀釋聚焦。多數手冊一或兩個 persona 足矣。

## 4. 非專家 fallback（核心手法）

**情境**：skill 使用者自己**也不熟這個主題**，於是答不出「讀者懂不懂 X」——因為他連 X 是什麼、X 在知識體系的哪一層都不確定。這時硬問六維只會得到含糊答案。

**手法：翻轉負擔。** 不要求使用者懂「主題的分類學」，只要求他懂「他的讀者」。

1. **skill 派一個研究步驟**（限定官方來源：官方文件、官方教學、規格書），就該主題草擬一份**前置知識 ladder**——從**最基礎到進階**的 candidate 概念清單，由淺到深排序。
   - 例（Spark 主題）：`分散式運算基本概念 → DataFrame／SQL → partition 與平行度 → shuffle → join 策略（broadcast vs sort-merge）→ 記憶體與 spill → Catalyst／執行計畫 → tuning 旋鈕`。
2. **使用者只對每一階標三選一**：**會 ／ 半懂 ／ 不會**。
3. 由此自動推出地板（最後一個「會」）與天花板（第一個「不會」），中間「半懂」的就是手冊要重點補強、不能假設的地帶。

這樣使用者**只需要判斷「我的讀者對這個詞熟不熟」**（他做得到，因為他認識他的讀者），**不需要自己發明「這主題該怎麼分層」**（他做不到，所以由 skill 的研究步驟代勞）。負擔從「懂主題」翻轉成「懂讀者」。

> ladder 是 candidate 清單、不是定論。使用者標記時若覺得某階拆得不對、或缺了一階，直接調整——研究步驟的產出是給他改的草稿，不是考卷。

## 5. 兩個減壓閥（別在 persona 上過度糾結）

persona 引導要夠扎實，但**不是完美主義關卡**。兩個機制讓你「夠好就開工」：

1. **proxy 人物法本身降低非專家焦慮。** 把「定義一個抽象 persona」換成「描述一個我認識的真人」，認知負擔大幅下降——使用者不需要會做需求工程，只需要會回憶一個同事。
2. **persona 不必一次到位，靠 review loop 校準。** R2／R3 讀者會**通讀**全書、Follower 會**照文字試做**，他們會在審稿時抓出 persona 假設錯了的徵兆：**術語 gloss 太晚**（對這位讀者來說某詞首見就該解釋卻拖到三章後）、**漏了前置步驟**（假設讀者已具備其實沒有的背景）。所以 persona 是「夠好就開工、靠 review loop 校準」，不是動筆前必須打磨到完美的閘。

換句話說：先用 proxy 人物快速釘出一版可審的 persona，進入寫作；信任後段的讀者通讀＋Follower 探針把偏差校正回來。不要在第一步就追求終局精度。

---

## 6. 產出物：persona spec 模板

逐維收斂後，產出以下 spec 區塊，放進手冊 spec。**此 spec 會注入每一個 reviewer prompt——R2（初階讀者）、R3（進階讀者）、F（Follower）依此扮演**，C（一致性）與 P（教學法）也據此判斷深度是否對得上讀者。

```
## 讀者 Persona
### P1（地板）：<proxy 一句話>
- 已能：<具體任務>／想達成：<job-to-be-done>／深度期待：<水位>
- 工具介面：<…>／環境約束：<…>／不是寫給：<反例>
### P2（stretch，可選）：…
```

填寫提醒：

- 每一格都用 §1 右欄的**具體錨點**寫法，不留形容詞。
- 「proxy 一句話」直接寫那個真人的一句話側寫（如「三個月前的我，會 SQL、剛接觸 Spark」），讓 reviewer 一眼進入角色。
- P2 可選；若只有一位讀者就省略，不要為了對稱硬湊。
- spec 定稿即注入下游 reviewer prompt；R2 扮 P1 通讀問「這裡讀得懂嗎」、R3 扮 P2 問「這裡會不會太淺」、F 照 P1／P2 的工具與環境約束**只用手冊文字**動手試做。
