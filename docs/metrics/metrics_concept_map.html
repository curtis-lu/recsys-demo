排序推薦評估指標概念地圖

precision / recall / MAP / nDCG / hit_rate / HR · per-query 與 per-item 的語意與關係

這份文件釐清排序推薦常見評估指標（precision@K / recall@K / map@K / ndcg@K / hit_rate@K / HR@K / mean_pos …）的精確語意，以及 per-query 與 per-item 兩種切法的差異。讀完你能回答：哪個指標回答什麼問題、為什麼同樣叫 recall 數值卻不同、 per-item 的 precision 到底存不存在。

① 地基：一張二維資料格

評估資料的最小單位＝(時間, 客戶, 產品) 一格，每格有兩個值： label（客戶實際上有沒有要這產品，0/1，這是真相）與 score（模型預測的興趣分數）。

　prod_Aprod_Bprod_C…prod_22
客戶 c1 @ 1月L=1, s=.9L=0, s=.4L=1, s=.2……
客戶 c2 @ 1月L=0, s=.7L=1, s=.6L=0, s=.1……
………………
  • 一個 query＝一個 row（橫的一列）＝一位客戶在某時間點的「全產品排序問題」。
  • 模型在每一列裡，把所有產品照 score 由高到低給名次（即 pos／rank，1＝分數最高）。
  • total_rel（每 query 一個值）＝這位客戶這次總共想要幾個產品＝該列 sum(label)。

② 核心：兩個正交的切法

同一張格子，有兩個垂直方向可以收斂成指標 —— 這是所有混淆的根源：

           ┌───────── 產品方向 per-item ─────────┐

一句話記法

per-query ＝把格子一列一列壓扁，再對「客戶」平均 → 清單整體品質

per-item ＝挑一直行產品，看它被排到哪 → 單一產品品質

③ 計算流程：三層

不論用什麼工具實作，這些指標的計算都可拆成「格子 → 每列 → 對客戶平均」三層：

Layer 1  每一格算貢獻值
         pos, cum_rel, prec_at_pos, dcg_term,
         top_k@K(=pos≤K?1:0), ap_contrib@K, ndcg_contrib@K
                  │
        ┌─────────┴──────────┐
        ▼                    ▼
Layer 2  per-query           （per-item 不經 Layer 2，
  一列 → 一個數              直接從 Layer 1 的格子收斂）
  map@K ndcg@K
  precision@K recall@K
        │                    │
        ▼                    ▼
Layer 3  對 query 平均        對「label=1 的格」平均
  → per-query 指標           → per-item 指標

重點：per-query 一定要先把整列壓成一個數（precision@K 的分母 K、 recall@K 的分母 total_rel 都是「一整列」的概念）；per-item 不壓列， 直接在格子層級做平均。

④ per-query 家族

先對單一一位客戶（一列）算一個數，再對所有客戶等權平均。

名詞單一 query 的定義白話
precision@K(top-K 裡 label=1 的個數) / K「我推前 K 個，有幾成是命中的」
recall@K(top-K 裡 label=1 的個數) / total_rel「這客戶想要的，我前 K 個撈回幾成」
map@K該列的 Average Precision@Kprecision 沿排名累積的加權版（越前面命中越加分）
ndcg@KDCG@K / iDCG@K有 log 折扣的排序品質（位置越前權重越高），正規化到 [0,1]
HR@K
（經典 top-N 指標）		1 if (top-K 內 ≥1 個 label=1) else 0「這位客戶的清單，至少中一個了嗎」；分子是 0/1 指示，不是個數

每位客戶各算一個 X@K，再對客戶等權平均得到該指標。恆有 HR@K ≥ recall@K； 且當每位客戶恰好只想要 1 個產品時 HR@K == recall@K（leave-one-out 評估常見場景， 也是文獻常把兩者混用的原因）。

⑤ per-item 家族 ＋ 條件軸

per-item 多了一個「對哪個邊下條件」的軸。固定產品 P、對所有客戶 row（先不限 label）， 用 rank(P)≤K 與 label 做 2×2 混淆矩陣：

產品 P @ Klabel=1（想要 P）label=0（不想要 P）
rank(P) ≤ K（推進 top-K）TPFP
rank(P) > K（沒推進）FNTN

條件下在不同邊 → 不同指標：

指標公式條件白話
hit_rate@K
＝per-item recall@K		TP/(TP+FN)label=1想要 P 的客戶，幾成被推進 top-K
mean_posE[pos｜label=1]label=1被想要時平均排第幾名（越小越好）
map_attr@Kmean(ap_contrib@K)label=1P 對「客戶 AP@K」的平均貢獻碎片
ndcg_attr@Kmean(ndcg_contrib@K)label=1P 對「客戶 nDCG@K」的平均貢獻碎片
per-item precision@KTP/(TP+FP)rank≤K把 P 推進某客戶 top-K 時，他真的想要的比例
exposure@K(TP+FP)/N無P 被推進 top-K 的客戶佔比（純供給面，不需 label）
base rate(TP+FN)/N無P 的母體正樣本率
per-item AUCscore vs label無把 P 當二元分類器的排序品質

數字範例（產品 A，K=2，4 位客戶）：

c1: L=1,rank=1 → TP　c2: L=1,rank=4 → FN　c3: L=0,rank=2 → FP　c4: L=0,rank=5 → TN

→ TP=FP=FN=TN=1，故 recall@2(A)=hit_rate@2(A)=½、 precision@2(A)=½、exposure@2(A)=½、base rate(A)=½。

關鍵釐清：常被說「per-item 沒有 precision@K」是不精確的。 正確說法是 —— 它不是「per-query precision@K 的可加碎片」（per-query precision 分母 K 是一列的截斷數，無法歸因到單一產品）； 但它是乾淨的條件機率 P(label=1 | rank(P)≤K)。同名、不同東西， 和 recall／hit_rate 的處境一模一樣。

_attr 後綴＝「它是某個 per-query 指標的可加碎片（query 的 AP@K＝該列各正樣本格 ap_contrib@K 相加）」； hit_rate 無 _attr＝「它不是 query recall 的碎片，是重新定義的 item-marginal recall」。

⑥ 命名陷阱：兩個都叫「hit rate」

recsys 文獻裡「hit rate @ K」至少有兩個不同定義：

　(a) per-item hit_rate@K(b) 經典 top-N HR@K
切法per-item（挑產品 column）per-query（壓一整列）
單一單位的值每個 label=1 格的 0/1（rank≤K?）每個 query 的「≥1 命中?」0/1
分母該產品的正樣本數客戶（query）數
本質per-item recallper-query「至少中一個」
跟 precision@K 差異—分子是有沒有(0/1) 非個數；分母是客戶數非 K

因為「hit rate @ K」既可能指 (a) per-item recall、又可能指 (b) 業界經典 top-N HR@K， 看到這個詞時務必先確認指的是哪一個。文獻中 per-item recall 常被另取名 （hit_rate / per-label recall）正是為了與 (b) 區別。

⑦ 釐清：hit_rate@K（per-item recall）為何 ≠ per-query recall@K

兩者都量「想要的有沒有被撈到」，但加權單位不同。最小例子（3 產品、2 客戶）：

客戶 c1 想要 {A, B}  → 模型排名 A=1, C=2, B=3
客戶 c2 想要 {A}     → 模型排名 A=1, B=2, C=3
算法明細結果
per-query recall@1
per-query 等權		 c1: top-1={A}，想要 2 個撈到 1 → 1/2 = 0.5
c2: top-1={A}，想要 1 個撈到 1 → 1/1 = 1.0		 mean(0.5, 1.0)
＝ 0.75
每位客戶一票
per-item hit_rate@1
per-positive-row 等權		 A: 在 c1 排1 ✓、c2 排1 ✓ → mean(1,1)=1.0
B: 在 c1 排3 ✗ → mean(0)=0.0		 macro mean(1.0, 0.0)
＝ 0.50
每個命中事件一票

0.75 ≠ 0.50 純粹因為：per-query 讓 c1 的兩個正樣本共用一個 recall 值再跟 c2 平均（客戶等權）； per-item 讓每個 label=1 格獨立計一票（命中事件等權）。兩者都是「正確的 recall」， 只是聚合不同 —— 這也是 per-item 的 recall 常被另取名（如 hit_rate）以與 per-query recall 區別的原因。

⑧ 退化情況：precision@all / recall@all

當 K ≥ 產品總數（即 "all"），每一格都被算進 top-K：

指標退化成意義
precision@alltotal_rel / 產品總數，平均後＝母體正樣本率（base rate）標籤密度健檢，不是排序指標
recall@all每個 query 都 1.0（全包一定都撈到），平均＝1.0恆 1，無資訊量

這兩個值反映標籤密度，不是排序品質；解讀時不應與 K 較小時真正的 precision@K / recall@K 混為一談。

⑨ 一頁速查

資料格 = (時間, 客戶, 產品) + label(真相) + score(預測)
query     = 一位客戶一列；模型在列內把所有產品照 score 排名
total_rel = 這列 label 總和（這客戶想要幾個產品）

┌─ per-query（壓一整列 → 對客戶平均）───────────────────────────┐
│ precision@K  命中數 / K          推 K 個有幾成準                │
│ recall@K     命中數 / total_rel  想要的撈回幾成（客戶等權）      │
│ map@K        AP@K                 累積精確度（前面命中更值錢）    │
│ ndcg@K       DCG/iDCG             log 折扣排序品質，[0,1]        │
│ HR@K         1[top-K 有 ≥1 命中]  至少中一個的客戶比例           │
└───────────────────────────────────────────────────────────────┘
┌─ per-item（挑一產品 column → 看條件）─────────────────────────┐
│ 條件 label=1 ─ hit_rate@K  ＝per-item recall（命中事件等權）    │
│              ├ mean_pos    被想要時平均排第幾（越小越好）       │
│              ├ map_attr@K  對客戶 AP@K 的平均貢獻碎片           │
│              └ ndcg_attr@K 對客戶 nDCG@K 的平均貢獻碎片         │
│ 條件 rank≤K ── precision@K P 進 top-K 時真的想要的比例          │
│ 不條件 ─────── exposure@K  P 被推進 top-K 的客戶佔比            │
│              └ base rate   母體正樣本率                         │
└───────────────────────────────────────────────────────────────┘
注意：per-item「沒有 precision」是誤解 → 它不是 per-query precision
      的可加碎片，但是乾淨的條件機率 P(label=1|rank≤K)。
