# 評估側診斷產物判讀手冊

跑完 `evaluation` 之後，report.html 會多出幾段診斷：「主指標的 CI」「per_item 歸因的 CI 欄」「對帳 Reconciliation」「象限 Quadrant」，對應的 JSON 產物在 `data/evaluation/<model_version>/<snap_date>/diagnosis/` 下（`metric_ci.json`、`reconciliation.json`、`quadrant_summary.json`）。本手冊解釋這些數字怎麼來、怎麼讀。方法論根源見 `docs/ranking-diagnosis-framework.md`；本文只講判讀。

全文圍繞**一條計算鏈**展開，章節順序就是鏈的順序：

```
（理論側）你的抽樣/加權配置 ──§3──▶ 理論偏移帶 [theory_min, theory_max]
                                                    │
（實測側）評估資料 ──§4──▶ gap ──§5.2 扣全局參考值──▶ gap_vs_global
                                                    │
                              §5.3 兩者相比 ──▶ residual ──▶ verdict（§5.4）

            gap_vs_global 同時直接併入另一軸（不經過 residual/verdict）：
            評估資料 ──§8.2──▶ within-item AUC ──┐
            gap_vs_global ──────────────────────┴──§8.6──▶ 象限 quadrant
```

§2 先建立整條鏈共用的尺度（log-odds）的數感；§6 給拿到報表時的實戰判讀順序；§7 用一個可自行重現的注入實驗驗證整條鏈；§8 把 `gap_vs_global` 併上另一軸（條件判別力）看象限；§9 講 CI；§10 把水準缺口換算成一個具體、可收復的指標數字（offset sweep 分流）；§11 把交換名次的指標代價算成一本帳——誰壓了誰、代價多少（壓制帳本 pair ledger）；§12 列已知限制。

## 名詞速查

讀內文時忘了某個詞，回來查這張表；每個詞的完整說明在括號指的節。

| 名詞 | 一句話定義 |

| `theory_min` / `theory_max` | 配置推得的理論偏移範圍；配置沒動過的產品＝[0, 0]（§3） |
| `gap` | logit(平均預測機率) − logit(實際正類率)，用模型原始輸出 `score_uncalibrated`；正＝高估（§4） |
| `gap_calibrated` | 同 gap 公式，改用校準後的 `score` 欄（§4.2） |
| 母體條件化 | 評估母體只含有買過的客戶 → 所有產品的 gap 一起被拉低的效應（§5.1） |
| 全局參考值 | config 中性產品的 gap 中位數＝母體效應的估計值（§5.2） |
| config 中性 | 配置沒動過的產品（沒有任何 override、也沒有 label 不對稱權重）——理論偏移為 0（§3.3） |
| `gap_vs_global` | gap − 全局參考值＝扣掉共同水準後，產品**自己**偏了多少（§5.2） |
| `gap_calibrated_vs_global` | gap_calibrated − 校準版全局參考值（§5.2） |
| `residual` | gap_vs_global 超出理論帶的部分；帶內＝0（§5.3） |
| `verdict` | 可解釋（\|residual\| ≤ 門檻）／不可解釋／無法評估（§5.4） |
| post-training / monitoring 模式 | 訓練後用 test 窗預測評估（平常看的 report）／用正式批次推論結果評估（§5.1） |
| `within_item_auc` | 只取同一個 item 自己的列算 ROC-AUC；隨機一正一負、正例分數較高的機率，對水準偏移免疫（§8.2） |
| `gap_band` | 象限縱軸的容忍帶寬度（log-odds 尺度）；`gap_vs_global` 在帶內＝正常水準（§8.3/§8.6） |
| `auc_threshold` | 條件判別力「好／差」的分界值（§8.2/§8.6） |

| `quadrant` | 象限標籤，由水準狀態×判別力狀態決定，六格＋無法評估（§8.6） |
| `is_aggressor` | 是否為加害者；判準只看水準是否偏高，跟判別力好壞無關（§8.3/§8.6） |
| `cross_purchase` | 交叉購買矩陣 P(買 k｜買 j)；判斷壓制是否為真實傷害的折扣因子（§8.7） |
| `delta_star`（δ*） | 讓「折內 macro mAP 扣收縮懲罰」最大化的 per-item 平移量（近似 mAP 最大化，精確定義見 §10.5）；log-odds 尺度，跟對帳層 offset 同尺度（§10.2） |

## 1. 對帳表在回答什麼問題

**一句話**：模型的機率水準偏移，是不是你自己的抽樣／加權配置造成的？

訓練資料的抽樣與加權會刻意扭曲模型看到的正負樣本比例——比方把某產品的負樣本丟掉一半，模型看到的世界裡「買的人」比真實世界多一倍，它輸出的機率就會整體偏高。這不是 bug，是配置的直接後果，而且**可以事先精確算出來**（§3）。對帳（reconciliation）就是把「理論上應該偏多少」和「實際偏了多少」放在同一張表上對：

- **對得上 → 可解釋**：偏移是你配置造成的，不用動模型。
- **對不上 → 不可解釋**：有配置以外的事在發生（特徵漂移、資料問題、模型病灶），值得追。

這是診斷框架「判讀第 1 步」：先排除自己造成的偏移，再談模型有沒有問題。

**貫穿範例**：下面這張表就是一份真實的對帳表——用 repo 內建合成資料＋`conf/base` 的示例配置跑一次 `evaluation --env local --post-training` 得到的結果（節錄主要欄位）。先看個長相就好，每一欄怎麼算、怎麼讀，是本文其餘部分的內容；**後面各節引用的數字都出自這張表**，讀你自己的報表時把它們換成你表上的值。

## 2. 尺度：log-odds 的數感

整條計算鏈的加減都發生在 log-odds（logit）尺度上，先建立手感。

**定義**：機率 p 的 log-odds ＝ ln(p/(1−p))，也就是「勝算取對數」。換算錨點：

| 機率 | 5% | 10% | 20% | 33% | 50% | 67% | 80% | 90% |
|---|---|---|---|---|---|---|---|---|
| log-odds | −2.94 | −2.20 | −1.39 | −0.69 | 0 | +0.69 | +1.39 | +2.20 |

**加減＝勝算的乘除**。log-odds 加 x，等於勝算乘 e^x。手感錨點：

| log-odds 位移 | 勝算變化 | 例：10% 的機率變成 |
|---|---|---|
| ±0.1 | ×1.11 / ÷1.11 | 11.0% / 9.1% |
| ±0.3 | ×1.35 / ÷1.35 | 13.0% / 7.6% |
| ±0.69 | ×2 / ÷2 | 18.2% / 5.3% |
| ±1.1 | ×3 / ÷3 | 25.0% / 3.6% |
| ±2.3 | ×10 / ÷10 | 52.6% / 1.1% |

讀法示範：verdict 門檻 **0.3**（§5.4）＝「勝算差 1.35 倍以內都容忍」；若某產品的 gap 是 **−0.46**，意思是「模型把它的勝算低估到實際的 63%」（e^−0.46 ≈ 0.63）；gap **+0.69** ＝「勝算高估了一倍」。後面章節出現任何 log-odds 數字，都可以這樣讀。

**「勝算差 1.35 倍」是什麼感覺**——同樣 ×1.35 的勝算差，落在不同基率的產品上，機率大概差這麼多：

| 實際正類率 | 5% | 10% | 20% | 50% |
|---|---|---|---|---|
| 勝算 ×1.35 後的預測機率 | 6.6% | 13.0% | 25.2% | 57.4% |

也就是「低基率產品被高估約 1～3 個百分點、中基率約 5～7 個百分點」的量級——看得出來、但還不至於翻轉判斷。相比之下勝算差 2 倍（0.69）是 10% → 18.2%，已經是會改變排序與資源配置的量級。

**為什麼對帳選這個尺度**：因為抽樣／加權效應在 log-odds 尺度是**常數加法**——同一個 +0.69 的位移，把 10% 變 18.2%、把 30% 變 46.2%、把 50% 變 66.7%。機率尺度上看是三件完全不同的事（+8 個百分點、+16、+17），log-odds 尺度上是同一件事。位移量跟原機率無關，不同產品、不同水準之間才能互相加減比較——§5 的「扣全局、比理論帶」全靠這個性質。

## 3. 理論側：從配置算出 theory_min / theory_max

### 3.1 抽樣為什麼造成偏移

以本 repo 的實際配置為例，`conf/base/parameters_dataset.yaml`：

```yaml
sample_ratio_overrides:
  "mass|ccard_ins|0": 0.5      # mass 客群、ccard_ins、負樣本(label=0) → 只保留一半
  "affluent|ccard_ins|0": 0.9
  "hnw|ccard_ins|0": 0.8
```

負樣本保留 r=0.5、正樣本全留：勝算＝正的機率÷負的機率，把負類砍到 r 倍就是把勝算除以 r；取對數後「除以 r」變成「減 ln r」——模型在採樣資料上學到的 log-odds 比真實世界**高出 −ln r，且與原機率無關**（§2 的常數位移性質）。代入：r=0.5 → +0.693、0.9 → +0.105、0.8 → +0.223。這就是統計上的 logQ 校正（GBDT 手冊 3＝`docs/handbooks/gbdt/gbdt_multiitem_imbalance.md` 第 10 章）。

### 3.2 cell 級公式

`reconciliation.json` 的 `theory.cells` 每一格（客群×產品×label 組合，稱 cell）都用一般式算：

```
offset = ln( (r_pos × w_pos) / (r_neg × w_neg) )
```

r＝抽樣保留率、w＝訓練權重（`training.sample_weights`）。為什麼權重可以跟保留率走同一條公式？因為在加權 log-loss 下，給樣本權重 w 等效於把它**複製 w 份**（pseudo-replication）——對模型學到的 log-odds 而言，「一筆算 2 份」和「留下 2 倍的筆數」是同一件事，所以 w 對勝算的影響方式與 r 相同。正負類同率同權時分子分母相消＝0——所以**全域的** `sample_ratio`（正負一起抽）不產生偏移，只有「對 label 不對稱」的配置才會。

### 3.3 產品層：帶的算法，以及帶是 [0, 0] 的產品

**算法**：把該產品出現在配置裡的每個 cell 各算一個 offset，**theory_min＝最小值、theory_max＝最大值**。ccard_ins 的三個 cell 是 {+0.105, +0.223, +0.693} → 帶＝[0.105, 0.693]。

**為什麼是帶不是單值**：同產品在不同客群的保留率不同，混合後的產品層效應取決於各客群買家佔比——那是資料的性質，config 算不出來，所以只能誠實地給範圍。副作用：**帶越寬，verdict 越寬容**（帶內都算可解釋），判讀時看一眼帶寬。

**帶是 [0, 0] 的產品**：你沒有為這個產品設任何 `sample_ratio_overrides`、也沒有 label 不對稱的權重——§3.2 公式的分子分母全是預設值，相消為 0。理論偏移 0，帶退化成單點 [0, 0]，意思是「照配置，這個產品不該有任何水準偏移」。本文把這類產品稱作 **config 中性**（＝配置沒動過的產品），§5.2 會拿它們當基準。

## 4. 實測側：從評估資料算出 gap

### 4.1 gap 的定義

每個產品算：

```
gap = logit(平均預測機率 p̄) − logit(實際正類率 ȳ)
```

分數用 **`score_uncalibrated`（模型原始輸出）**——因為校準層本身就在修水準，要看模型「原本」偏多少（校準層的效果由下一欄單獨評，見 §4.2）。正值＝模型高估這個產品、負值＝低估；量級用 §2 的尺讀。

### 4.2 gap_calibrated

同一條公式改用校準後的 `score` 欄。兩欄對照的用途：`gap` 顯示模型原始輸出偏多少，`gap_calibrated` 顯示過了校準層之後還剩多少——差距就是校準層修掉的量。

### 4.3 誠實限制（近似）

嚴格說，logQ 位移加在**每一筆**預測的 log-odds 上，理想量法是逐筆取 logit 再平均；但那樣無法用一次聚合完成（工程取捨）。本表用「先平均再取 logit」，兩者不相等（Jensen 不等式）——產品內分數越分散差異越大。量級感：分數集中在 0.05±0.02 時幾乎無差；分數從 0.01 散到 0.5 時可差 0.1～0.2。在 0.3 的門檻下通常是二階效應，但「分數分布很寬的產品剛好壓線」時要想起這件事。

## 5. 對帳：從 gap 到 verdict

### 5.1 為什麼不能拿 gap 直接比理論帶

先定義兩個詞：**post-training 模式**＝訓練剛結束、拿訓練 pipeline 在 test 窗留下的預測評估（`evaluation --post-training`，平常看的 report 就是它）；**monitoring 模式**＝拿正式批次推論的結果評估上線表現。

直接比會犯一個系統性錯誤，原因出在評估母體的構成。post-training 模式只評「test 月有買過任何產品的客戶」——`training_eval_predictions` 只寫入這些人。這個篩選讓**每個產品**的實際正類率 ȳ 都被墊高：留下來的每位客戶都保證至少買了一件（直覺：8 個候選至少中 1 件，平均每產品購買率就有 12.5% 起跳），而模型是照「全體客戶」的世界學的——那個世界裡多數人什麼都沒買。ȳ 比模型的預期高、且每個產品都一樣高，於是**所有產品的 gap 一起往負向移動差不多大的一段**，跟任何產品自身的好壞無關。

回頭看 §1 的示例表就能看到這個效應的長相：theory 帶是 [0, 0] 的 7 個產品，gap 欄全部擠在 −0.39 ～ −0.65 的窄區間——照理論它們一個都不該偏，卻一起沉了差不多深，這就是母體效應；表中唯一的例外 ccard_ins（theory 帶 [+0.105, +0.693]）gap 只有 −0.133，沒有跟著沉下去，因為它的理論偏移在往上頂。**在你自己的報表上認的就是這個 pattern**：theory 帶為 [0, 0] 的產品群，gap 是否集中在同一個區間。

這叫**母體條件化**：它是評估母體的性質，跟任何單一產品的校準品質無關。如果拿絕對 gap 直接比理論帶，7 個健康產品全會被誤判「不可解釋」。所以要先把這個共同水準扣掉。

### 5.2 全局參考值：共同水準怎麼估、怎麼扣

**算法**：把所有 **config 中性**產品（＝配置沒動過、照理論不該偏的產品，§3.3）的 gap 收集起來，取**中位數**。為什麼它們能當基準：這些產品理論偏移是 0，所以它們實際偏了多少，量到的就是「所有產品共同承受的母體效應」。為什麼取中位數而不是平均：萬一某個中性產品其實有自己的真實問題（gap 異常大），平均值會被它拖走，中位數不會——基準不容易被單一異常產品帶偏。**中性產品不足 3 個時**中位數太不穩，直接退回 0（等於不扣、回到絕對比較），JSON 的 `global.method` 會標 `insufficient_neutral_items_fallback_zero`。

對 §1 示例表：7 個中性產品的 gap 排序後，中位數是 −0.462——恰好是 ccard_cash 那一列的 gap。這就是全局參考值。然後每個產品算：

```
gap_vs_global = gap − 全局參考值
```

**兩個閱讀時會遇到的現象，都是算法的必然**：

- **恰有一個中性產品的 gap_vs_global 顯示 0.000**：中性產品是奇數個時，中位數就是排序正中間那個產品自己的 gap，它減自己＝0（§1 示例表就是如此：ccard_cash 那列的 gap_vs_global 為 +0.000）。這代表「它被選為基準」，不代表它特別健康。
- **`gap_calibrated_vs_global` 用的是另一個參考值**：校準後分數的母體位移量不同，所以對中性產品的 gap_calibrated 另取一次中位數（JSON 的 `global.reference_calibrated`）。校準層有效時，`gap_calibrated_vs_global` 應遠小於 `gap_vs_global`、趨近 0。

**基準本身可不可信**，看 JSON `global` 區塊的兩條線索：`n_neutral_items`（基準靠幾個中性產品撐——越少越不穩）；`pooled_gap`（用另一種算法算的全局水準粗讀數：把**全部產品**——含非中性——的 p̄/ȳ 按 n_rows 加權合併後取 logit 差。與 reference 量級應相近；兩者的差來自非中性產品的配置效應與跨產品分歧，示例表那次執行是 −0.379 vs −0.462。差距異常大時，降低對基準的信任）。陷阱：若某個中性產品其實有自己的真實問題，它會污染基準；中位數有抵抗力，但中性產品本來就少時要提高警覺。

### 5.3 residual：超出理論帶的部分

```
residual = gap_vs_global − clip(gap_vs_global, theory_min, theory_max)
```

clip(x, a, b)＝把 x 壓回 [a, b]：小於 a 取 a、大於 b 取 b、在範圍內取 x 本身。直覺：理論帶能解釋的先扣掉，剩下的才是 residual。三個例子（前兩個就是 §1 示例表的 ccard_ins 與 fund_bond 列，可對照著看）：

1. **落在帶內 → residual = 0**。ccard_ins：gap_vs_global=+0.329，帶 [0.105, 0.693]，本來就在範圍內 → residual = 0。實測偏移完全在理論預期內。
2. **帶是 [0, 0] → residual 恰等於 gap_vs_global**。fund_bond（無注入時）：gap_vs_global=−0.186，任何數 clip 到 [0,0] 都是 0 → residual = −0.186。沒有理論可扣，整個 gap_vs_global 都是待解釋量——所以表上多數產品 residual==gap_vs_global，是常態不是巧合。
3. **超出帶 → residual＝超出最近邊界的量**。假想 gap_vs_global=+1.0、帶 [0.105, 0.693] → residual = 1.0 − 0.693 = +0.307，壓過 0.3 門檻 → 不可解釋。

### 5.4 verdict

- **可解釋**：|residual| ≤ 門檻（config `explained_threshold`，預設 0.3——用 §2 的尺：勝算差 1.35 倍以內都容忍）。
- **不可解釋**：超過門檻。有配置之外的事，看 §6 判讀順序第 2 步。
- **無法評估**：gap 算不出來，`reason` 欄寫原因——該產品實際全買或全沒買（logit 未定義），或 config 有它的理論偏移但評估資料裡根本沒有它的列（下架？打錯產品名？）。無法評估不影響其他產品的判定。

**為什麼門檻預設 0.3**：因為它被夾在兩個量級中間，兩端都能在 §1 示例表上指認。

- **往下是量測底噪**：gap 是有限樣本上的估計，又疊了 logit(平均) 的近似（§4.3）與中位數基準自身的估計誤差（§5.2）——健康產品的 gap_vs_global 本來就不會恰好是 0。示例表 7 個中性產品的 gap_vs_global 散佈在 −0.186 ～ +0.071，這就是實際的底噪量級。門檻若設 0.15，表中的 fund_bond（−0.186）就會被誤報成「不可解釋」——對著雜訊追根究柢。
- **往上是值得追的效應**：常見的配置動作（負樣本保留率砍半）就是 0.69 的量級；真正的模型病灶把勝算估錯兩倍以上也不罕見。門檻若設 0.7，「勝算高估一倍」這種該追的問題會靜默通過。

0.3（勝算差 1.35 倍——低基率產品被估錯 1～3 個百分點，§2 的小表）落在底噪之上、目標效應之下。要誠實說明：它是**工程預設，不是統計推導出的常數**（config `explained_threshold` 可調）——你的環境底噪更大（產品少、正例少、基準靠很少的中性產品撐）就該調高；想抓更細的偏移、且中性產品夠多夠穩，可以調低。

## 6. 實戰判讀

**判讀順序**：

1. 掃 `verdict` 欄。全部「可解釋」→ 機率水準沒有配置之外的異常，結束。
2. 有「不可解釋」→ 看 `gap_vs_global` 與理論帶差多遠、往哪個方向，對照該產品近期的配置改動與特徵變化；這是後續診斷步驟（offset sweep、§8 的象限層）的輸入。
3. 有「無法評估」→ 看 `reason` 欄。
4. 想確認校準層 → 比 `gap_calibrated_vs_global` 與 `gap_vs_global`：前者明顯小＝校準在做事。monitoring 模式沒有 `score_uncalibrated` 欄時會自動退回 `score` 並在描述標註——此時 gap 已含校準層效應，判讀打折。

**欄位對照表**（每列自足，公式出處在括號節）：

| 欄位 | 怎麼算 | 判讀 |
|---|---|---|
| `theory_min` / `theory_max` | 該產品各 cell 理論 offset 的 min／max（§3.3）；配置沒動到＝[0, 0] | 帶寬＝跨客群異質度；越寬 verdict 越寬容 |
| `gap` | logit(`p_mean`) − logit(`y_rate`)，用 `score_uncalibrated`（§4.1） | 含母體效應，**不要**直接拿來判產品好壞 |
| `gap_vs_global` | `gap` − 全局參考值（§5.2） | 扣掉共同水準後產品自己偏多少；verdict 的依據 |
| `gap_calibrated` | 同 gap 公式，改用 `score` 欄（§4.2） | 同樣含母體效應，看下一欄 |
| `gap_calibrated_vs_global` | `gap_calibrated` − 校準版參考值（§5.2） | 校準有效時應遠小於 `gap_vs_global`、趨近 0 |
| `residual` | gap_vs_global 超出理論帶的部分（§5.3） | \|residual\| ≤ 門檻 → 可解釋 |
| `verdict` | §5.4 的三值 | 判讀順序第 1 步 |
| `reason` | 無法評估的原因 | 只有無法評估的列有值 |
| `p_mean` / `y_rate` / `n_rows` | 平均預測機率／實際正類率／評估列數 | n_rows 小的產品 gap 本身就吵 |

逐產品表之外，`reconciliation.json` 有全表層級的 **`global` 區塊**：`reference`（全局參考值）、`reference_calibrated`（校準版）、`method`（中位數或 fallback）、`n_neutral_items`、`pooled_gap`——§5.2 的可信度檢核看這裡。

## 7. 實例走讀：已知答案注入實驗

驗證這條計算鏈最直接的方法，是刻意注入一個已知的偏移、確認對帳表把它精確量出來。以下對 §1 示例表中的 fund_bond（注入前：帶 [0, 0]、gap −0.648）動手，完整走一遍（可自行重現）：

1. 對 `fund_bond` 三個客群的負樣本全部注入保留率 0.5 → 理論帶收斂成單點 [0.693, 0.693]（§3）。
2. 重訓重評。`fund_bond` 的絕對 gap 從 −0.648 移到 +0.042——**位移 +0.690，幾乎恰好等於 ln 2**：模型完整吸收了配置造成的偏移，§3.1 的公式定量命中。
3. 注意一個連動：注入後 fund_bond **不再是 config 中性產品**，全局參考值改用剩下 6 個中性產品重算（新參考值 −0.442）——所以這裡的 gap_vs_global 不能拿 §5.2 的 −0.462 反推。`gap_vs_global` = 0.042 − (−0.442) = +0.484，residual = −0.209（帶下緣往回量），|residual| ≤ 0.3 → 可解釋（§5.3）。
4. `gap_calibrated` = −0.127：校準層把注入的偏移修掉大半——§4.2 兩欄對照的用途。
5. 還原配置重訓，帶回 [0, 0]、對帳表回全綠。

對帳層的兩種用途在此都示範了：**正向**（配置效應可預測、可驗證）與**反向**（哪天有產品不可解釋，你知道那不是配置造成的）。

## 8. 象限層：水準 × 條件判別力

象限（quadrant）是對帳層之外的第二張診斷表——`quadrant_summary.json`，report.html 的「象限 Quadrant」段落把它畫成散布圖＋兩張表。它問一個對帳表答不了的問題：**這個產品分不分得出該推給誰，而且這件事有沒有連帶壓到別人**。

### 8.1 象限在回答什麼問題

把每筆預測分數粗分成三塊：

```
分數 = query 效應 + item 水準 + 條件部分
```

這裡的「分數」是決定排序的原始輸出（`schema` 的 `score`／`score_uncalibrated` 欄，二元分類模型的原始輸出本來就落在 log-odds 尺度上）——所以「同一個 query 內每個候選加同一個數」在這個尺度上是合法的加法，跟 §2 建立的心法（log-odds 尺度上才能直接加減）是同一件事，不是另立一套算法。

- **query 效應**：同一個 query（同一個 time×entity 組合）裡，每個候選都加同一個數，完全不影響誰排前面——對名次、對指標都沒有意義。
- **item 水準**：這個產品平均被打得多高或多低——§1～§7 對帳表量的就是這一塊，`gap_vs_global` 是它的量尺。
- **條件部分**：分數要不要隨著「這位客戶適不適合這個產品」而動，這才是模型該學的東西，量尺是本節的 within-item AUC（§8.2）。

兩塊各對應一種病灶：水準塊出問題＝產品整體偏高或偏低（不管買誰的東西都一樣的偏移）；條件塊出問題＝模型分不出這個產品該推給誰（分數在該買的人跟不該買的人之間幾乎沒有差別）。

**為什麼兩塊要一起看**：想像兩個「條件判別力」一樣差的產品——模型對它們的預測都近乎一個常數，分不出該推給誰。如果這個常數剛好**高**，它在每個 query 都排前面，會把其他產品的正例往下擠（自己的名次很好看，代價是壓別人）；如果這個常數剛好**低**，它自己的正例永遠墊底（自己名次難看，但不礙著別人）。同樣的「條件判別力差」，水準一高一低，後果完全相反——只看判別力軸看不出這個差別，只看水準軸也看不出「差」的成因是不是判別力。象限把兩軸疊在一起，才分得出「冷門受害者」跟「常數高分型加害者」。

### 8.2 AUC 的數感

橫軸的量尺是 **within-item ROC-AUC**：只取同一個產品自己的列，定義是「隨機抽這個產品的一個正例、一個負例，正例分數較高的機率」。跟 §2 的 log-odds 一樣，先建立手感：

| AUC | 讀法 |
|---|---|
| 0.5 | 丟銅板——完全分不出該推給誰 |
| 0.6 | 略有訊號 |
| 0.75 | 可用 |
| 0.9 | 很強 |

**常數分數恰好是 0.5**：一個產品不管客戶是誰都給同一個分數，任兩筆比大小都是平手——這正是條件判別力為零的極端狀態，數字上剛好卡在「丟銅板」那格，不是巧合。

**對水準偏移完全免疫**：把某個產品所有列的分數統一加上 10，產品內部誰高誰低完全不變，AUC 不會動分毫——它只在乎產品自己內部的相對順序，從不跟別的產品比。這正是它能拿來當第二軸的原因：跟縱軸 `gap_vs_global`（專量水準）正交，兩者互不污染，才能疊成一張不含混的象限圖。

**用哪一欄分數算**：跟 `gap_calibrated`（§4.2）同一欄——決定上線排序的 `score`，不是 `gap` 用的 `score_uncalibrated`。嚴格單調的變換不影響 AUC（排序不變），所以這個選擇本身影響不大；但如果校準層把某一段分數壓平（原本分得開的兩筆變成同分），那一段的 AUC 會因此下降——這正是要量的行為（校準抹平了原本的排序訊號），不是要迴避的雜訊。

**「0.6」讀成「沒有慘到跟丟銅板一樣」，不是「判別力很強」**：對照上面的錨點表，0.6 只比丟銅板（0.5）高一點，落在「略有訊號」那格；quadrant 標籤把 AUC ≥ 0.6 判成「好」，讀法是「這個產品勉強有分辨力、沒有掉進近乎隨機的區間」，跟真正判別力強（0.75 以上）是兩回事。§8.6 六格裡的「健康」是這個寬鬆意義下的健康，不是判別力優異的意思——想抓更嚴格的標準，把 `auc_threshold` 調高即可（§12）。

### 8.3 縱軸為什麼用 gap_vs_global、不用 residual

§5 對帳表把 `gap_vs_global` 拆成理論帶內（residual=0，verdict=可解釋）跟帶外（residual≠0）——那一步問的是「這個水準偏移能不能由配置解釋」，是**歸因**。象限的縱軸直接借用 `gap_vs_global`（不重算，對帳層算過一次就夠），問的是另一個問題：**不管能不能解釋，這個產品現在的水準有沒有偏、有沒有壓到人**——是**行為觀測**。

同一份示例資料裡 `ccard_ins` 就是兩者會給出不同答案的活例子。對帳表（§1）算出它的 `gap_vs_global` = +0.329，落在理論帶 [+0.105, +0.693] 之內，residual=0，verdict=可解釋——配置（它在 `sample_ratio_overrides` 裡的抽樣覆寫，§3.1）完全能解釋這個偏移。但象限層量到的行為並沒有因為「可解釋」就消失：它佔據了 27.5% 的 query 的 top-1 名次，而它自己的實際正類率只有 20.9%（佔據率高過正類率，§8.4 會再細看這個對照）；它以負例身分壓在別人正例上方的次數是 231 次，全場僅次於 exchange_usd 的 260 次。這裡的 `gap_band` 門檻是 0.35，+0.329 還沒超過，象限判它「正常」水準、「健康」——但只要把 `gap_band` 調到小於 0.329（例如 0.25），同一個 `gap_vs_global` 就會翻成「偏高」，quadrant 變成加害者。**對帳表回答「這個偏移是不是配置造成的」；象限回答「這個產品現在的行為是不是在壓別人」——兩者可以同時成立，不衝突，只是問題不同**。

逐欄讀法：

- **quadrant**：這批示例資料只出現「健康」跟「冷門受害者」兩格——完整的六格＋無法評估的定義見 §8.6。
- **gap_vs_global**：跟 §1～§7 對帳表同一欄、同一個數字，直接借用，不重算。
- **auc**：within-item AUC，數感見 §8.2。三個 `fund_*` 產品全落在 0.460～0.521——貼著「丟銅板」的 0.5，判別力幾乎等於沒有；`ccard_ins`、`exchange_usd` 等五個健康產品都在 0.69 以上。
- **ap_sampled**：跟主指標段同一顆 `ap`，這裡標成 `ap_sampled` 是要提醒它跟 §9（`metric_ci.json`）同一份診斷抽樣估出來、附信賴區間——本表沒印 CI 兩欄，但 JSON 裡跟 `ap_sampled` 放在一起。`n_pos` 個位數的產品這顆數字本來就不穩：`fund_mix` 的 `n_pos`=23，是這批資料裡最少的一個，三個 `fund_*` 產品的 `n_pos`（54、23、77）整體都在後段——`fund_mix` 的 `ap_sampled`=0.205 這個點估計，配的 CI 會很寬，見 §9「`n_pos`(抽樣) 個位數代表該列 CI 不可靠」的提示，先看 `n_pos` 再信 `ap_sampled`。
- **top_share vs y_rate**：`top_share` 是這個產品佔據 top-`k`（這裡 k=1）名次的 query 比例，`y_rate` 是它的實際正類率——並排看的讀法是**佔據率明顯高過正類率＝水準訊號**（它排前面的次數比它「真的該排前面」的次數多）。`exchange_usd` 是這個讀法最乾淨的示範：`top_share`=70.0% 遠高於 `y_rate`=47.4%，但它的 `auc`=0.731（判別力好）、`gap_vs_global`=-0.076（水準正常，還略偏低）——一個健康的熱門產品長這樣：常拿第一名，是因為它真的常被買、也真的分得出該推給誰，不是靠水準灌出來的。`ccard_ins` 的 `top_share`=27.5% 對 `y_rate`=20.9% 也是同方向的落差，配合 §8.3 的壓制次數一起讀。
- **suppression_count**：這個產品「以負例身分排在該 query 首位正例上方」的**次數**（原始計數，不是比率）——分母的語境是 `n_pos_queries`=654 個有正例的 query，不是 654 個 query 每個都會被算一次，是「輪到它以負例之姿擠在別人正例前面」才計一次；想看佔比自己除以 654，例如 `exchange_usd` 的 260 次約佔 40%、`ccard_ins` 的 231 次約佔 35%。`exchange_usd`、`ccard_ins` 是全場最高的兩名；三個 `fund_*` 產品的 `auc` 接近丟銅板但 `suppression_count` 都很低（27、2、67，換算約 4%、0.3%、10%）——判別力差不等於會壓別人，還要看它排不排得到前面（`gap_vs_global` 沒有偏高，它就沒有機會壓人）。
- **n_pos**：進這張表的正例列數（**全量**評估資料的計數），樣本量的直接指標——`n_pos` 小的產品，`auc` 跟 `ap_sampled` 都要打折看。注意它跟 §9 的 `n_pos(抽樣)` 不是同一顆：`ap_sampled` 的有效樣本是診斷抽樣後的那份；示例資料兩者相等（資料量沒觸發抽樣上限），資料量大到觸發截斷時兩者會分開，屆時 `ap_sampled` 的可信度看 §9 的 `n_pos(抽樣)`，不是本欄。

**候選解釋（不是定論）**：三個 `fund_*` 產品「判別力差＋樣本少」湊在一起的組合，是框架手冊（`docs/ranking-diagnosis-framework.md` Ch 3 診斷項目 8）「葉預算餓死」的典型輪廓——先驗（水準）有學到、個人化（條件部分）沒學到。但這張表本身給不出這個結論，要確認是不是真的餓死，得去看 Gain 帳本（同一手冊同一節）；這裡只是點出「像什麼」，不是「就是什麼」。

### 8.5 散布圖讀法

report.html 的散布圖把每個 item 放到（AUC，`gap_vs_global`）平面上：橫軸是 AUC（越往右判別力越好），縱軸是 `gap_vs_global`（越往上水準越偏高）。圖上有兩條參考線：淺藍色水平帶＝水準大致正常的範圍（`|gap_vs_global| ≤ gap_band`，這裡是 ±0.35）；垂直虛線＝判別力門檻（這裡是 `auc_threshold`=0.6）。

兩條線把平面切成幾塊，對到 §8.6 的象限標籤：帶內、虛線右側＝健康；帶內、虛線左側＝冷門受害者；帶頂之上、虛線右側＝加害者（水準偏高、判別力好）；帶頂之上、虛線左側＝加害者（常數高分型）；帶底之下、虛線右側＝受害者（水準偏低、判別力好）；帶底之下、虛線左側＝雙重受害。

這批示例資料的點：五個「健康」產品都落在虛線右側、淺藍帶以內；三個 `fund_*` 都落在虛線左側、一樣在帶以內——沒有一個點跑到帶外，所以圖上看不到加害者或受害者的角落，只有健康跟冷門受害者兩叢。`ccard_ins` 是帶內最靠近帶頂的一點（+0.329，帶頂是 +0.35）——貼著線，§8.3 已經講過這代表什麼。

### 8.6 象限標籤

六格加一個「無法評估」，水準狀態（`gap_vs_global` 相對 `gap_band`）× 判別力狀態（AUC 相對 `auc_threshold`）：

| (水準, 判別力) | quadrant |
|---|---|
| (正常, 好) | 健康 |
| (正常, 差) | 冷門受害者（水準對、判別力差） |
| (偏高, 好) | 加害者（水準偏高、判別力好） |
| (偏高, 差) | 加害者（常數高分型） |
| (偏低, 好) | 受害者（水準偏低、判別力好） |
| (偏低, 差) | 雙重受害 |

任一軸缺值（例如對帳層停用、或該產品 AUC 算不出來）→ 那一軸「無法評估」→ quadrant 整格「無法評估」。

**加害者的判準只看水準偏高，跟判別力好壞無關**——判別力好但整體被抬高的產品會壓別人，判別力差、分數卡在一個高常數的產品一樣會壓別人（甚至更難查，因為它自己名次很好看，不會被自己的指標揪出來）。這是為什麼「偏高」那一整列（不分判別力好壞）都標成加害者。

### 8.7 交叉購買矩陣讀法

`quadrant_summary.json` 的 `cross_purchase` 區塊是另一張表，回答「壓制到底算不算真傷害」——如果被壓的那個正例的買家，原本大機率也會買壓他的那個產品，傷害就打了折（買家兩個都想買，排前面那個對他來說都一樣）。

矩陣讀法：P(買 k｜買 j) 是「列→欄」的方向——列是 j（條件，已知買了什麼），欄是 k（結果，問買了 k 沒有）；對角線恆為 1（買了 j 的人當然買了 j）。矩陣不對稱，方向不能反著讀：

| P(買k\|買j) | ccard_bill | ccard_cash | ccard_ins | exchange_fx | exchange_usd | fund_bond | fund_mix | fund_stock | n_buyers |
|---|---|---|---|---|---|---|---|---|---|
| **ccard_bill** | 1.00 | 0.11 | 0.26 | 0.21 | 0.26 | 0.06 | 0.03 | 0.07 | 1344 |
| **ccard_cash** | 0.19 | 1.00 | 0.29 | 0.19 | 0.27 | 0.04 | 0.03 | 0.07 | 792 |
| **ccard_ins** | 0.18 | 0.12 | 1.00 | 0.21 | 0.27 | 0.05 | 0.02 | 0.08 | 1891 |
| **exchange_fx** | 0.09 | 0.05 | 0.13 | 1.00 | 0.55 | 0.04 | 0.02 | 0.06 | 3170 |
| **exchange_usd** | 0.08 | 0.05 | 0.12 | 0.43 | 1.00 | 0.04 | 0.02 | 0.06 | 4073 |
| **fund_bond** | 0.12 | 0.04 | 0.14 | 0.20 | 0.27 | 1.00 | 0.02 | 0.07 | 644 |
| **fund_mix** | 0.11 | 0.08 | 0.13 | 0.17 | 0.25 | 0.05 | 1.00 | 0.05 | 332 |
| **fund_stock** | 0.09 | 0.06 | 0.15 | 0.20 | 0.25 | 0.04 | 0.02 | 1.00 | 1006 |

`exchange_fx` 那一列、`exchange_usd` 那一欄是 0.55：買了 `exchange_fx` 的人有 55% 也買 `exchange_usd`。反過來，`exchange_usd` 那一列、`exchange_fx` 那一欄是 0.43：買了 `exchange_usd` 的人只有 43% 也買 `exchange_fx`。兩個方向不一樣——`exchange_fx` 的買家更黏 `exchange_usd`，不是對稱關係，讀哪個方向要看你在問誰對誰的問題。

**必看 `n_buyers` 這欄基數**：P(k|j) 是條件機率，分母就是 j 的買家數。`fund_mix` 的 `n_buyers`=332，是全表最小——它那一列（買了 `fund_mix` 的人還買了什麼）的比例是由 332 個人估出來的，比 `exchange_usd` 那一列（4073 人估）的同類數字抖動大得多，讀 `fund_mix` 那一列時要打折。

**母體跟本節其他表不同**：交叉購買矩陣吃的是 label 資料的完整歷史（所有快照日的購買事件），不是象限表那一批評估視窗資料——`ccard_ins` 的 `n_buyers`=1891 遠大於象限表同產品的 `n_pos`=137，就是這個差異的直接痕跡。它回答的是「歷史上誰跟誰共購」（長期替代性），`suppression_count` 回答的是「當期這一批排名裡誰壓了誰」；兩張表併看是拿長期關係解讀當期傷害，但兩者的分母屬於不同母體，不可互除、也不要拿共購率去換算壓制佔比。

**跟壓制次數的關係**：`suppression_count`（§8.4）只算「以負例身分排在別人正例上方」的次數，不管被壓的那個客戶其實會不會也買壓他的產品。把兩張表放在一起看：某產品的高壓制次數，如果壓的對象剛好是交叉購買率高的產品，對那位客戶來說「兩個都買」的實際損失就小；壓制次數集中在交叉購買率低的產品組合上，才是真正該優先處理的實質傷害。

## 9. metric_ci.json 與 CI 欄

主指標段與 per_item 歸因段的 CI 欄回答另一個問題：**指標數字有多少統計不確定性**。

- CI 是**抽樣估計**：在有正例的 query 上抽樣（上限與保底見 config `evaluation.diagnosis.sample`），對客戶（cust_id）整簇重抽做 bootstrap。樣本規模與 n_boot 印在報表描述——樣本小的估計不要當真。
- `n_pos(抽樣)` 欄＝該產品進入 CI 估計的正例列數；個位數代表該列 CI 不可靠，先看這欄再看區間。
- 觀察名單＝正例數低於 `evaluation.metric.min_positives` 而移出 macro 平均的產品；指標照列，只是不進等權平均。

## 10. 分流層（offset sweep）

分流閥（offset sweep）是對帳層與象限層之外的第三張診斷產物——`offset_sweep.json`（跟其他診斷產物同一個目錄：`data/evaluation/<model_version>/<snap_date>/diagnosis/`），report.html 的「分流 Offset sweep（水準 vs 條件判別力）」段落把它畫成摘要表＋per-item 表＋waterfall 圖。它問一個比象限層更進一步的問題：**如果讓每個產品的分數整體平移（不重訓），指標最多能救回多少？救不回的部分才是真正卡在訓練裡的判別力缺口。**

### 10.1 分流閥在回答什麼

指標缺口（macro mAP 沒有你想要的那麼高）可以拆成兩塊：**水準**（level）——某個產品整體被排得太高或太低，這一塊不用重訓，理論上只要幫它加一個常數（per-item 平移）就能修；**條件判別力**（conditional discrimination）——同一個產品內部，模型分不出這一列該不該買，這一塊只能靠動訓練（特徵、資料、目標函數）解決。這正是 §8.1 那個「query 效應／item 水準／條件部分」拆解裡，水準塊與條件塊的另一種量法。

offset sweep 直接把第一塊「能救多少」量出來：對每個產品的 logit 分數加一個常數 δ，座標下降搜尋讓 macro per-item mAP 最大的那個 δ*（一次調一個產品，輪流迭代）。mAP(δ*) − mAP(0) 收復的部分＝水準缺口；mAP(δ*) 之上、到理論上限之間收不回的部分＝條件判別力缺口——象限層（§8）已經在從另一個角度講這兩塊，offset sweep 把它換算成一個具體的數字。

### 10.2 δ 的尺度與數感

δ 加在 logit（log-odds）分數上，跟 §2 建立的尺度、§3 對帳層的 offset 完全同一把尺——讀法照抄 §2 的錨點表：δ=+0.69 ＝ 該產品每一列的勝算翻倍。δ* 是搜尋出來、讓 macro mAP 最大的那個位移，錨點反過來讀一樣通：δ*=−1.0 ＝「把該產品的勝算砍到 1/e≈0.37 倍才是最優點」——不是說模型現在錯了這麼多，是說「往這個方向平移，指標會更好看」。§10.6 的示例表會看到實際跑出來的 δ* 值，量級多落在 ±1 以內。

### 10.3 只有相對差可識別（centered 欄）

δ* 有一個看起來反直覺的性質：對**所有**產品的分數同加一個常數 c，每個 query 內的候選還是同一個順序（大家都加了 c，相對名次不變），排名不變、macro mAP 自然也不變。所以座標下降找到的 δ* 向量，其實混了一個「隨便挑一個共同平移量」的自由度——這個自由度有個名字叫 **gauge**（規範自由度）：可觀察量（排名、mAP）不隨它變，但描述它的參數（δ*）會。單看某個產品的原始 δ*，看到的數字裡摻了這個共同平移，不是它自己「應該」偏移多少的乾淨讀數。

`per_item` 因此同時給兩欄：`delta_star`（座標下降實際搜到的原始值）與 `delta_star_centered`（減去所有產品 δ* 的平均，去掉共同平移）。判讀規則：**跨執行、跨 model_version 比較用 centered**——不同執行搜到的共同平移量不保證相同，原始值不能直接對比；**同一次執行內比較兩個產品的相對高低，兩欄給的差值完全一樣**（都是各自減掉同一個常數），哪欄都行。

用 §10.6 的示例表示範這個落差：fund_bond 的原始 `delta_star`=+0.40，`delta_star_centered`=+0.494——兩者差 −0.094（raw 減 centered），這正是這次搜尋選中的共同平移量（8 個產品的 delta_star 平均約 −0.094，用平均值反推即可對上；這次是負的，代表這批產品整體被往下修，不是每次都同一個方向）。單看 +0.40 看不出 fund_bond 跟其他產品的相對高低；看 +0.494（跟其他產品相對比）才是乾淨的讀法。

### 10.4 折內／折外

δ* 不是在全部診斷抽樣的 query 上搜出來的——`_fold_split`（`offset_sweep.py:53-87`）先把 query 整個切成兩半：折內（fit，`holdout_fraction` 之外的部分）跟折外（holdout，佔 `holdout_fraction` 比例，config 預設 0.5，`parameters_evaluation.yaml:137`）。座標下降只在折內的 query 上搜 δ*，折外那一半完全沒參與搜尋、只在搜完之後拿 δ* 去算一次 mAP。

這個切分是必要的：如果直接在全部資料上搜 δ*、又在全部資料上報收復量，等於拿同一份資料考自己的答案——δ* 會盡量去擬合這批資料裡的雜訊，收復量因此系統性偏高，跟訓練／驗證集分開的邏輯完全一樣。

**折別怎麼分**：折別＝query 鍵（`time`×`entity`）的 hash 分桶（CRC32，與資料列序無關），不是照列數精確切一半——`holdout_fraction`=0.5 是目標比例，實際折內／折外的 query 數會有出入（§10.6 示例資料是 317/337，不是剛好一半）。副作用：per-item δ* 在小樣本下對折別組成敏感（哪些 query 剛好落在哪一折會小幅影響搜到的值），讀方向與量級即可，不要摳到小數點後第三位。

§10.6 示例資料的落差就是這個效應的量：折內收復量 +0.0417（mAP(0)=0.5545 → mAP(δ*)=0.5963），折外收復量只有 +0.0295（mAP(0)=0.5281 → mAP(δ*)=0.5576）——折內比折外樂觀約 0.012，這個差就是「用同一批資料搜又考」多出來的虛高。**判讀一律看折外**：折外的收復量才是「這個 δ* 用在沒看過的資料上，指標真的能收復多少」的誠實估計。

### 10.5 收縮與「δ* 全零」的讀法

座標下降不是單純找「哪個 δ 能讓 mAP 最高」，目標函數多一個懲罰項：

```
目標 = mAP_fit − shrink_lambda · δ² / M
```

`shrink_lambda`（config `evaluation.diagnosis.offset_sweep.shrink_lambda`，預設 0.1，`parameters_evaluation.yaml:136`；程式碼 `offset_sweep.py:187`）把 δ 往 0 拉；M 是產品數。搜尋規則另外偏向 0：候選 δ 按 |δ| 由小到大掃（`offset_sweep.py:164`），**只有嚴格更好才換**（`offset_sweep.py:188`）——同樣的目標值，優先留在 0。兩件事合起來的效果：一個 δ 只有在「移動它帶來的 mAP 增益大到蓋過 λ·δ²/M 的懲罰」才會被搜尋選中；雜訊級的小增益推不動它。

所以**「δ* 全部搜出來是 0」是一個有意義的結論，讀作「這批產品的水準大致正確，抬高或壓低任何一個都不划算」**，不是「沒算出來」或「這個節點沒跑」。

反過來，centered δ* 明顯非零、且折外收復量為正（§10.6 示例表就是這樣：fund_bond centered +0.49、fund_stock centered +0.34，折外收復 +0.0295），代表真的存在收得回的缺口——但要誠實讀這個缺口的性質：它多半不是「模型校準錯了」，是**macro 指標本身在要求的再平衡**。把冷門產品（fund_bond、fund_stock 這類 n_pos 少的產品）整體推高，它們自己的 per-item AP 大幅上升，這個上升在 macro（等權平均）裡跟熱門產品的 AP 佔一樣的權重，代價則攤在熱門產品身上（被冷門擠掉幾個名次，AP 只掉一點點）——一賺一賠不對稱，等權平均下淨賺，指標淨升。

這跟 §1～§7 對帳表「水準都在理論帶內」**不矛盾，是兩把不同的尺**：對帳量的是**校準**——分數是否忠實反映實際購買率（母體效應扣除後）；offset sweep 量的是**對指標最優**——在「每個產品的 AP 等權算進 macro」這個規則下，怎麼平移分數能讓這個指標數字最大。一個產品可以校準完全正確（對帳可解釋）、同時又不是指標最優點（offset sweep 建議推高它）——兩個答案都對，只是問題不同。要不要真的吃下這個缺口（把冷門產品系統性推高）是排序策略的取捨，不是可以從數字直接讀出對錯的問題；相關的取捨光譜見 `docs/ranking-diagnosis-framework.md` Ch 6（item 權重怎麼選：macro～micro 光譜）。

**「δ* 幾乎不動」不代表沒有水準錯位，可能是傷害還不夠痛**：座標下降只在乎「移動 δ 帶不帶得動 mAP_fit」，帶不動的原因有兩種，讀法完全不同——一種是這個產品真的沒有水準錯位；另一種是它有水準錯位，但錯位對這批 query、這個等權指標造成的實際傷害，小到蓋不過 §10.5 開頭那個 λ·δ²/M 的收縮懲罰，座標下降因此正確地選擇不動它。第二種最容易發生在**熱門、高基率的產品**身上：它自己因為錯位而占到更多 top 名次，自身 AP 反而上升，抵銷了它壓給別人的損失，一漲一跌在 macro 等權平均下幾乎互相抵銷——這種產品即使真的被系統性推高，δ*≈0 仍然是座標下降給出的正確答案，不是它沒事。要分辨是哪一種，光看 offset sweep 這張表看不出來——同一批資料上「有沒有錯位」與「錯位造不造成傷害」是兩件事，後者要換一把尺，見 §11 的壓制帳本：它不管划不划算，把每一筆排錯的名次都如實記下來，兩張表併看才看得出「sweep 靜默、帳本卻在暴增」這種訊號。

### 10.7 waterfall 讀法

waterfall 圖（report.html 「分流 Offset sweep」段落的圖，`_offset_sweep_waterfall`，`report_builder.py:561-602`）把 §10.6 per-item 表的折外 LOO 貢獻疊成一條路徑：

- **起點柱**＝折外 mAP(0)（絕對值，示例資料是 0.5281，灰色 `totals` 柱）。
- **中間每根柱**＝一個產品的 δ* 平移在折外貢獻的收復量（`loo_contribution_holdout`，§10.6）——只有 δ*≠0 的產品才有柱（δ*=0 的產品沒動、自然沒有貢獻可畫，示例資料的 fund_mix 就是這樣），柱依貢獻絕對值由大到小排列。藍色＝正貢獻、橘色＝負貢獻——顏色只代表這根柱把 mAP 往上還是往下推，不代表 δ* 本身是正是負（exchange_usd 是被推**低**、卻貢獻正的收復量，柱子一樣是藍色，見 §10.6）。
- **交互殘差柱**＝§10.6 算過的加總差額（示例資料 −0.0289，這次是負值）——δ* 是聯立搜出來的，各產品的移動互相牽動彼此的名次，一次拿掉一個算的貢獻不保證加總等於同時移動全部的效果，這根柱把差額補平，讓 waterfall 的終點對得上；柱子往下代表這次是「加總高估、聯立效果打折」，柱子往上代表「加總低估、聯立效果加乘」，兩種都會發生（§10.6）。
- **終點柱**＝折外 mAP(δ*)（絕對值，示例資料是 0.5576，灰色 `totals` 柱）。

圖上刻意沒畫的東西：**mAP(δ*) 之上，到「這批資料理論上能到多高」之間的距離**——也就是平移完全收不回的那塊，是條件判別力缺口。這個上限是未知的（沒有一個乾淨的公式能算出「這批資料的模型理論最高分」），所以圖上留白，不是漏畫；讀法是「waterfall 終點到頂之間有多遠，就有多少缺口只能靠動訓練解決」，具體要動哪個訓練槓桿回 `docs/ranking-diagnosis-framework.md` Ch 5。

## 11. 壓制帳本（pair ledger）

壓制帳本是分流層（§10）之外的第四張診斷產物——`pair_ledger.json`（同一個目錄：`data/evaluation/<model_version>/<snap_date>/diagnosis/`），report.html 的「壓制帳本 Pair ledger（誰壓了誰、代價多少）」段落把它畫成 heatmap＋三張表。offset sweep 回答「值不值得整體平移」；壓制帳本回答一個更基本的問題：**不管划不划算，誰的負例排在誰的正例上方、每一次這樣的排錯讓指標付出多少代價**。

### 11.1 在回答什麼問題

§8.4 象限表的 `suppression_count` 已經記過一次「壓制」，但範圍窄得多：它只算「以負例身分排在該 query **首位**正例上方」的次數（`occupancy_spark.py:57-84`）——同一個 query 若有兩個正例，只有排在最上面那個之前的負例才算數，而且不管交換名次對指標的影響有多大，一次都只計 1 次。壓制帳本把範圍打開到**每一個**正例（不只首位）、每一個排在它上方的負例，而且不論次數改論代價——同一個 pair 排錯，交換名次讓 AP 掉得越多，這裡的帳就記得越重。兩張表因此不是同一件事的兩種算法，是同一個現象（誰的負例排在誰的正例之上）的兩種取樣範圍與計價方式：`suppression_count` 是**次數**、粗粒度、只看首位；壓制帳本是**指標幣值**（\|ΔAP\|）、細粒度、看全部正例。

**λ 會計的出身**：\|ΔAP\| 的定義借自 pairwise 排序訓練法（以 lambdarank 為代表）的一個常用技巧——對每一個排錯的 pair，算一個「如果把這兩筆的名次交換，排序指標會變多少」的量（λ 梯度），拿來當這個 pair 在訓練時該被推多用力的訊號。壓制帳本用的是同一個定義，但**只記帳、不拿去更新任何模型**——它不是訓練迴路的一部分，純粹是「如果把每一對排錯的名次都扶正，指標各自變化多少、加總起來多少」的事後盤點，所以叫「記帳不訓練」。

### 11.2 |ΔAP| 的定義與數感

把手算例完整走一遍。假設某個 query 只有三個候選，依分數排出來的名次是：

```
rank 1: B（負例）
rank 2: A（正例）
rank 3: C（正例）
```

這個 query 對 mAP 的貢獻，是每個正例「在它自己名次上的累積精度」加總：A 在 rank 2，前 2 名裡有 1 個正例（A 自己）→ 貢獻 1/2；C 在 rank 3，前 3 名裡有 2 個正例（A、C）→ 貢獻 2/3。原始總和 = 1/2 + 2/3 = 7/6。

**把 A 與 B 交換名次**（A 升到 rank 1、B 降到 rank 2）：A 現在前 1 名裡有 1 個正例 → 貢獻 1/1；C 還在 rank 3、前 3 名裡一樣是 2 個正例 → 貢獻仍是 2/3。新總和 = 1/1 + 2/3 = 5/3。

**ΔAP = 5/3 − 7/6 = 0.5**——這就是（壓制者=B、受害者=A）這個 pair 記的帳。

**把 C 與 B 交換名次**（C 升到 rank 1、B 降到 rank 3，A 留在 rank 2 不動）：C 現在前 1 名裡有 1 個正例 → 貢獻 1/1；但 A 雖然名次沒動，它前面現在多了一個正例（C）而不是負例（B），所以 A 的累積精度也跟著漲——A 從「前 2 名 1 個正例＝1/2」漲到「前 2 名 2 個正例＝2/2＝1」，多出 1/2。新總和 = C 的 1/1 ＋ A 多出的 1/2 ＋ A 原本的 1/2（因為 A 自己名次沒變，contrib[b] 這項本來就要扣掉再補回）——整理成帳本實際用的公式：**ΔAP = 新位置貢獻 − 原位置貢獻 + 中間正例的外溢** = 1/1 − 2/3 + 1/2 = **5/6**。

同樣都是「把一個負例跟一個正例交換名次」，量出來的代價卻不同（0.5 vs 5/6≈0.833）：C 被壓得更深（B 跟它中間隔著另一個正例 A），交換名次不只讓 C 自己受益，還帶動 A 一起受益（因為 A 前面的正例密度也跟著變高了）——**外溢項就是在算這個連帶效果**。這正是壓制帳本要記的「代價」，不是一個簡單的名次差。

**k 會改變帳本的計價單位**：如果指標只看 k=1（只在乎誰在第一名），上面兩對交換都變成「把一個正例抬進 top-1」，ΔAP 恆等於 **1.0**——因為不管交換發生在哪個名次區間，只要交換的結果是把正例送進 top-1，帳面上就是「從 0 分變滿分」的同一件事。實務含意：`evaluation.metric.k` 設多少，帳本的每一筆代價都跟著變，不是換了 k 帳本的相對排序就一定不變（§6 欄位對照表也是同一套 metric_params，判讀時對照你自己 config 的 k）。

### 11.3 矩陣與邊際表讀法

`matrix` 的結構是 `matrix[壓制者][受害者] = {pair_count, dap_sum}`——第一層鍵是壓制者（同 query 排在正例上方的負例 item）、第二層鍵是受害者（被壓的正例 item）。heatmap 的縱軸（y）是壓制者、橫軸（x）是受害者，顏色深淺是 `dap_sum`、hover 上附 `pair_count`。

**同 query 也是正例的 item 不記帳**：壓制者的資格只看它在該筆的 label 是不是負例——同一個 query 裡如果客戶同時買了基金跟信用卡，基金正例排在信用卡正例前面，信用卡不會被記一筆「被基金壓」的帳。買了基金的人 top-1 剛好是他也買的信用卡，對這位客戶而言不是傷害（他兩個都想要），帳本只記「正例被排在它上方的負例壓下去」這一種情境。

`by_suppressor`／`by_victim` 是矩陣沿兩個軸各自的邊際加總，附 `dap_share`（占全部 \|ΔAP\| 的比例）。矩陣本身的單一格也值得看：這批示例資料裡最大的一格是 `exchange_usd` 壓 `exchange_fx`（`dap_sum`=47.72、`pair_count`=101）——兩個外幣類產品彼此排序糾纏最深，壓制帳本能看到 `by_suppressor`／`by_victim` 這種聚合表看不到的「誰跟誰」的細節。

### 11.4 substitution ablation 讀法

對每個 item：把它在**全部**列的分數換成同一個常數——這個 item 在診斷抽樣裡的 base rate（正類率）取 logit（`base_logit`），重算一次 macro mAP（`map_substituted`），跟現狀 `map_current` 相減得到 `delta_vs_current`。

**讀法**：`delta_vs_current` **正**＝把這個 item 換成一個不分青紅皂白的常數，指標反而更好——代表它現在的個性化分數是**淨傷害**；**負**＝換成常數指標變差——代表它的個性化分數是**淨貢獻**。這跟直覺可能相反：常數分數表示「完全不個性化、每個人看到的都一樣」，如果連這樣都比現在好，說明現在的個性化分數不是在幫忙，是在扯後腿（通常是因為它常態性地排得比它該排的位置高，壓到了同 query 裡其他 item 的正例）。

**只做「換掉它」方向**：反向 ablation（只留這個 item 用個性化分數、其餘全部換成常數）沒有實作——所以這張表只能回答「拿掉這個 item 的個性化會不會更好」，回答不了「只留這個 item 的個性化夠不夠」。

**base rate 的誠實限制**：`base_rate` 是這個 item 在**診斷抽樣**（不是全量）裡的正類率，樣本小的 item 這個常數本身就有抽樣誤差，讀法同 §9 對小 `n_pos` 的提醒。

### 11.5 by_segment 讀法

對每個正例列，先把它自己受到的傷害加總（同一列身為受害者、被壓過的所有 pair 的 \|ΔAP\| 加總），再按 `evaluation.segment_columns` 設定的欄分組——回答「傷害集中在誰身上」，是自動偵測傷害聚集的一個簡化版：只能看你設定過的分群欄，不會自己去發現「原來是某個沒設過的欄在分群」。

`n_suppressed_pos_rows` 是這群裡「至少被壓過一次」的正例列數，除以 `n_pos_rows` 得到**受害發生率**；`dap_share` 是**傷害總量占比**——兩者答的問題不一樣，讀表時要並看，不能只看其中一個就下結論。

### 11.6 示例走讀

**壓制者邊際**（`dap_sum` 降冪）：

| item | pair_count | dap_sum | dap_share |
|---|---|---|---|
| exchange_usd | 342 | 192.79 | 41.5% |
| ccard_ins | 340 | 158.99 | 34.2% |
| exchange_fx | 210 | 49.92 | 10.7% |
| ccard_bill | 146 | 41.18 | 8.9% |
| fund_stock | 123 | 12.46 | 2.7% |
| ccard_cash | 60 | 6.48 | 1.4% |

**substitution ablation**（`delta_vs_current` 降冪）：

### 11.7 與 offset sweep 的關係

分流閥（§10）給的是「水準缺口總量與處方」——一個產品該不該整體平移、平移能救回多少指標；壓制帳本給的是「誰壓誰」的明細——不管划不划算，如實記下每一筆排錯的代價。兩者用同一份診斷抽樣、同一套 metric_params，卻可能給出看似矛盾、實則互補的讀法，下面用一個可自行重現的注入實驗說明。

對 `ccard_ins` 的 logit 分數注入 +1.0，模擬它整體被推得更高。做法：在 `conf/base/parameters_evaluation.yaml` 的 `evaluation.diagnosis` 下設

```yaml
debug_inject_offsets: {ccard_ins: 1.0}
```

重跑 `evaluation` 即可（驗證測試專用鍵：只影響分流層與壓制帳本兩個診斷節點的計算，不碰模型與其他產物；用完清回 `{}`）。效果上類比 §7 的對帳層注入——都是刻意製造一個已知的水準錯位——但機制不同：§7 是改抽樣配置後**重訓**，這裡是評估診斷期直接把常數加在 log-odds 分數上，不用重訓。注入後：

1. **壓制帳本如實暴增**：`ccard_ins` 的 `dap_sum` 從 158.99 漲到 314.74（幾乎翻倍）、`dap_share` 從 34.2% 漲到 61.8%、`pair_count` 從 340 漲到 471——它排得更前面，壓到別人正例的次數與代價都線性反映出來，這正是帳本的本分：不判斷划不划算，只記錄發生了什麼。
2. **但這次水準錯位對全樣本 macro per-item mAP 的實際損傷只有 +0.0034**——小到幾乎量不到。原因：`ccard_ins` 自己的正例因為排名整體上移，自己的 AP 也跟著漲，這個自漲的部分抵銷了它壓給別人的損失，一漲一跌在 macro 等權平均下幾乎打平。
3. **offset sweep 正確地不出手**：收縮懲罰在 δ=−1.0（把注入的量還原）時是 0.1×1²/8=0.0125，大於實際損傷 0.0034——座標下降判斷「還原這個位移換來的指標增益，蓋不過收縮的代價」，δ* 幾乎不動。這不是 sweep 遲鈍，是它誠實：對這個等權指標而言，這次注入的水準錯位確實沒有造成值得動手術的傷害。
4. **把收縮拿掉（`shrink_lambda`=0）重跑，sweep 就會反制**（δ*≈−1.3）——證明 sweep 不出手是收縮門檻的結果，不是它偵測不到位移本身；有懲罰時「划不划算」蓋過「有沒有位移」。

**這正是「sweep 靜默＋帳本暴增」這個組合讀法的價值**：兩張表對同一個水準錯位給出方向相反的訊號——sweep 說「不用動」，帳本說「壓制翻倍」——單看 sweep 會漏掉這個錯位造成的具體排序傷害（哪怕它沒有拖累整體指標，對被壓下去的那些客戶跟產品組合來說仍是真實發生的事）；單看帳本又會誤以為這是一個「划算」去修的問題，忽略了它對整體指標其實無感、修正的邊際報酬很低。兩張表沒有誰比較對，讀法是：**sweep 靜默但帳本暴增，代表這是一個「有排序後果、但對這個等權指標不痛不癢」的水準錯位**——要不要處理是排序策略的取捨（同 §10.5 最後一段），不是指標能替你決定的事。

**看到壓制者邊際表第一名之後，依序查**（比照 §6 的用法）：

1. 矩陣單一格（§11.3）：它主要壓在誰身上——傷害集中在一兩個受害者，還是散在全場。
2. 交叉購買矩陣（§8.7）：被壓的正例，買家是不是大機率也會買壓制者——是，傷害打折；否，才是實質傷害。
3. substitution ablation（§11.4）：它的個性化分數是淨貢獻還是淨傷害——淨貢獻的高壓制者（像示例的 exchange_usd）通常是健康的熱門產品，硬壓它得不償失。
4. 回看 offset sweep 的 centered δ*（§10.6）：sweep 也點名要壓低 → 指標認為值得修；sweep 靜默 → 本節上面那種「有排序後果、指標不痛」的型，處理與否是策略取捨。

## 12. 已知限制

- 產品層理論帶是跨客群聚合近似（§3.3）；cell 級精確值在 JSON。
- gap 的 logit(平均) 是近似（§4.3）。
- 全局參考值依賴「config 中性產品夠多、且自身沒有病灶」（§5.2）。
- 生產規模（例：22 產品 × 10M query）的計算成本與 override 條數很多時的表現未實測，部署到生產前需另行驗證。
- 對帳只看「水準」（level）；「產品內排序品質」是另一軸的診斷，見 §8 的象限層（方法根源見 `docs/ranking-diagnosis-framework.md` 的行為層象限）。
- within-item AUC 算的是同一批評估資料——跟 gap 一樣是母體條件化之後的母體（§5.1），不是全體客戶；不同產品間比較 AUC 高低沒問題（AUC 從不跨產品），但拿它去推斷「這個產品在全體客戶裡的判別力」會犯跟 gap 一樣的母體錯位。
- `ap_sampled` 跟 §9 的 `ap` 同源，來自診斷抽樣，不是全量——小 `n_pos` 的產品這顆數字連同它背後的 CI 都不穩，見 §9 的提示。
- 交叉購買矩陣的母體是完整 label 歷史，與同節其他欄（單一評估視窗）不同母體（§8.7）——併看可以，互除不行。
- `auc_threshold`（0.6）與 `gap_band`（0.35）是起手值，不是統計推導出的定論——語意上跟 §5.4 `explained_threshold` 的夾擠論證（太緊會追雜訊、太鬆會漏掉真正的效應）同一套邏輯，但作用對象不同：這裡調的是「行為觀測」的門檻，不影響對帳層的歸因判定。
- （a）offset sweep 計算成本 ≈ 輪數 × 產品數 × 格點數 次指標評估：參考量測，8 產品×654 query 約 6 秒；22 產品×20 萬 query（診斷抽樣上限 `max_queries`，非全量，見 (d)）外推，每次評估落在秒級、總計可達數小時——先把 `grid.step` 調粗（0.05→0.1，格點數約砍半）再跑大樣本。
- （b）per-item 折外 LOO 貢獻是逐一拿掉重算（leave-one-out），不保證加總等於總收復量，差額在「交互殘差」（§10.6/§10.7）。
- （c）分數整欄超出 (0,1) 時退回原始分數尺度平移，不做 logit 變換（產物 `notes` 會註記；實作見 `diagnosis/metric/_common.py:35` 的 `to_logit`）。
- （d）offset sweep 的全部數字是診斷抽樣上的估計，不是全量；樣本規模看產物對應的 `sample` 配置區塊（`evaluation.diagnosis.sample`，同 §9 CI 的抽樣機制）。
- offset sweep 的折別是 query 鍵 hash 分桶的近似比例，不是精確對半切；per-item δ*（尤其 centered 值）在小樣本下對折別組成敏感，讀方向與量級，不要摳到小數點後第三位（§10.4）。
- 壓制帳本 \|ΔAP\| 是 query 層 AP 貢獻和的精確變化量，不是 macro per-item mAP 的全式變化（per-item 分母跨 query）——判讀只用相對量比較，不要拿它反推 macro mAP 本身會變動多少（§11.1）。
- substitution ablation 只做「把這個 item 換成常數」方向；反向（只留這個 item 用個性化分數、其餘全部換成常數）未實作（§11.4）。
- 壓制帳本的 base rate（substitution）與傷害統計（by_segment）都取自診斷抽樣，不是全量——小 `n_pos` 的產品這些數字一樣不穩，讀法同 §9（§11.4/§11.5）。
- （e）壓制帳本計算成本：參考量測，本機約 0.83 秒（§11.6 走讀那次，654 query×8 產品的診斷抽樣規模）；22 產品×20 萬 query 外推，pair 枚舉約 4–5 分鐘、substitution 另計約 1 分鐘——量級遠低於 (a) 的 offset sweep，但抽樣上限一樣是主要槓桿，緩解手段同 (a)：降 `max_queries`。
