# 第 09 章審稿：進階 analytics engineer 視角
## 人設：reverse ETL 從業者（特徵庫 + 名單回推業務端）

審稿日：2026-06-23

---

## 逐節深度評

### §9.1 問題定位（pull vs push 的分野）
**評：夠營運。**

pull/push 二分清楚，flowchart 幫助定位。對剛接觸 reverse ETL 的人是很好的第一頁。能 pull 就不 push 的原則也是我實務中不斷重複說的話，寫進來很值得。

**缺口：無**（本節定性為入門框架，不要求深）

---

### §9.2 推送通道（JDBC/檔案/NiFi/Sqoop/API）
**評：偏淺——觸到表面，但遺漏三個在我工作中最常卡住的課題。**

#### 已覆蓋且有用的部分
- `numPartitions` 說明對第一次寫外部 DB 的人是關鍵提醒，「瓶頸與風險都在對方」那一段方向對。
- API 場景的「落地後獨立步驟推送」原則適合受限環境。

#### 缺口一：增量 vs 全量的選擇依據（最核心、完全缺席）

本章只示範「整張名單全批寫進去」。但在真實業務場景裡，這個選擇本身就是個會踩雷的設計決定：

- **全量覆寫（JDBC overwrite / 先 DELETE 再 INSERT）**：語意乾淨，但對大表會鎖表、對 OLTP 衝擊大、寫入窗口長。
- **增量 delta 推（只推本次新增或有變動的列）**：對大表友善，但你得在 Spark 端先算出「哪些客戶今天跟上週不一樣」——這需要比對邏輯，而且一旦比對出 bug，靜默遺漏（沒推）比多推更難發現。
- **選依據**：量（30 萬 vs 1000 萬）、目標端鎖粒度、對方系統能不能接受短暫髒讀、以及「客戶離開名單要不要在對方端刪掉」（soft delete vs 真刪）。

這個選擇本章沒有給任何指引，但它直接決定後面的冪等設計（§9.3）和失敗恢復策略。

#### 缺口二：回壓與節流（我每次接新業務系統都會踩）

「`numPartitions` 要保守，先問 DBA」說的是並行連線數，但還有另一個維度：**對方系統的 rate limit 或批次窗口**。

行銷自動化平台、簡訊閘道、外撥系統幾乎都有：每分鐘最多接多少筆、每日最大配額、非營業時間才收、每批最多幾千列。這些約束不是「有沒有連得到」的問題，而是對方收了、但超出它能消化的速度就丟掉或排很深的佇列。Spark 沒有內建的速率控制（`sleep` 在 driver 端可以，在 executor 端不可靠）——「落地交換區 → 獨立控速步驟推」的模式就是為此存在，但本章沒有把這個理由說清楚，只說「API 就落地後推」，沒說為什麼比直接打更安全。

#### 缺口三：多目的地一致性（最難說「全送成功」的場景）

名單常常要同時送多個下游：行銷系統＋CRM 各一份，或同一份名單存多個環境（正式＋備份）。

- 推到 A 成功、推到 B 失敗，要怎麼辦？A 已經有新名單、B 還是上週的——業務端看到的是不一致的版本。
- 這不是 upsert 能解的問題，這是「多目的地事務」的問題：Hadoop 外面沒有分散式事務，你只能靠「兩段式確認 + 補推 + 告警」。
- 本章沒有提到多目的地場景，但 flowchart 裡已經畫了 DB＋CRM 兩個目標——讀者看到圖會自然問這個問題。

### §9.3 冪等與重試（目標端 upsert）
**評：夠營運——這節是全章深度最紮實的一節。**

業務鍵 + upsert 的框架講清楚了，MySQL/PostgreSQL/Oracle 三家語法並列實用，「append 當重試」的反例提示也很精準。

**缺口：部分成功的處理方式**

upsert 解決的是「同一批推兩次不重複」，但沒有解「這 30 萬筆裡有 5 萬筆因為 key 格式對不上而 reject，25 萬筆成功了」的情境。

這叫**部分成功（partial success）**，對 JDBC 批次寫入特別常見（對方 DB 有 check constraint，你的資料有幾百筆違反）。Spark 的 JDBC sink 遇到 batch 裡任何一筆報錯會整批失敗——這個行為讀者不一定知道。「排程視為整批失敗 → 重推全批」在有 upsert 的情況下安全，但如果問題是資料本身違反對方 constraint，重推還是會失敗，這時沒有「跳過壞列繼續」的內建機制，需要先把壞列過濾掉或單獨處理。

這個場景本章完全沒提，但在我的工作裡每隔幾個月就會碰到一次。

---

### §9.4 就緒閘與發佈（接 §8.6）
**評：夠營運——正確且有必要。**

「只推驗過且已發佈版本」的強調、DAG 相依設計（算→驗→發佈→推）都對，「推出去收不回來」的警語也正確落在這裡。

**缺口：發佈狀態的一致性（跨環境或跨批次的邊界）**

本章描述的是「這批驗過才推」，但沒有處理「我現在要推的發佈版本，和上次推的之間，有沒有跳號（missing batch）」的情境。

舉例：週一 v1 推成功、週二因為閘沒過擋住了、週三 v3 推成功——對方業務系統的這張表現在有週一的 v1 資料和週三的 v3 資料，中間週二的更新永遠沒進去。對有些業務語意（例如「每週累計某指標」）這是個隱性錯誤，光靠「有發佈才推」抓不到。這不一定要本章解決，但值得一句話提醒讀者這個邊界存在。

---

### §9.5 PII、Ranger 與稽核
**評：夠營運（方向對），但深度停在原則層，缺實操層。**

「遮罩在 Hadoop 內做、稽核要留痕」的方向正確。

**缺口一：schema drift 的 PII 再曝光風險**

欄位遮罩是在推送前把某幾欄去掉或代碼化。但如果下游介面（§9.6）在某次改版後多了一個欄位、而你的推送作業沒有明確的欄位白名單，這個新欄位就會自動出現在推送結果裡——可能是 PII。

防法：推送的 SELECT 清單要**顯式枚舉白名單欄位**，不用 `SELECT *`。本章沒有提這個具體的 footgun。

**缺口二：稽核留什麼才夠查**

§9.5 說要「留紀錄」，§9.7 範例寫「記下推了幾筆、到哪、誰觸發」。但實務上出事時要查的問題更具體：「這位客戶是哪個版本的名單推進去的、推送時間點是什麼、當時的分數是多少？」——這需要推送端留下 row-level 或 batch-level 的版本指標，不只是筆數 + 時間戳。

稽核的粒度設計本章沒有給。

---

### §9.6 介面對齊（格式與 key）
**評：夠營運——這節很實用，直擊最後一哩的細節坑。**

主鍵對齊、型別差異、編碼、批次大小都點到了，「介面約定雙方認」的原則也正確。

**缺口：schema drift（對方改欄、你的推送掛掉）**

本章提到「介面變更要審慎，你改了名單欄位對方匯入可能掛掉」——這是「你的變更影響對方」的方向。但還有反方向：**對方業務系統改了他的 DB schema（改欄名、改型別、加 NOT NULL constraint），你的推送作業下次跑到才發現**。

這叫 **schema drift**，發生在你沒有版控那張表的情況下（反正是對方的 DB）。對策是推送前先做一次 schema 探測（查對方 DB 的 `INFORMATION_SCHEMA` 或乾跑第一批小量資料），發現不符就提前告警、不硬寫。這個議題本章沒有觸及，但它是 reverse ETL 比一般 ETL 更難維運的核心原因之一。

---

### §9.7 貫穿範例
**評：夠清楚——把本章流程串成一個具體 story，節點清晰。**

六步驟走法對讀者很有幫助，特別是明確指出「稽核」是第六步（有時工程師會做 1–5 但忘記 6）。

**小觀察（非缺口）**：範例的第 4 步「推送前遮罩」選擇在推送 task 裡做，而不是在發佈時固化一個已遮罩版本——這是個合理取捨但值得一句話解釋（「遮罩在推送端做，是為了同一份原始資料可以推給不同系統送不同欄位子集」）。

---

### §9.8 取捨
**評：夠營運。**

push vs pull、即時 vs 鬆耦合、平行度取捨這三點都是實務中真正的決定軸。「交換區常常是更穩的預設選擇」這個結論我完全認同。

**缺口：增量 vs 全量的取捨（在 §9.2 已指出，這裡再次缺席）**

§9.8 列了三個取捨維度，但「什麼時候應該做增量推、什麼時候做全量推」這個決策軸沒有出現，而它在實務上和前三個同等重要。

## 缺口三級彙整

### 第一級（不補，讀者會卡住的）

1. **增量 vs 全量選擇依據**（§9.2 缺、§9.8 也缺）：全章只示範全量覆寫，但增量 delta 推是大表的標準作法——沒有討論怎麼選、各有什麼陷阱（遺漏 vs 鎖表）。
2. **Schema drift 的防禦**（§9.6 缺）：對方改 DB schema 你不知道，推送才爆——這是 reverse ETL 最常見的靜默維運事故，本章完全沒有提對策。
3. **部分成功（partial batch reject）**（§9.3 缺）：JDBC 遇到 constraint 違反整批失敗、或成功了 n% 的處理方式，本章沒有觸及。

### 第二級（補了加分，不補可接受但有說明缺口）

4. **回壓與節流**（§9.2 缺）：API/平台的 rate limit 和收批窗口約束，是「落地後獨立推」的真正動機，本章沒說清楚。
5. **多目的地一致性**（§9.2 缺）：flowchart 已畫了兩個目標，但沒討論一個成功一個失敗的處理策略。
6. **稽核的粒度設計**（§9.5 缺）：留「多少筆」不夠，出事時需要「哪個客戶是哪一版推進去的」，row/batch-level 版本指標怎麼設計。

### 第三級（工具層細節，可單獨查文件，本章不必涵蓋）

7. PII 白名單欄位顯式 SELECT（而非 `SELECT *`）——§9.5 可一句話帶過。
8. 發佈版本跳號（§9.4）——值得一句話提醒讀者這個邊界存在。

## 整體判斷

本章作為 reverse ETL「入門定向」章，§9.3（upsert）和 §9.4（就緒閘）是全章最紮實的兩節，清楚回答了「怎麼保證冪等」和「什麼時候才能推」。§9.5 和 §9.6 方向對但停在原則層。

**對「長期、受控地把資料推回業務系統」的讀者而言，本章解決了第一次設計的入門問題，但在維運一段時間後最常踩的三個坑（schema drift、增量/全量選擇、部分成功處理）是完全缺席的。** 如果手冊的目標是「可以當工作中的參考手冊翻」，這三個缺口會讓這章在實際維運事故時幫不上忙。如果目標是「建立第一次設計的決策框架」，則現在的深度對初學者已夠。

建議定位：**入門定向夠，維運參考手冊尚不足。**
