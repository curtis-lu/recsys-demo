## (a) 每章 Diátaxis 分類表

> 判準：tutorial＝帶著做的學習路徑｜how-to＝解決特定問題的步驟｜reference＝查得到的事實表｜explanation＝建立理解的敘事。
> 本手冊**整體定位＝參考手冊（reference handbook）**，但骨架刻意做成「explanation 打底 → how-to 落地 → reference 速查」的混合，這對「自學型參考手冊」是恰當且高明的選擇（對標 High Performance Spark：解釋與操作交織）。下表標每章**主要象限**與**有無模式混淆**。

| 章 | 主要象限 | 次要 | 模式混淆判定 |
|---|---|---|---|
| 01 心智模型 | **explanation** | — | 無。純 explanation，零操作步驟混入，乾淨。教科書級。 |
| 02 Spark UI | **how-to**（診斷流程） | explanation | 受控。§2.2「怎麼打開」是 how-to、§2.7/§2.8 是 how-to 流程、§2.1/§2.10 是 explanation 框住；§2.9 是 reference 表。四象限齊備但用小節清楚切開，**非混淆**。 |
| 03 SQL 寫法 | **how-to** | reference | 受控。每招固定模板「原理(explanation)→before/after(how-to)→看 UI→取捨」是**刻意的單節內四象限編織**，名著常見手法；§3.5 補充表、§3.1 地圖是 reference。一致性極高。 |
| 04 Spark 設定 | **how-to** | explanation | 受控。§4.5「記憶體裡裝了什麼」是 explanation、§4.6/§4.8 是 how-to、§4.9 是收斂。混合受小節邊界控制。 |
| 05 儲存效率 | **explanation→how-to** | reference | 大致受控，**最易模式混淆的一章**：§5.2/§5.4 含多個「進階/補充」內嵌小節（Parquet 內部、DataNode/locality），是 explanation 深掘插進 how-to 主線——但都用 `### 進階：`／`### 補充：` 明確標籤隔開，屬「分層遞進」而非混淆。**唯一須留意**：§5.5 一節同時塞「成因/徵兆/解法 + REPARTITION vs COALESCE + 兩個實戰問題」三件事，密度偏高（見 (d)）。 |

| 08 營運(一) | **how-to** | explanation | 受控。「每節先通用原則(explanation) → 再落到 dbt/Airflow/cron(how-to)」是全章宣告的固定編排，分層清楚。 |

**結論（a）**：全 8 章**沒有一處典型的 Diátaxis 模式混淆**（解釋途中硬插一長串無關操作、或 how-to 裡塞大段離題原理）。混合都發生在「單節內刻意編織」或「用 `### 進階/補充` 標籤＋難度門檻分層」，符合 High Performance Spark / Databricks 的做法。這是這本手冊**教學結構上最強的一點**。

## (d) 章首前提 ＋ 章末導覽 ＋ 來源 footer 一致性

**章首「本章前提」**：8 章全部具備，格式高度一致（`> **本章前提**`＋已讀哪些章＋可假設懂什麼＋本章要幹嘛）。教學效果**優秀**：
- 02–09 章的前提都**精確列出前置章節的具體概念**（不只「讀過第 01 章」，而是「partition、shuffle、stage、task、spill、skew 的心智模型」），這正是 spec §10 reader subagent 的 PRIOR_CHAPTERS 機制落地，讓讀者自我校準。
- 05 章前提額外加一段「本章偶爾引用 02/04 但會當場給足脈絡，沒讀過不影響」——**處理「軟依賴」的示範**，避免嚇退跳讀者。Databricks 漸進式編排的好習慣。

**章末「一句話帶走」＋後續指路**：8 章全部具備，用 `> boxed` 式收斂主軸＋「接下來→第 X 章」分流。一致且有效。

**來源 footer**：每節末 `📚 來源`＋章末「資料來源與精確度說明」，8 章全部具備且極其紮實（標示意值、標方向正確無逐字數字、標版本對齊）。**這是教學「誠實性」的標竿**——對齊 handbook-writing-guide §5「結論誠實」與 §11「簡化要明說」。讀者能看出哪裡不該照單全收，這在技術手冊裡罕見地好。

**發現的不一致（教學效果瑕疵，非技術錯）**：
1. **第 08 章章末導覽方向箭頭錯**：寫 `*上一章 →* [07 ...]（撰寫順序暫緩）`，但 07 應是「上一章」用 `←上一章`，且 08 的實際上一章（就閱讀順序）是 06。同檔 06 章末也指向「下一章 → 07（撰寫順序暫緩）」。**「（撰寫順序暫緩）」是寫作鷹架洩漏進成品**（handbook-writing-guide §6 明列禁止：不要對讀者講你的寫作決定）。讀者讀到「暫緩」會困惑「我是不是少了一章」。建議：07/10/11 未寫期間，導覽直接跳過缺章指到實際存在的下一章，不要留「暫緩」字樣。
2. **index 與實況落差**：index 列 07/10/11，章末導覽也連向它們（如 02/06/09 指向「第 10 章」「第 11 章」），但檔案不存在。這是**斷掉的學習路徑指標**——讀者點擊會 404。屬覆蓋度（architecture reviewer 範疇），此處僅就「教學連續性」記一筆：在缺章補齊前，跨章「→ 第 10/11 章」的指路會讓自學者撞牆。

## (e) 跨章敘事弧（整本當一門「課」）

把已寫 8 章當一門課讀，起承轉合：
- **起（01–02）**：建立心智模型（01 explanation）＋學會量測（02 how-to）。「先理解機器在做什麼，再學會看它做了什麼」——這是**教學上最正確的開場**（對標三本名著皆以執行模型開篇），且 02 結尾的「先量再調」把後續所有章釘成「先診斷後處方」的方法論主軸。
- **承（03–05）**：三把調優槓桿——改寫法(03)、調設定(04)、改存法(05)。三章用同一句「少讀/少搬」串起（01 §1.10 立的主軸在 03 §3.1、05 §5.1 反覆回扣），**主軸貫穿性極強**。
- **轉（06）**：拉高一層——「這件事該不該用 Spark」。從「把 Spark 調快」轉到「選對引擎」，視角升維，轉得自然（章首前提明說「前五章都在用 Spark 做快，這章談上一層的決定」）。
- **合（08–09）**：從「跑一條查詢」升到「長期營運資料產品」——可靠(08)＋可信(09)。這是 spec §11「終極能力目標」的兌現，敘事弧從 ad-hoc 一路爬到 feature store 營運。

**敘事弧成立，且回扣（callback）做得好**：每章章首前提都回指前章結論，每章「少讀/少搬」「先量再調」「snapshot 切線」等主軸詞反覆出現，整本讀起來像**一條有主軸的學習曲線**，不是散裝的 tip 合集。這是優於多數技術手冊（含部分名著章節）的地方。

**敘事弧的斷點（皆因缺章，非已寫章節之過）**：
- **07 章缺席**造成弧上一個洞：03–06 多處 forward-ref「第 07 章（DataFrame API）」「要接 Python/ML→第 07 章」，但 07 未寫。06→08 之間（編號上隔著 07）讀者會感到跳號。
- **10（場景索引）/11（速查）缺席**：02/06/08/09 章末都把讀者導向「第 10 章看自己這類工作」「第 11 章速查」，這是課程的「複習與檢索」收尾，缺了等於課上完沒有總複習與工具箱。

> 註：07/10/11 屬覆蓋度/章序（architecture reviewer 範疇），此處只就「敘事弧連續性」點到——**已寫的 8 章彼此之間的弧是完整且優秀的**；洞都在未寫章。

## (f) 具體教學法改進建議（對標範本/框架）

1. **第 08 章補一個 capstone「把它全部串起來」節**（對標：spec §12.2 體例 + 其餘 7 章一致性 + High Performance Spark 每概念配 worked-example）。現況 08 是唯一沒有端到端走查的章，破壞體例對稱。建議：「一支 `cust_feature` 排程從首次上線到穩定營運」串起冪等→相依→回填→監控→維護五件事。
2. **移除導覽裡的「（撰寫順序暫緩）」鷹架字樣**（對標：handbook-writing-guide §6 不洩漏寫作鷹架）。缺章期間導覽直接跳到實際存在的鄰章。
3. **缺章的 forward-ref 暫時降級為「規劃中」或就近改指已寫章**（對標：Diátaxis—斷掉的指路會破壞 how-to/reference 的可信賴性）。例如「→ 第 11 章速查」在 11 章寫好前，可改指各章自己的速查表（§2.9/§8.7/§8.8/§9.7）。
4. **AQE 首次出現（§1.6）的縮寫鋪陳獨立一句**（對標：Learning Spark 2nd 對新縮寫的鋪陳節奏；spec §12.6 術語第一次出現就解釋——已做到「解釋」，可再優化「節奏」）。把 AQE 定義從 shuffle 三麻煩同段抽出、給它一個獨立的引介句，降低該處密度。屬微調。
