## §2.2 怎麼打開 Spark UI（在 CDP 上）（行 34–58）

這節資訊量很大，是我這種「沒碰過叢集」的人最需要、但也最容易卡的一節。它區分了 live UI（還在跑）和 History Server（跑完），方向正確。但好幾個叢集術語第一次出現沒給人話，我會接連卡住。

- **卡關 2（術語沒先定義，重要）**：「**port 4040**」「順延到 4041、4042…」——我會寫 SQL，但不一定知道 port（連接埠）是什麼、為什麼是個數字、我要把它打去哪裡。雖然後面說「在 CDP 上你通常不該直接連 4040」，但這個數字突然冒出來、又叫我別用，我會困惑「那你跟我講這個數字幹嘛？」。建議一句話說明 port 是什麼，或點明「4040 是底層原理，CDP 上你走下面的 YARN 路徑就好」。
- **卡關 3（術語沒先定義，重要）**：「**YARN 的 ResourceManager 網頁**」「點它的 **ApplicationMaster** 連結」——這是審查清單特別點名的兩個詞。第 01 章只在 §1.1 一個 `>` 引述裡提過「YARN（叢集的資源管家）」，但**沒講過 ResourceManager 和 ApplicationMaster**。對我來說這是兩個全新的專有名詞：ResourceManager 是一個網頁？一個服務？ApplicationMaster 是我那個查詢嗎、還是另一支程式？兩者什麼關係？我會卡在「我到底要先開哪個網頁、再點哪個連結」。這是本節最該補人話的地方——例如「ResourceManager 是 YARN 的總管網頁，列出叢集上所有正在跑的 application；ApplicationMaster 是你這一個查詢在 YARN 裡的代表，點它的連結會把你帶到這個查詢專屬的 Spark UI」。
- **卡關 4（術語沒先定義，中度）**：「**event log**（Spark 把 UI 要顯示的資訊寫進持久化儲存的功能）」——這裡其實有給括號解釋，算 OK，但「持久化儲存」對我又是一個新詞（是寫到 HDFS 嗎？寫到哪？）。可接受，但若改成「寫到磁碟存起來、之後還找得到」會更白話。審查清單點名 event log，這裡有當場解釋，達標。
- **卡關 5（術語沒先定義，中度）**：「**Spark History Server**」「**Cloudera Manager**」「**App ID**」——History Server 有說是「負責重建那個 UI 的服務」，OK。Cloudera Manager 給了括號「你們平台的管理介面」，OK。App ID 沒解釋，但從上下文（「點你那次查詢的 App ID 就進到它的 Spark UI」）我大概猜得到是「每次查詢的編號」。建議 App ID 補半句，因為我下一個問題是「我怎麼知道哪個 App ID 是我剛剛跑的那個？清單上一堆，我怎麼認？」——這對營運排程的人是真實痛點。
- **卡關 6（步驟不可操作，中度）**：live UI 那段只給了「從 YARN ResourceManager 點 ApplicationMaster 進去」一句話，沒有像 History Server 那樣的「1.→2.→3.」編號步驟。History Server 那段有編號（打開 Cloudera Manager → 點 Spark → 點 History Server Web UI），很好操作；但 live UI 那段相對含糊，我照著做不一定找得到。兩段詳略不對稱——live UI 反而是「查詢還在跑、想現場看」時最急用的，卻講得比 History Server 簡略。
- **概念圖（行 46–54 Mermaid）**：左右兩塊（還在跑／已跑完）對照清楚，能呼應正文。但圖裡同樣出現 ResourceManager / ApplicationMaster / Cloudera Manager / App ID 這些我還沒消化的詞，圖本身不會幫我解釋，所以圖的自明性取決於正文有沒有先把詞講白。**圖 OK，但依賴正文補術語。**
- **跨章引用**：「排程作業（第 08 章）」「第 08 章監控會再談」——對照 index，第 08 章是「營運資料排程與資料產品」，指對了。
- **我會這樣想／會問**：這節我會問最多問題：「port 是什麼、為什麼叫我別用 4040」「ResourceManager 和 ApplicationMaster 到底是兩個什麼東西、先點哪個」「History Server 的清單上幾百個 App ID，我怎麼認出哪個是我的」。這節是全章我最可能讀不下去、需要找同事問的地方。

## §2.3 跑之前先看計畫：`EXPLAIN`（行 62–87）

這節整體好讀。用法直接（在查詢前加一個字）、範例 SQL 沿用第 01 章 §1.8 那條 join+group by，有連貫感，這個好。三個關鍵字（Exchange / BroadcastHashJoin vs SortMergeJoin / PartitionFilters & PushedFilters）都有當場給人話，且每個都連回第 01 章的成本來源，達標。

- **小確認（術語有定義，達標）**：審查清單點名的 `Exchange`、`BroadcastHashJoin`、`SortMergeJoin`、`PartitionFilters`、`PushedFilters` 都在這節第一次出現、且都當場給了人話解釋（Exchange=一次 shuffle；BroadcastHashJoin=把小表廣播、不為 join 做 shuffle；SortMergeJoin=兩邊都 shuffle 再排序合併；PartitionFilters=只讀需要的分區目錄；PushedFilters=下推到讀檔層提早篩）。這節做得好。
- **卡關 7（缺脈絡，輕微）**：`EXPLAIN` 各模式（`EXTENDED`／`COST`／`CODEGEN`）只一句帶過，然後說「日常診斷用 `EXPLAIN FORMATTED` 就夠」。這個結論很好（不讓我糾結），但 `CODEGEN`（印產生的程式碼）對我是天書，我會想「產生的程式碼是什麼？我看那個幹嘛？」——其實作者的意思就是「你不用看」，那不如更乾脆：把 EXTENDED/COST/CODEGEN 收進一句「還有幾種更細的模式，日常用不到」，不必各自點名反而引我分心。屬可斟酌。
- **卡關 8（缺脈絡／不可操作，中度）**：三個關鍵字講得很好，但**我不知道它們在 EXPLAIN 的輸出裡長什麼樣、出現在哪一行**。正文老實說了「上面未貼逐字輸出；實際排版以你環境為準」（行 87 來源框與章末第 3 點），這個誠實我尊重；但對「這輩子沒看過 EXPLAIN 輸出」的我，連一個縮略的範例輸出片段都沒有，我會難以把「找 Exchange」這個動作對應到真實畫面。我會問「EXPLAIN 印出來是一大坨文字嗎？Exchange 那行大概長怎樣、我要往哪裡掃？」哪怕給三五行示意（標明是示意、非逐字），都會讓這節從「看得懂道理」變成「真的會做」。這是本章「步驟可操作性」最弱的一處。
- **我會這樣想**：道理懂了、三個關鍵字也記住了，但真打了 `EXPLAIN FORMATTED` 後，面對螢幕一大片輸出，我還是會慌——因為我沒看過它長怎樣。

## §2.5 從 SQL 頁籤看「這條查詢花在哪」（行 118–131）

聚焦兩個數字（output rows、shuffle bytes written total），方向清楚、好操作。把「join 後列數暴增＝爆量 join」講成具體訊號很實用。

- **卡關 10（術語沒先定義，中度）**：算子清單裡出現「**HashAggregate**」（行 120：「每個方塊是一個算子（Scan、Filter、Exchange、HashAggregate、BroadcastHashJoin…）」）——這是審查清單特別點名的詞，且它在這裡第一次出現、**沒有任何人話解釋**。Scan/Filter/Exchange/BroadcastHashJoin 前文都解釋過或顧名思義，但 HashAggregate 對我是全新詞。我會問「HashAggregate 是什麼？是 GROUP BY 嗎？」——其實它就對應 `GROUP BY`/聚合，但正文沒講，我只能瞎猜。建議當場補半句（「HashAggregate＝做 GROUP BY 聚合的算子」）。這是本章漏接的關鍵術語之一。
- **卡關 11（缺脈絡，中度）**：又是「我沒看過 SQL 頁籤長怎樣」的問題。正文說「方塊上掛著實際數字」，但沒有任何畫面示意（連 ASCII 草圖都沒有），我難以想像那張 DAG 圖實際長相、數字掛在方塊的哪裡。與 §2.3 同病：道理懂、畫面陌生。一張極簡示意（哪怕標「示意」）會大幅幫助。
- **卡關 12（太抽象，輕微）**：「某個 `Filter` 之後列數從 3000 萬掉到 3 萬，很正常」——這個有具體數字，好。但「某個 `Join` 之後列數暴增（比輸入還多很多）」只有形容詞「暴增」「多很多」，沒給數字。對照寫作指引 §3「每個抽象主張都要有具體數字落地」，這裡可補一個小數字（例如「輸入 3000 萬、join 後變 9 億，就是一對多炸開」），讓我體會「暴增」的程度。
- **AQE 提醒（行 129 `>` 框）**：「EXPLAIN 看到的是還沒被 AQE 改過的計畫，最終計畫要在 SQL 頁籤才看得到，兩邊對不起來是正常的，以 SQL 頁籤為準」——這個提醒非常好、解掉了我一定會踩的坑（「我 EXPLAIN 看到 SortMergeJoin，怎麼 UI 變了？」）。AQE 在第 01 章 §1.6/§1.10 已介紹過，這裡沿用沒問題。讚。
- **我會這樣想**：知道要看 output rows 和 shuffle bytes written total 了。但 HashAggregate 沒解釋讓我卡一下；而且我還是沒看過 SQL 頁籤的真實長相。

## §2.6 從 Stages 頁籤看「這個 stage 慢在哪」（行 135–170）

這節是全章操作性最強、最讓我覺得「我真的會用」的一節。把 Summary Metrics 用分位數呈現、再教「Max ≫ Median＝skew」「spill 非零＝記憶體不夠」兩條判讀法則，具體、可操作。Mermaid 對比圖（正常 vs skew）也直觀。

- **卡關 13（術語沒先定義，中度）**：「**分位數**」「**25th percentile（前 25%）**」「**Median（中位數）**」「**75th percentile**」——審查清單點名 percentile/分位數。正文把 25th/75th 用括號「（前 25%）（前 75%）」帶過、Median 給了「（中位數）」。對「會 SQL、但沒統計底」的我，「中位數」我懂；但「前 25%」這個說法其實有歧義——是「最小的 25% 裡最大的那個」嗎？我會卡「25th percentile 到底是哪個值」。中位數我能直覺（一半以上一半以下），但 25th/75th 用「前 25%／前 75%」反而讓我更模糊（「前」是由小到大還是由大到小？）。建議用一句白話釘死：「把所有 task 的這個指標由小到大排隊，25th percentile＝排在前四分之一位置那個 task 的值；Median＝排中間那個」。這樣我才真的會讀那張表。
- **卡關 14（判讀很好，達標）**：「Median 3 秒、Max 5 分鐘＝skew」「Max ≫ Median 就是 skew 的招牌訊號」——這是全章最棒的「具體數字落地」，我一看就懂、而且記得住。spill 那段「兩個數字非零就是記憶體不夠」也乾脆可操作。這兩條法則是本章最有價值的產出。
- **卡關 15（術語細節，輕微）**：spill 兩列的官方定義直接引用——「`Shuffle spill (memory)`＝shuffled data 在記憶體中反序列化形式的大小」「`Shuffle spill (disk)`＝資料在磁碟上序列化形式的大小」。對我來說「反序列化形式 vs 序列化形式」是兩個方向相反的詞、又沒在本章解釋（第 01 章 §1.6 只說過「序列化＝把資料轉成可傳輸的位元組」，沒講反序列化）。我會問「為什麼記憶體那個是『反序列化』、磁碟那個是『序列化』？這差別重要嗎？」——其實對「會不會 spill」的判讀，我只需要看「非零＝壞」，這兩個官方定義的細節反而讓我分心。建議要嘛用一句話安撫「你只要記得：這兩個只要非零，就是溢寫到磁碟了，越大越慢；memory/disk 兩種計法的差別不影響你的判讀」，要嘛把官方定義降級到來源框。
- **概念圖（行 150–166 Mermaid）**：正常 stage（task 都 3–4 秒）vs skew stage（一根 5min 🔥、其他 3 秒）的對比非常直觀，🔥 emoji 讓「那根肥 task」一眼可見。圖下還有一句把圖對回 Summary Metrics 表（「右邊那根…在表上就是 Duration 的 Max 遠大於 Median」），銜接得好。這張圖自明、達標。
- **我會這樣想／會問**：這節我學得最扎實。唯一會卡在「25th/75th percentile 到底指哪個值」和「序列化/反序列化那兩個定義要不要管」。其餘我覺得我真的會看那張表了。

## §2.7 把四種症狀對到 UI 上的數字（行 174–196）

把第 01 章四個敵人（shuffle 過大／skew／spill／掃太多）各自對到「看哪裡＋什麼訊號＋翻哪章」，結構清楚，是很好的收束。每種症狀都給了 UI/EXPLAIN 上的具體訊號，呼應前面各節。

- **卡關 16（範疇／一致性，中度）**：本章標題與前提框（行 1–5）說本章主題是「Spark UI 與 EXPLAIN 找瓶頸」，章末 §2.9 收束也說「認出是 shuffle 過大、skew、spill 還是掃太多」——**四種症狀**。這很一致、很好。但要注意：§2.7 第 4 點和 §2.8 表格都把「**小檔**」當成「掃太多」的一部分或獨立一列；而「四種症狀」其實是「shuffle／skew／spill／掃太多(含小檔)」。我數的時候會短暫困惑「到底是四種還是五種（小檔算不算獨立一種）」。§2.7 把小檔併進第 4 點「掃太多/小檔」，§2.8 表格卻把「小檔」拆成獨立一列（行 210）。建議統一：要嘛全程「四種、小檔歸在掃太多底下」，要嘛明說小檔是第四種的一個子情況。屬主旨一致性的小瑕疵。
- **卡關 17（缺脈絡，輕微）**：第 1 點 shuffle 過大的解法寫「避免不必要的 `DISTINCT`/`ORDER BY`」——第 01 章 §1.5 確實把 DISTINCT/ORDER BY 列為寬依賴(shuffle)，所以這裡呼應得上，OK。但「先聚合再 join」這招我沒在前文看過、這裡只是塞在括號裡帶過，我會好奇「為什麼先聚合再 join 會比較省」——不過它明說「→第 03 章」，留到後面講合理，可接受。

# 全章彙整

## 主旨一致性

**一致、達標。** 開場（§2.1「先量再調」）、四種症狀（§2.7）、章末（§2.9）首尾扣合，沒有過度宣稱、沒有與內文打架的結論。「跳過量直接調，多半白忙」誠實。唯一的一致性小瑕疵是「四種症狀（§2.7）」與「§2.8 七列速查表」「§2.7 第4點把小檔併入掃太多、§2.8 又把小檔獨立成列」之間的**數目落差**，會讓讀者短暫困惑到底幾種（見卡關 16、19）。

## 範疇平衡

- **太薄／詳略不對稱**：§2.2 的 **live UI 路徑**相對 History Server 講得簡略（History Server 有 1→2→3 編號、live UI 只有一句），但 live UI 是「查詢還在跑想現場看」時最急用的（卡關 6）。
- **缺一個貫穿全章的東西：真實畫面/範例輸出**。§2.3（EXPLAIN 輸出）、§2.5（SQL 頁籤 DAG）兩處都只講「該找什麼」、不給任何畫面示意，對「這輩子沒開過 Spark UI / 沒看過 EXPLAIN 輸出」的目標讀者，是把「看得懂道理」和「真的會操作」之間那一步留給了讀者自己跨（卡關 8、11）。這是全章對「可操作性」影響最大的範疇缺口。
- **最扎實的一節**：§2.6（Stages 頁籤），具體數字（Median 3s / Max 5min）、可操作法則（Max≫Median＝skew、spill 非零＝記憶體不夠）、自明的對比圖，是全章典範。其他節（尤其 §2.3、§2.5）若補上類似的「具體畫面＋具體數字」會更齊。

# 三級彙整

## 真缺陷（建議補）

1. **§2.2 ResourceManager / ApplicationMaster 沒給人話**（卡關 3）：審查清單點名的兩個詞，第一次出現且本章/第01章都沒解釋，是「沒碰過叢集」讀者最會卡死的點。需各補半句人話＋兩者關係。
2. **§2.5 HashAggregate 沒給人話**（卡關 10）：審查清單點名，第一次出現零解釋，讀者只能猜「是不是 GROUP BY」。補半句「做 GROUP BY 聚合的算子」即可。
3. **§2.6 percentile（25th/75th）白話定義不足**（卡關 13）：審查清單點名。「前 25%／前 75%」說法有歧義（由小到大還由大到小？），讀者讀不準那張 Summary Metrics 表＝讀不懂本章操作性最強的一節。需用「由小到大排隊、排在 1/4 位置那個 task 的值」釘死。
4. **§2.3 / §2.5 缺任何範例輸出或畫面示意**（卡關 8、11）：對「沒看過 EXPLAIN 輸出 / 沒開過 Spark UI」的目標讀者，這是「看得懂道理但不會做」的最大缺口。至少各給一段標明「示意」的縮略片段（EXPLAIN 三五行、SQL 頁籤 DAG 草圖）。
