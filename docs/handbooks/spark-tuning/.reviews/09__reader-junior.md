**JDBC 程式碼區塊的可讀性**


- 我看得懂這是「把資料寫到資料庫」的概念

**「照這章做得了嗎？」的誠實評估**

讀完這個範例，我知道流程是 `算 → 驗 → 發佈 → 遮罩 → 推 → 稽核`。但「做得了」要看我有沒有：
1. 行銷系統 DB 的連線資訊（url/user/password）——我去哪拿？

3. 對方表已建好、且有 `(cust_id, campaign_date)` unique constraint
4. Ranger 授權（我的帳號能推）

手冊給了我問對問題的能力，但這四件事都要靠人協調，不是我自己能直接動手的。這不一定是章節的問題，但我讀完後知道「我還不能按下去」，原因是協調工作還沒做。

## 三級彙整

### 一級：嚴重（術語零解釋，第一次出現就卡死）

1. **JDBC**（§9.2 程式碼前就出現在表格）：我知道它是「連資料庫的東西」，但不知道它是協定、工具、還是 Spark 的模組。全章最核心的技術通道，卻沒有一句「JDBC（Java Database Connectivity，一套讓程式連關聯式資料庫的標準協定；在 Spark 裡以 `.format("jdbc")` 呼叫）」
2. **OLTP**（§9.2 表格）：「打垮對方 OLTP」——我猜等於「行銷系統的資料庫」，但不確定。若猜錯，我的 numPartitions 決策基礎就會出錯
3. **egress**（§9.2 表格 NiFi 列）：完全沒解釋。讀者很容易跳過這個字，但它代表「資料流出方向」，是 NiFi 用法的關鍵

### 二級：中度（需要前章記憶，本章沒補）

4. **WAP / active-version**（§9.4）：只用括號說「見 §8.6」，對第 08 章記憶模糊的讀者是障礙；建議在括號內補一句話定義
5. **executor**（§9.2 API 那段）：前章有定義，但本章突然出現在 push 語境裡，需要一句「（executor：Spark 工作節點上真正執行計算的程序）」
6. **schema 演進**（§9.6 末段）：同上，類比依賴前章知識，本章不補就是懸空類比
7. **RDBMS**（§9.2 Sqoop 說明）：等於「關聯式資料庫」，用縮寫比中文更陌生

### 三級：輕微（有說但不夠、或可操作性留白）

8. **landing zone / 交換區**：「雙方約定的目錄」說明夠用，但沒有說這個目錄通常在哪（SFTP？HDFS 某路徑？），還是要問 IT？
9. **密碼／金鑰管理**：`.option("user", ...).option("password", ...)` 說「走金鑰管理，別寫死」，但對我來說「金鑰管理是什麼、去哪拿」是個黑盒
10. **PII 遮罩的實作**：§9.5 說「遮罩在 Hadoop 內做」，但沒給任何程式示意，對 Spark 新手留白太大
11. **「推失敗怎麼辦」的 DAG 銜接**：§9.4 流程圖只畫成功路徑，沒有「推到一半失敗、手動補跑」的示意，讀者容易誤以為失敗時要重跑整條 DAG

## 「照這章做得了嗎？」的最終答案

**能做決策，不能獨立動手。**

這章讓我能回答「我要走 push、挑 JDBC sink 通道、numPartitions 要保守、要讓對方端有 upsert、只推驗過的版本、要顧 PII」。但實際按下去之前，我還缺：
- 行銷系統 DB 的連線資訊（URL / 帳密）——要找 IT 或行銷系統維運
- 對方 DBA 確認 numPartitions 上限
- 確認對方表有 unique constraint（§9.3 upsert 才能運作）
- Ranger 授權推送的帳號權限

這些不是章節能替我解決的，但章節若能在 §9.7 範例最後加一個「行動項 checklist：動手前先確認這四件事」，會大幅提升「讀完可以去做」的感受。
