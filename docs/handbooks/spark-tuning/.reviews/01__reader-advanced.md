# 進階讀者審稿：01 · Spark 怎麼跑你的 SQL

**讀者人設**：進階 analytics engineer，負責維運共用特徵庫（feature store）供模型訓練、做 reverse ETL 把資料送回業務端「客戶行銷及管理系統」。關心長期排程資源配置、executor 形狀取捨、多租戶場景下的資源共用。

**審稿輪次**：聚焦確認輪（前輪已判範本級，本輪不大改、只標深度與缺口）。

---

## 逐節深度評

### §1.1 你的查詢，其實是一群機器一起做（含 Metastore 順帶認識）

**判定：夠營運**

Driver/Executor 的角色分工說得清楚，Mermaid 圖簡潔有效。「順帶認識」的 Hive vs Hive 歧義澄清對我這種同時操三個引擎（Spark、Hive、Impala）的人來說是剛好的踩坑預防。特別欣賞「你平常在 Hue 選的那個 Hive，就是第 2 個（引擎）；它查到的表，登記在第 1 個（Metastore）裡」這個句子——這正是跨引擎運維最容易混淆的點，趁早寫清楚省去後面很多解釋成本。

缺口：無（本章定位是建直覺，這個深度適當）。

---

### §1.2 你的資料被切成幾塊？partition 從哪來

**判定：夠營運**

128 MB 的切塊邏輯和「partition 數 ≈ 最多同時幾件事」的直覺說得準。「太少 → spill、太多 → 開銷 + 小檔」兩頭拉力對我日後診斷管線問題是夠用的指引。

缺口：對長期排程那條線有一個實務問題本節沒有觸及——**feature store 的寬表通常是 append-only 逐月累積，讀進來的 partition 數會隨時間膨脹**，但這算操作細節，forward 到第 04/05 章合理，本節不需要展開。

---

### §1.3 Spark 不會馬上算：先攢計畫，再一次跑

**判定：夠營運（偏精簡、但刻意如此）**

Lazy evaluation 的說明很到位，「能看到整份計畫才動手優化」是後面 Catalyst 的前置直覺，邏輯連貫。對我來說「整份計畫」的概念也直接解釋了「為什麼我在 Python 裡串十個 transformation 不會一直 Spark 任務飛出去」——這是日常維運很常遇到的疑惑。

缺口：本節對我實際會遇到的一個情境偏淡——**在長跑批次 pipeline 中，cache/persist 的時機選擇**。這和 lazy evaluation 直接相關（你得知道它何時才真正算，才知道 cache 在哪裡插有意義）。但這是操作細節，forward 到後章完全合理，此處建直覺即可。

---

### §1.4 從 SQL 到一群 task：工作分成四層

**判定：夠營運，Spark UI 對齊是加分項**

四層層級關係（Application / Job / Stage / Task）清楚，且直接說「第 02 章你會在 UI 看到這四層」——這個 forward-ref 非常有用，把心智模型的建立和診斷工具的使用直接連起來了。

我在看 Spark UI 時最常需要的判斷點是「這個 stage 為什麼這麼慢」，而「一 action 一 job / 一 shuffle 切一個 stage / 一 partition 一個 task」這三條規則已足夠讓我在 UI 裡定位問題所在的層級。

缺口：「一個 Application 從頭到尾共用同一批 executor」這句話對**多租戶排程情境**有一個實務後果值得一提——多個長跑 pipeline 共用叢集時，同一個 application 佔著 executor 不放（尤其是不開 dynamic allocation 時）會饑餓同事的 job。這是「資源配置長期排程」那條線的關鍵卡點，但我接受把它 forward 到第 04 章，本節只要讓我知道「application 層級 = executor 佔用週期」就夠了。建議在本節結尾加一句 forward-ref 像「一個 application 佔著 executor 的時間長短，以及怎麼讓它不餓死同事，第 04 章的 dynamic allocation 專門處理這件事」，比現在只有 §1.7 才出現 dynamic allocation 這個概念早了一點，讓讀者在正確的心智位置先把問題和解法放在一起。

---

### §1.5 兩種運算：自己算的（便宜）vs 要交換的（貴）

**判定：夠營運**

窄/寬依賴的直覺用「自己算自己的 vs 要跟別塊交換」來分，比用 dependency graph 術語解釋直觀得多。Mermaid 圖把「同一 cust_id 散在各 partition」畫出來了——這正是我在 feature store 最常看到的真實場景（按日期 partition 的寬表，一個客戶的特徵橫跨多個 partition）。

缺口：無。

---

### §1.6 為什麼 shuffle 是頭號敵人

**判定：夠營運，三個麻煩的分層是本章最好的一節**

這節是本章最有份量的一節，三個麻煩（spill 雪上加霜、stage barrier / skew、shuffle.partitions 200 的重設）講得透，而且「spill 是在 shuffle write 本來就一定寫磁碟之外的另一次寫」這個澄清非常重要——這個常見誤解我自己之前也有，澄清後理解完全不同。

對長期排程那條線最直接有用的是「資料傾斜 + stage barrier」的組合——這解釋了「為什麼我的 feature store 寬表 join 客戶維度表偶爾會卡一整個 stage 不動，UI 顯示 199 個 task 完成、1 個卡著」，而且不需要翻到第 03 章就能理解問題的根本結構。

AQE 的說明（主要幫「塊太小太多」，不幫「塊太大」）方向說清楚了。但我有一個營運角度的疑問：**AQE 在長跑 pipeline 裡不一定是「只幫一次」的，它是每個 stage 都能介入嗎？** 這個細節若能一句話點出（「AQE 在每個 shuffle 後都可以重新評估」），讀者在理解多 stage pipeline 的行為時會更準確。但這也可以是第 04 章細談的內容，本節點到「AQE 幫的方向」已夠。

---

### §1.7 executor 的 core/instance/memory 取捨

**判定：對「建直覺」夠用；對「長期排程資源配置」這條線偏淺，缺一個關鍵 forward-ref**

這節是本輪聚焦點，詳評如下。

**做到的部分：**

- 「同時能跑的 task 數 ＝ executor 台數 × 每台 core 數」這條公式很好，是後面設定計算的基礎。
- 胖/瘦 executor 的對比表把兩邊的代價列出來了：太胖的 GC 停頓、HDFS 吞吐卡住、fault tolerance；太瘦的 overhead 百分比高、broadcast table 記憶體浪費。
- 「4～5 core 起手值」有 Cloudera 官方來源支撐，不是拍腦袋的建議。

**偏淺的地方（對我的場景「長期排程資源配置」）：**

1. **Wave 的概念只提了一次，沒有給它更多份量。** 「200 個 task、50 個 slot、得跑 4 個 wave」這個計算對我理解「為什麼加台數不是線性加速」非常重要——wave 數不整除時，最後一個 wave 可能只有幾個 task 卻要等跟前面一樣長的 overhead。本節只在公式後面的例子提了一次，沒有進一步點出「wave 數是估計壁鐘時間的關鍵路徑指標」。這對「長期排程你要配多少 executor 才能讓 SLA 收得進時間窗」是最直接的計算工具。

2. **Dynamic allocation 只在結尾 forward-ref 了一句，但沒有說「為什麼它和 executor 形狀的取捨是相關的」。** 我在長期排程裡最頭痛的問題不是「一個 job 要幾台」，而是「多個 pipeline 排程錯開時，executor 怎麼在 job 之間流動而不是被第一個 job 長時間佔住」。Static allocation 下 executor 形狀直接決定你能配幾個 job 同時跑；dynamic allocation 下形狀的取捨邏輯不同（粒度越細，YARN 越容易找到空缺插入）。這層關係在本節完全沒有碰到——我接受「怎麼設下去」放第 04 章，但**「為什麼 executor 形狀和多租戶資源共用有關」這層直覺**放在本節會讓讀者帶著正確問題去讀第 04 章。

3. **Memory overhead 的 ~10% 只在來源欄裡提到，正文沒有整合進去。** 胖/瘦對比表說「overhead 約佔一成，台數越多被吃越多」，但沒有說這個 overhead 不是 heap（JVM 外的記憶體，用於 JVM 本身的原生程式碼、Python 行程等），所以配資源時「80 GB / 20 GB 的胖瘦範例是刻意省略這個 10%」這件事只在精確度聲明欄裡提，讀者如果直接拿這些數字去 YARN 設，會發現設不進去。這是一個「方向正確但設一個真實數字會被踩的坑」——建議正文加一句「實際每台還要扣 ~10% 的 JVM overhead（`memoryOverheadFactor`），所以 YARN 上能申請到的 heap 比總配置稍少」，讓讀者知道這個坑存在，操作細節 forward 到第 04 章。

**分層接受度：**

「建直覺 / 操作細節第 04 章」的分層我完全接受——本節的目的是讓讀者帶著問題去讀第 04 章，這個目標達到了。主要缺口是「長期排程那條線缺一個能帶著走的問題框架」：目前讀完我知道「4～5 core 是起手值」，但我不太知道「在多租戶排程場景裡，我該帶著什麼問題去找第 04 章的答案」。

**建議補一句話（不改整節）**：在結尾的 forward-ref 裡加一個句子點出排程維度，例如：「長期排程場景下，executor 形狀還牽涉到多個 pipeline 同時在叢集上搶資源時怎麼不互相饑餓——這是 dynamic allocation 和 queue 設定的領域，第 04 章一起談。」

---

### §1.8 把它全部串起來：一條 SQL 的旅程

**判定：夠營運，是本章最好的整合範例**

用「join 大表 + 兩次 shuffle + 四個 stage」這個範例把前面所有概念串起來，DAG 圖把 shuffle 的兩半（write/read）跨 stage 的關係畫出來了。「同一條 SQL，換個寫法成本差很多」用 broadcast join 做例子，具體且有可操作的方向。

對我維運 feature store 的情境（多張特徵寬表 + 維度表 + reverse ETL 回業務端），這個範例的「兩個 shuffle 兩個 stage」結構正好對應到我日常的 join 模式。

缺口：無。

---

### §1.9 為什麼 Spark 通常比老 Hive（MapReduce）快

**判定：夠營運，分寸拿捏好**

「Spark 不是永遠快；現在的 Hive 多半跑在 Tez 上」這個誠實說法對長期維運來說比「Spark 比 Hive 快」的簡單宣稱有用得多——因為我實際碰到「為什麼這個 job 改用 Spark 還是一樣慢？」的情況，而這句話幫我預先設定了正確的心理預期。

缺口：無。

---

### §1.10 一句話帶走

**判定：夠用**

「少搬、少讀、用對機器」三點覆蓋了全章主軸，forward-ref 到各章也清楚。

## 缺口彙整

### 重要（影響「帶著正確問題去讀後面章節」）

1. **§1.7 缺多租戶/排程維度的 forward-ref**：目前結尾只說「dynamic allocation 隨需伸縮，第 04 章詳談」，但沒有讓讀者意識到「executor 形狀取捨在多個 pipeline 共用叢集時有不同的考量邏輯」。建議結尾加一句話把這個排程維度的問題框架點出來。

2. **§1.7 缺 wave 概念的份量**：wave 數是估計 SLA 的關鍵指標，對長期排程場景是「配幾台 executor 才夠」的主要計算邏輯，但本節只在範例中一帶而過。

3. **§1.4 缺 Application 佔用 executor 時間與多租戶的連結**：「一個 application 從頭到尾共用同一批 executor」這句話是多租戶饑餓問題的根源，可以在此加一句 forward-ref 讓讀者帶著問題去讀第 04 章。

### 次要（精確度/踩坑預防）

4. **§1.7 memory overhead 只在精確度聲明欄出現，正文沒整合**：10% overhead 是一個實務設定時會踩的坑（YARN 上能配到的 heap 比設定值小一截），建議在正文加一句讓讀者知道這個坑存在，操作細節 forward 到第 04 章。

### 接受（本章不需要展開）

5. §1.2 的 feature store 寬表 partition 膨脹問題：forward 到第 04/05 章合理。
6. §1.3 的 cache/persist 時機：forward 到後章合理。
7. §1.6 的 AQE 是否每個 stage 都介入：第 04 章的細節，本節方向說清楚已夠。

## 三級彙整

### 本章優勢（不動）

- 從「你已知的 SQL」切入，不需要分散式系統背景就能建立心智模型，對 SQL-first 同事是正確的入口。
- Hive/Hive Metastore 的歧義澄清是長期維運的重要基礎，放在 §1.1 順帶做是正確時機。
- §1.6 的三個麻煩（spill/stage barrier/200 重設）分層清楚，是本章最有份量的一節。
- §1.8 的整合範例完整地把前面所有概念串成一個可操作的心智圖。

### 本章缺口（建議補）

- **§1.7 是全章深度最薄的節**（對照其對排程維運的重要性）：wave 概念份量不足、多租戶排程維度缺 forward-ref、memory overhead 的坑只在精確度欄出現。建議正文各補一句話，不需要大改。
- **§1.4 結尾可加一句 forward-ref**，把「application 佔 executor 時間 → 多租戶饑餓」的問題框架植入讀者心裡，讓他們帶著正確問題去讀第 04 章。

### 分層接受度

「建直覺 / 操作細節 forward 到後章」的設計完全接受。本章達到了「讓 SQL-first 讀者在不進 Spark 引擎細節的前提下，有能力理解後面每一章在幹什麼」的目標。上面點出的缺口都是「帶著正確問題」的層面，不是「操作細節講不夠深」的問題——這個區分很重要。
