### 3.5 join 策略——broadcast vs sort-merge

**卡關點（一個術語，一個假設）**

- **「Sort-Merge Join」**：「兩張表都按 join key 做一次 shuffle（把相同 key 的列搬到同一台機器），各自排序後合併」→ 我能理解，沒問題。

- **「Broadcast Hash Join」**：「把這張小表整份複製（broadcast）到每一台 executor」→ 我懂 broadcast 的意思，但**「hash」在哪裡？** `BroadcastHashJoin` 的「Hash」做什麼？我在 Hive 查文件時看過「hash join」，但這裡沒解釋為什麼是 HashJoin 而不是 NestedLoopJoin（這個後面的補充表有解釋，但正文主體沒有）。**類型：可加強**（一句話說「Hash 是把小表在記憶體裡建成 hash 表，大表的每一列拿去查，O(1) 對比；不需要排序」）

- **「AQE（Adaptive Query Execution，…第 01 章 §1.6 提過、第 04 章詳談）」**→ 括號解釋有了，而且 §1.6 是前置章節應讀過，我能理解。

- **預設 10MB**：「當它估計到一邊小於門檻 `spark.sql.autoBroadcastJoinThreshold`（預設 10MB）時，就自動走 broadcast」→ 我知道有個門檻了，但我日常的 dim_branch（幾百列）顯然遠小於 10MB，為什麼還是沒自動 broadcast？原因在 §3.6 說（沒有統計），這裡只說「多數時候自動幫你選」，但初學者可能會問「為什麼我的沒有自動」，這個問題的答案在下一節，這樣的引導是合理的。**沒有真缺陷**。

**補充表**

- **「`ShuffledHashJoin`：較少見；Spark 多半偏好 sort-merge，通常要靠 `SHUFFLE_HASH` hint 才會走它」** → 我不會主動觸發它，那我遇到的機率？不知道，但「較少見」已經告訴我可以先不管，這樣說法就夠了。

### 3.8 聚合的成本

**卡關點（一個術語，一個操作細節）**

- **「寬依賴」**：「第 01 章說 `GROUP BY`、`DISTINCT` 都是寬依賴，每一次都是一次 shuffle」→ 「寬依賴」這個詞在這裡才出現（前面各節沒用過），我讀過第 01 章但可能記不住這個術語。一句話提醒什麼是寬依賴會更穩（或者確認第 01 章確實有解釋且讀者應記得）。**類型：可加強**

- **「partial aggregation」**：括號說明「先在自己手上的資料算個小計、再把小計拿去 shuffle」→ 解釋清楚，但術語用了括號說明後又說「術語叫 partial aggregation」，順序稍微奇怪（先術語、後解釋更自然，或統一用中文就好）。這是小排版問題。**類型：可加強（微）**

- **「高基數（欄位）」** → `cust_id`，1000 萬個不同值：「高基數」這個詞沒有解釋。我從資料庫背景知道「cardinality」，但「高基數」作為中文術語是第一次看到。**類型：真缺陷**（一個括號說明「基數＝這個欄位有幾個不同值；高基數＝不同值非常多、如 1000 萬個客戶」就夠了）

- **`approx_count_distinct` + HyperLogLog++**：「HyperLogLog++」這個演算法名字突然出現，沒解釋。但下一句說「指紋摘要」的比喻很好，幫我理解了它做什麼，不需要懂演算法細節。接受。
- 「預設最大相對誤差約 5%」→ 我知道接受多少誤差了，可以操作。

**操作層面**：「算「本月活躍客戶數大約多少」「某活動觸及人數量級」這種看趨勢 / 量級的指標，5% 誤差通常無所謂、該用；但對帳、稽核、要跟財報對得起來的精確數字，就得用精確的 `COUNT(DISTINCT)`」→ 這個取捨說得非常精準，切中我日常的業務情境。

## 整體評估

強章，前輪判斷維持。「少讀 / 少搬」的架構讓我記得住，每招的 before/after SQL 讓我可以直接套用，§3.11 的貫穿範例是本章精華。

三個真缺陷（A1–A3）都集中在「我想操作但找不到起點」：看不到 EXPLAIN 的實際格式、不懂「高基數」這個詞、沒有 salting SQL 骨架。這三個補完後，本章對初階讀者的可操作性就完整了。
