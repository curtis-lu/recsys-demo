### 「進一步：REPARTITION 還是 COALESCE？」小節（L179–206）★本輪重點：是否過載★

逐塊記我讀的時候在想什麼：

1. **對照表（L183–186）**：兩列（COALESCE / REPARTITION）× 三欄（做什麼 / 要 shuffle 嗎 / 什麼時候用）。**這個表本身好**，我看得懂。
2. **「平行度傳染」段（L188–193）**：`COALESCE` 不切 stage → n 往上游傳染成「整段平行度上限」；`COALESCE(1)` ＝一個 task 扛全部。`REPARTITION` 切 stage 邊界 → 上游照原平行度。**這段是真的有料、也解釋得清楚**，「砍得少剩下還夠平行→COALESCE；砍得兇或要保上游平行→REPARTITION」一句總結很好。但它**已經是第二個知識點**了（表是第一個）。
3. **`REPARTITION(n)` vs `REPARTITION(n, key)`（L195–200）**：又分四個子項——`(n)` 平均打散 / `(n, col)` 同 key 進同塊（附 skew 警告）/ `REPARTITION_BY_RANGE` 按範圍 / `REBALANCE`（需 AQE，自動）。**這是第三個知識點，而且裡面又冒出兩個新 hint（RANGE、REBALANCE）和又一次 AQE。**
4. **分區表特例（L202）**：動態分區 + 裸 REPARTITION → `分區數 × n` 小檔爆炸；正解 `REPARTITION(dt)`；skew 用 `REBALANCE(dt)`。**這是第四個知識點，而且是全小節最燒腦的一個**（「每個 task 為碰到的每個分區值各寫一個檔」這個機制我要想兩遍）。
5. **CTE 放哪（L204）**：hint 放最外層 SELECT。**第五個知識點。**
6. 取捨講白（L206）：回到 128MB–1GB。

**§5.5「進一步」小節總評（直接回答本輪提問「是不是太長太滿、一次太多東西」）**：

> **是的，這一小節過載了。** 它在一個「進一步」小標題底下連續塞了 **5 個獨立知識點**：① 對照表 ② 平行度傳染 ③ (n) vs (n,col) + RANGE + REBALANCE ④ 分區表特例 ⑤ CTE 放哪。每一個單獨看都清楚，但**連讀下來像在喝消防栓的水**。出現了 **4 種 hint**（COALESCE / REPARTITION / REPARTITION_BY_RANGE / REBALANCE）＋兩種寫法（n vs n,col）＋AQE 兩次，對「沒學過分散式」的我，一次要分辨這麼多近似的東西，**讀完記不住誰對應誰**。

> 我讀到 ④ 分區表特例時，已經有點「這還沒完嗎」的疲勞感。而且 ④ 本身很硬（動態分區 × 裸 repartition 的小檔爆炸機制），它值得自己一個小節，塞在「進一步」尾巴會被淹沒、讀者反而錯過這個其實很重要的雷。

# 結尾彙整（真缺陷 / 可加強 / 誤讀 三級）

## A. 真缺陷（建議補）

1. **§5.4 補充小節（DataNode/locality）太硬、術語密度過高**（L115–128）。五個 locality 等級英文常數一次給、process/rack 沒當場解釋、四因素 ①②③④ 堆在「你其實不用調」的東西上。**結論（自動發生、你不用調、做好存法自然好）應提前；等級與四因素應大幅瘦身。** 這是全章對「沒學過分散式」讀者最難消化的一節。

2. **§5.5「進一步：REPARTITION 還是 COALESCE」小節過載**（L179–206，本輪核心提問）。一個小標題下塞 5 個獨立知識點、4 種 hint、AQE 兩次。**確認過載，應拆。** 至少把「分區表特例」（L202，重要又燒腦）抽成獨立小標題、把 RANGE/REBALANCE 降級成「用到再查」。

3. **§5.5 成因四的「~400 塊、約 5 倍」推導缺中間步驟**（L150）。77→400 的橋（1 萬檔 × 5MB = 50GB ÷ 128MB ≈ 400）沒寫出來，逼讀者自己補算。違反寫作指引「中間值要能加總回最終值」。**把這一步算式補上。** 同段「1MB 當 5MB 算」也要明寫「＝真實 1MB ＋ openCost 4MB」。

4. **§5.5 成因四一段塞太多**（窄依賴＋打包＋openCost＋400塊推導，L150）。建議拆兩段。

5. **§5.5 整節範疇失衡**（約 76 行，全章最重，約 §5.6 的兩倍多）。配合缺陷 2、4 瘦身後可緩解。
