## §3.7 少搬（三）：join 的兩個隱藏陷阱——key 型別不一致、爆量 join（行 203–232）

這節兩個陷阱都很實用、很「銀行日常」（bigint vs string 的 cust_id、漏寫 join 條件）。陷阱一四件套齊全；陷阱二原理＋怎麼發現＋解法齊全（沒有 before/after SQL 的「after」，但這類問題本來就是「別寫成那樣」，可接受）。

## 結尾彙整：真缺陷 / 可加強 / 誤讀

### A. 真缺陷（建議補）

1. **§3.11 貫穿範例 ② 的型別前提與 §3.7 對不上**（卡關 20）：§3.7 設定「t.cust_id bigint、c.cust_id string」（兩表型別天生不同、治本要從建表對齊）；但 §3.11 改寫後直接斷言「兩邊都 bigint」，暗示 c.cust_id 本來就是 bigint、原作者多寫 CAST。讀者無法判斷 c.cust_id 到底什麼型別、能否安全拿掉 CAST。**這是「貫穿範例能否只靠前面讀懂」任務的真實破綻**，需統一兩節的型別設定。
2. **HyperLogLog++ 只給名字、沒給一句「為何能近似」的直覺**（卡關 15）：讀者被要求接受「它就是快」卻不知為何，違反 §3「抽象主張要落地」。補一句白話（保留小概要、不存每個值）即可，不需數學。
3. **AQE 第一次出現（§3.5）無任何人話、全押後第 04 章**（卡關 7）：但 §3.5/§3.10/§3.11/§3.12 反覆依賴 AQE 給讀者「不用手動」的安心感。首次出現必須補一句人話（執行途中邊跑邊調計畫的機制）。
4. **「隱式型別轉換」「笛卡兒積」兩個審查清單點名詞，首次出現未翻白話**（卡關 10、11）。
5. **§3.7 一對多/爆量 join 缺可手算的數字例子**（卡關 12、範疇薄）：全章唯一「抽象主張沒數字落地」的招，補「重複 3 次→100 筆變 300 筆」這種小例子。

### B. 可加強（斟酌）

6. **「列式/columnar」沒點破譯名陷阱**（卡關 3）：「列式」字面像按列、實則按欄，SQL 讀者極易理解反，建議明寫「columnar 直譯叫列式、實際是按欄存」。
7. **driver 在前提框未列、本章首用未介紹**（卡關 9）：broadcast 原理一直講「driver 收集→executor 各一份」，driver 卻沒被引介。
8. **map-side 的「map」沒解釋**（卡關 13）：沒學過 MapReduce 的讀者對「map」無概念，建議去掉英文或補半句。
9. **§3.10 設定名/門檻過載**（卡關 17、範疇厚）：5 倍/256MB 兩個門檻稀釋「AQE 自動搞定」主訊息，建議後推第 04 章、正文只留「有內建門檻自動判定」。
10. **§3.10 salting 缺 SQL，正文未先打預防針**（卡關 19）：章末有交代但正文沒，讀者會以為漏寫；正文補一句「故意只給概念」。
11. **§3.11 第 4 條「SELECT * 沒犯」提了個沒發生的問題**（卡關 21）：改成正面確認，別用「沒犯」講法。
12. **「謂詞」「ANALYZE TABLE」「資料塊/row-group」「Hash（join）」可各補半句白話**（卡關 4、5、6、8），都屬輕度、不補也能讀。
13. **§3.2 點破「磁碟分區目錄」vs「第 01 章執行時 partition」是兩個層次、同名易混**（卡關 1 之延伸），可省讀者暗自打結。

### C. 誤讀 / 不改或微調

- **§3.9 Window 算子/Exchange 在 EXPLAIN 長怎樣**（卡關 16）：屬「EXPLAIN 輸出無範例片段」的跨章老問題（第 02 章讀者日誌已記），本章靠 §2.3 既有習慣可接受，不算本章欠。
- **「加鹽」由來沒講**（卡關 18）：不必解釋為何叫鹽，有「加隨機後綴」人話即可，微調或不改。
- **§3.4「資料塊壓縮」前提**（卡關 5 後半）：不影響理解，可不補。

**總評**：以「SQL 會、分散式不會」的讀者視角，本章是這套手冊裡**可操作性最高、結構最清楚**的一章——少讀/少搬骨架好記，多數招四件套齊全、取捨誠實、鷹架乾淨。最該優先處理的是 **A 類 5 點**：尤其 (1) §3.11 貫穿範例型別前提與 §3.7 矛盾（這直接影響「範例能否自足讀懂」這個交付重點）、(2) HyperLogLog++ 與 (3) AQE 兩個「全章依賴、卻只給名字不給直覺」的術語。B 類多為「補半句白話」的低成本改善，集中在 §3.7–§3.8 那批數學/分散式黑話詞。
