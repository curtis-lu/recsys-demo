# 第 08 章審稿——進階 analytics engineer 視角
> 人設：會 SQL 也會一點 Python，營運**共用特徵庫供模型訓練**、做 reverse ETL 回業務系統。關心：資料產品可信度對長期營運的深度。
> 日期：2026-06-23

---

## 逐節深度評

### §8.1 本章地圖

**深度評**：四象限地圖清晰，「跑成功 ≠ 算對」的定錨一句話建立正確動機。對我這條線而言，少了一條預告：「G2（training–serving 一致性）待補」在此完全無跡——讀者第一次讀時不知道這一章刻意省了這塊。可以在 flowchart 底下一行加個 `ⓘ 本章不含 serving 端一致性（§x.x 補）` 作為指路，避免讀完整章後才發現缺口。

---

### §8.2 資料品質驗證

**深度評**：`not_null`、`accepted_values`、`unique_combination_of_columns`、`severity` + `error_if` 的組合，以及 pure SQL union-all 品質閘的 shell 接法，覆蓋率好。文字明確告知「你不主動驗，就只能等下游出事」，心態建立到位。

**深度缺口：品質閘「擋住後怎麼辦」流程幾乎是空白。** 當前的描述是：「驗出問題 → 擋住下游 → 通知到人，三步缺一不可。」但第三步「通知到人」之後呢？實際營運共用特徵庫最讓人頭痛的不是「能不能偵測」，而是偵測到之後的行動路徑：

1. 是等修好再重跑，還是讓某些消費者拿前一個版本暫代？
2. N 個下游依賴不同 snapshot、一個 snapshot 品質閘失敗，是整條線停，還是只停那個 snapshot？
3. 誰有權決定「這個品質問題可以先放行（降級）」、誰的決定不算數？

這三個問題缺任何一個的答案，PM 或資料工程師就只能在深夜的告警通知裡各自猜答案。目前的章節停在「告警通到人」這步，沒有給讀者一個哪怕最精簡的決策樹或 SLA 承諾的框架。

**缺口二：「品質閘要驗哪個 snapshot」在多消費者場景沒有明說。** §8.2 的 SQL 範例釘死 `WHERE snapshot_date='2026-05-31'`——但共用特徵庫可能一次產出多個 snapshot（回補時尤甚）。「每個 snapshot 各跑一次品質閘、還是只驗最新的」這個問題在 §8.6 才隱約出現（audit 帳本），§8.2 這裡先挖坑沒有填。

---

### §8.3 時間點正確性與特徵洩漏

**深度評**：這節是全章最貼近「特徵庫運算師」日常的一節，三種洩漏長相的分類清晰，SQL 正反範例對比乾淨。as-of join 的對照說明讓讀者理解 snapshot 設計的價值，非常有用。

**已知 G2（training–serving 一致）是薄弱點**，本評審不重複指出，但針對「若要補，該補什麼、放哪」給建議（見末節）。

**深度缺口：「特徵計算邏輯的版本」沒有出現。** §8.3 描述的是「SQL 時間上界要設對」——但若一個 snapshot 的特徵被重算（§8.6 的回補場景），且計算邏輯本身改了（不只是 bug fix、而是特徵定義改了），用新邏輯回填舊 snapshot 會讓「新模型拿到的訓練資料」和「舊模型當初看到的」不同。目前只靠 `build_version` + `code_version` (git sha) 在 audit 帳本裡記錄——但訓練端如何知道「我應該把哪些 snapshot 視為同一套特徵定義」，章節裡沒有線索。這不是要求章節寫完 feature store，而是至少要點一下「改特徵定義要比 bug fix 更謹慎——它不只是一次 backfill，而是讓歷史 snapshot 產生了異質性」。

---

### §8.4 多消費者契約

**深度評**：「只加不改」原則清晰，「同名欄改語意最危險」這句點得極準——這是我在實際營運中最常見的無聲災難，章節說清楚了。「破壞性變更版本化＋過渡期」方向正確。

**深度缺口：契約破壞的「偵測」機制幾乎是零。** 整節的預設是：契約由人工遵守（「你不應該改語意」）。但在有多位工程師、多個 pipeline 的共用特徵庫中，「某人不小心改了欄位邏輯」這件事有可能發生而且**不會有任何訊號**——除非下游的 §8.2 品質閘恰好有值域/漂移測試能抓到它。

這個缺口具體是：**欄位語意改了，怎麼知道？** 目前沒有任何機制能偵測「`feat_a` 的計算邏輯在昨天和今天之間悄悄不一樣了」（除非剛好有漂移告警，但那是統計訊號、不是語意閘）。業界常見的輕量防線：

- 在 audit 帳本的 `code_version` 欄加上「這個 build 用了哪個 `feature_definition_version`（邏輯定義的版本，獨立於 code commit sha）」——告訴下游「如果你拿的 build_version 之間的 feature_definition_version 不同，當心」。
- 對「語意敏感欄」（如 `total_spend` 的時間窗從 30 天改成 90 天）在 dbt/YAML 的欄位描述上加機器可讀標記，讓測試框架在 definition 改動時強制觸發重算通知。

章節目前是「說了不能改、但改了也不知道」的狀態。對長期共用特徵庫，這個缺口不小。

**缺口二：「過渡期多長」完全留白。** 「確認沒人用了再刪舊欄」說得對，但如何確認？誰來確認？目前是口頭承諾。至少提一句「靠 audit 帳本裡的 `built_by` / 下游讀表的 SQL 掃描」之類的具體線索，讓讀者知道這不是全靠人工盤點。

### §8.5 資料版本與可重現性

**深度評**：三招由輕到重的節奏合理，雙層分區 + current view 的 SQL 範例完整、可直接抄用。「`MAX(build_version)` 取最新依賴版本字串可排序」的 footnote 提醒很實用——這是真實踩過的坑。audit 帳本 `INSERT INTO` vs `INSERT OVERWRITE` 的對比說明精確。

**深度缺口：「清理和哪個 build 可以安全刪除」缺少查詢範例。** 章節說「靠 audit 帳本清理前先查引用」，但沒有給讀者一個「怎麼查」的 SQL。實際操作時，我需要知道：「哪些 build_version 目前有任何下游模型還引用著？」如果 audit 帳本只記錄「產出」、沒有記錄「哪個 build_version 被哪個模型訓練用過」，那這個查詢根本跑不出來。這裡缺了「消費端也要在 audit 帳本裡記一筆」（或有個對稱的 `feature_consumption_log` 概念）的提示。

---

### §8.6 進階：bitemporal·WAP·回滾

**深度評**：bitemporal 兩條時間軸的概念說明精確，WAP 三步對應本章小節的表格極好——讓讀者看出本章是一個整體設計而非散章。回補走查情境清晰、靜態 vs 動態 PARTITION 的警告很實用。SCD2 的邊界說明誠實（「這是維度表的場景，不是我們的場景」）。

**深度缺口：版本回滾的「範圍」問題沒有被點出。** §8.6 說「回滾零成本：append 一列指回舊 build」——這對**單一 snapshot** 的回滾是對的。但如果你做了一次**跨多個 snapshot 的回補**（例如修了一個影響過去 6 個月的 bug，一口氣重算了 6 個 snapshot），然後發現這次回補本身有問題、要回滾，就不是 append 一列了——你需要對 6 個 snapshot 各 append 一列指回舊 build，而且這 6 筆 promote 必須是原子的（不能補了一半）。這個場景目前完全沒有提到，但在實際特徵庫運維中，批次回補後發現問題、需要批次回滾，是完全真實的場景。至少要在 footnote 點一下「批次回滾時要對所有涉及的 snapshot atomically revert」。

**缺口二：`promoted_at` 的 tie-break 問題在 footnote 提了，但正文的 SQL 沒有處理。** footnote 說「同一時刻多次 promote 需另加序號 tie-break」——但 current view 的 `ROW_NUMBER() OVER (... ORDER BY promoted_at DESC)` 若有 tie，行為是 non-deterministic。讀者要麼在 SQL 裡加 `promoted_at DESC, rowid ASC`（需有序號欄），要麼在正文裡說清「實作時建議加 auto-increment 序號以防 tie」，不能只丟在 footnote。

## G2 若要補：該補什麼、放哪

**補什麼（三個核心問題）：**

1. **特徵計算邏輯的同一性（identity）**：training 用 snapshot 分區計算特徵的邏輯，和 serving 端即時/批次計算特徵的邏輯，是否由同一份 SQL/程式碼生成？如果兩套邏輯各自維護，即使時間點對了，skew 還是會靜默發生。最輕量的防線：serving 端**也讀同一張 `cust_feature_current` view**，不另外算——也就是 §8.3 的 snapshot 設計天生消除了這個問題，但要明說「serving 直接讀快照表是關鍵前提」。

2. **哪個 snapshot 對應哪次推論**：serving 端每次打分時，用的是哪個 `snapshot_date` 的特徵？如果推論是週批次，用的是最近一個月底 snapshot，就要確保 serving 讀到的和訓練時用的最新 snapshot 是同一類型的時間點（月底 vs 週結）。

3. **缺值填充的一致性**：training 時如果對某些 null 特徵做了特定填充（如填 -1、填中位數），serving 端相同邏輯要對齊。這在純 snapshot 設計下最容易漏——填充邏輯通常在 ML pipeline 裡，不在特徵表裡。

**放哪：**

建議獨立為 `§8.3.1 training–serving 一致：snapshot 設計的天然優勢與殘存風險`，放在 §8.3 末段（現在 as-of join 旁註的後面）。理由：§8.3 已經在講時間點正確性，serving skew 是同一個問題的延伸——「訓練時對了，上線時也要對」。篇幅建議一個 `###` 子節，給 2–3 個具體校驗點（serving 讀哪張表、哪個 snapshot、缺值邏輯），不需要展開 feature store 的完整設計。

## 三級彙整

### 優（這一章做得好的）

- §8.2 品質閘的 dbt + pure SQL 雙路徑，涵蓋有框架和無框架兩種現實，實用。
- §8.4「同名欄改語意最危險」——這句點中了共用特徵庫的頭號無聲地雷。
- §8.6 WAP 三步對應表（Write/Audit/Publish 對應到本章小節）是全書到目前最好的「設計模式有業界名字」的示範。
- bitemporal 兩條時間軸、以及 SCD2 邊界說明（它是維度表的場景），誠實且清晰。

### 可加深（不影響通過，但對目標讀者有實質提升）

- §8.2：品質閘失敗後的行動路徑（等修好？讓消費者用前版？誰決定降級？）至少給一個精簡決策框架。
- §8.4：欄位語意破壞的偵測機制——光說「不要改」不夠，要點一個可操作的檢查手段。
- §8.5：「哪些 build 可安全清理」需要消費端也記錄使用紀錄，audit 帳本目前只記產出端，不記消費端，清理就沒有完整依據。
- §8.6：批次回滾（跨多個 snapshot）的原子性問題要至少點出。

### 缺口 / 擋住往下讀的問題（training–serving 之外）

- **§8.4 欄位語意改變無偵測機制**：對「長期共用給多個模型」的特徵庫，這不是口頭規範就夠的，需要可操作的偵測手段（哪怕是「特徵定義版本欄 + build 異質性告警」）。這個缺口若不填，讀者把這章當成整章可信度的保證書，但它其實對「語意靜默漂移」是盲的。
- **§8.6 批次回滾的原子性**：多 snapshot 的回滾不是 N 次獨立操作——中間失敗會讓一部分 snapshot 回滾、一部分沒回滾，進入「半新半舊」狀態。這是不能只靠 footnote 帶過的場景。
