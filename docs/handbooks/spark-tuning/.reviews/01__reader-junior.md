### §1.6 為什麼 shuffle 是頭號敵人 ← 聚焦點主節

這是本次審稿的核心聚焦點，細看。

**具體例子（3000 萬筆）**：用銀行帳務當例子非常對味，馬上代入。shuffle write → shuffle read 兩步走分得清楚。

**spill 的釐清段**（麻煩一）：「先釐清一個常見誤解：shuffle write 本來就一定會寫本機磁碟……spill 是在這之外的另一次磁碟寫」——這段對有點基礎的人很有幫助，但對完全不懂的初階讀者，這段邏輯要讀兩遍才懂。我的理解路徑是：

1. 我先記住「shuffle write 一定寫磁碟」
2. 再記住「spill 是另一次磁碟寫」
3. 但「為什麼 spill 的暫存資料會塞不下記憶體」——前面只在 §1.2 說過 spill 是「暫時放不下的資料先寫去磁碟」，但沒說是誰的記憶體、為什麼在 shuffle 時特別容易塞不下。

**卡關點（輕度）**：spill 的「誰的記憶體塞不下」這條線在 §1.6 仍然不太完整。我能接受最終解釋在別章，但「為什麼 shuffle 比窄依賴更容易 spill」這件事沒有明說——直覺是「因為要先把同 key 的資料聚到一起，量更大」，但章節沒有把這個直覺說出來。

**麻煩二（stage barrier + skew）**：解釋很好。「199 個 task 三秒做完，剩 1 個跑了五分鐘，其他人只能乾等它」——這個畫面非常生動，我能想像。

### §1.6 AQE 第一次出現的密度 ← 聚焦點一

**原文**（麻煩三段落，最後一句）：
> 好消息是 Spark 3.3 的 **AQE**（Adaptive Query Execution，查詢途中自動調整執行計畫的機制）**會在執行時自動把過多的小 partition 合併**成大小剛好的塊（第 04 章詳談），所以這個值你多半不必手動煩惱——但你要看得懂「為什麼是 200」。（注意 AQE 主要幫的是「塊太小太多」這頭把它們併大；「塊太大」那頭得靠別的辦法，後面會談。）

**我的閱讀感受**：

這整段是「麻煩三」的結論，前兩句在說「預設 200 個 partition 的問題」，然後突然跳出 AQE 當作「解法」，再補了一個括號限縮（AQE 只幫「塊太小」那頭）。

**具體卡關的地方**：

1. **「200→AQE 自動合併」這一跳確實需要回讀**。我第一次讀到 AQE 的時候，腦子裡的狀態是「我剛剛在理解為什麼 shuffle 後固定是 200 塊」，結果突然插入一個縮寫 AQE 加括號全名，又立刻說「第 04 章詳談」——這讓我有「好，我還沒準備好收下這個新術語，但它馬上就被推後章節了」的感覺。

2. **密度過高的具體表現**：在同一句話裡，我要同時消化：
   - AQE 縮寫 + 英文全名（Adaptive Query Execution）
   - AQE 的功能（途中自動調整執行計畫）
   - AQE 的具體行為（合併過小 partition）
   - AQE 只幫單邊（塊太小，不幫塊太大）
   - 這個 200 的值「多半不必手動煩惱」

   這五件事擠在兩句話加一個括號裡，密度確實偏高。

3. **讀到「但你要看得懂『為什麼是 200』」**，我的反應是「我剛剛看懂了 200 是從哪來的，但我還不確定我懂不懂 AQE 是什麼」——這句結論來得有點突然，因為我的注意力被 AQE 分走了。

**建議的鋪陳改法（讀者視角）**：

與其在「麻煩三」段落裡把 AQE 當作插敘收尾，不如用一個更自然的過渡：先把「200 的問題」說清楚（好消息有解法但稍後才講），然後單獨一句「這個問題 Spark 3.3 之後有自動修法（AQE），第 04 章再談」。不需要在這裡給 AQE 的全名加括號解釋——第 04 章自然會展開；在這裡提只需要讓讀者知道「這個問題你不孤單、有機制在幫你」就夠了。

也就是說：AQE 第一次出場，**目的是「給讀者信心」，不是「定義術語」**。但現在的寫法兩件事都試圖做，結果兩件事都做得不夠好——信心還沒建立，術語又講不完整（因為細節在第 04 章）。

## 彙整

### 真缺陷（必補）

1. **AQE 第一次亮相的脈絡不對**（§1.6 麻煩三最後）：AQE 在「理解 200 partition 問題」的中途突然以「解法」身份插入，帶著全名 + 括號定義 + 功能說明 + 限制說明，密度過高，且讀者還沒準備好收下這個新術語——因為 §1.6 的主題是「shuffle 有三個麻煩」，不是「認識 AQE」。建議把 AQE 的全名與功能說明移到第 04 章初次展開；§1.6 只需一句「這個問題 Spark 3.3 的 AQE 機制會自動緩解，第 04 章詳談」即可，讓讀者帶著「好，有人幫我」的信心離開這一節。

2. **§1.6 spill 的「誰的記憶體塞不下」從未明說**：§1.2 說 spill 是「暫時放不下的資料先寫去磁碟」，§1.6 釐清 spill 是「在 shuffle write 之外的另一次磁碟寫」，但**從未說清楚「為什麼 shuffle 時特別容易 spill」**（因為要先把同 key 的資料聚到一起、暫存在 executor 記憶體裡，量更大）。這條因果線缺一個句子就能補上，缺了它「spill 是雪上加霜」這個說法讀起來像斷言而非解釋。

### 可加強（斟酌）

3. **§1.7 節首缺一行「閱讀引導」**：不自行設定 executor 的純 ad-hoc 讀者（例如只用 Hue 的分析師）讀 §1.7 的四個「太胖的問題」時，不確定自己「需不需要記這些」，容易在這裡失去動力。建議在節首加一句「如果你的叢集由 IT 統一管理資源、你不需要自行設定 executor，這節可以快速瀏覽、等需要時回來查」。

4. **§1.4 SparkSession 未解釋**：只在 Hue 跑 SQL 的人，不一定知道自己在用 SparkSession。加一行「你在 Hue 每開一個 Spark 查詢視窗，背後就是一個 SparkSession」即可消除困惑。

5. **§1.7 dynamic allocation 未解釋**：出現在 blockquote 裡（「用 dynamic allocation 隨需求伸縮而不佔著資源餓死同事」），是全文中少數沒有任何括號說明的英文術語，初階讀者會停頓。加「（動態分配，讓 executor 台數隨工作量自動增減）」即可。

6. **§1.2 gzip 的措辭**：「像 gzip 這種不能從中間切開的檔」——對不熟悉壓縮格式的讀者，gzip 是陌生詞。可改為「像 gzip 這類壓縮格式（它的特性是不能從中間切開）」，讓括號說明主語更清楚。

### 誤讀（不改）

7. **§1.5 「寬依賴」的「寬」字**：我第一次看到「寬依賴」覺得「什麼叫『寬』？」——但馬上看到「要把散在各 partition、但同一個 key 的資料聚到一起」，這個例子比術語名稱更清楚，所以這個名字不直覺是正常的，不需要特別解釋字面意思，例子本身已充分。不改。

8. **§1.8 DAG 圖的「兩半」補注**：「每個 stage 結尾的 shuffle write，和它箭頭指向的下一個 stage 開頭的 shuffle read，是同一次 shuffle 的兩半」——我在圖上找了一下才對上，但最後對上了。這是圖的固有限制（平面 mermaid 難以把「兩半」概念直接畫出來），補注本身已足夠。不改。

9. **§1.9 MapReduce 未完整解釋**：對這章的心智模型不必要，§1.9 的功能是「為什麼 Spark 比 Hive 快」，MapReduce 只是對比對象，從功能面說明已夠。不改。
