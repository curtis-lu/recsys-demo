# 第 07 章 審稿：進階 Analytics Engineer 讀者視角

**人設**：會 SQL 也會一點 Python，負責維運共用特徵庫供訓練用、同時做 reverse ETL 回業務系統。關心：長期穩定排程的深度。

**審稿日期**：2026-06-23

---

## 一、逐節深度評與缺口

### §7.1 本章地圖（三層工具）

**深度**：三層工具（dbt／Airflow／cron）分工說清楚了，DAG 一詞兩義的警告也有。Airflow 3.x 版本差異的 caveat 及時放在開頭，這是務實的。

**缺口**：
- 對我來說，三層說明更像前置知識確認，不是給我的新內容。「reverse ETL 回業務系統」的排程定位完全沒提——我的作業有一個獨特需求：下游不是模型，而是**另一個系統的 API 或資料庫寫入**，這條線和「特徵表→訓練」的相依結構不一樣（特別是失敗語意：特徵算錯可重跑，reverse ETL 打出去的 API 呼叫可能不幂等）。不過我也理解本章定位是通用排程，這條線不展開可以接受。
- Mermaid 圖只畫了 Airflow→dbt→表，漏掉 cron→dbt 的路徑（cron 場景其實也會觸發 dbt run），這對「只有 cron 的讀者」來說圖和文字有一點不對稱。

### §7.2 冪等與可重跑

**深度**：這是本章最扎實的一節。`INSERT OVERWRITE` 指定分區、`partitionOverwriteMode static/dynamic`、`SELECT *` 欄位錯位——三個 footgun 都是我實際踩過的，而且說法直接不兜圈子。`BY NAME` 的版本限制（Spark 3.5+）也有點出，對 3.3.2 環境讀者很重要。

**缺口**：
- **缺 staging 表過渡的討論**。實務上共用特徵庫幾乎不會直接覆寫線上表，而是先寫 staging、驗完再做 `ALTER TABLE … EXCHANGE PARTITION` 或 rename。這個「先寫 staging→驗→換進去」的流程，可以讓下游在換入瞬間得到「全量就緒」的原子性，而不是「覆寫到一半被讀到」的半成品。本節完全沒有這條線，對長期多支特徵庫這個需求有缺口。（是否屬第 08 章範疇？我認為不是：這是寫入冪等性的延伸，不是資料對不對。）
- **缺 write-audit-read 三步驗證模式的提及**。覆寫完之後、讓 _SUCCESS 出現前，應該至少做一次 count 確認不是 0 列寫入。這是本節冪等邏輯的收尾，但完全沒說。

### §7.3 排程相依

**深度**：`ExternalTaskSensor` 的核心用法給了，sentinel 檔的手刻替代也有。timeout 設定有提到。

**缺口**：
- **扇出（fan-out）的情境說了，但沒說「扇出表的 schema 變動」這條更難的相依**。我管的共用特徵表如果改欄（哪怕只是加欄），下游 N 個模型的 DAG 要不要一起升版？這是跨 DAG 相依的第二個維度——不是「就緒」，而是「版本對齊」。本節完全不碰這個，我在最後一節的速查表才看到「改一張共用表一次打爛 N 個下游」被指到第 08 章，但這條問題在排程相依層面也需要一個閘門的說法。
- **sensor 的 `execution_date_fn` 跨週期對齊問題完全沒提**。我的上游 DAG 是月底跑、下游是每週跑，`ExternalTaskSensor` 的 `execution_date_fn` 需要手動算「本週對應哪個月底的上游」，這個是 Airflow sensor 最常踩的坑之一，空白。
- **ExternalTaskSensor 逾時後的處置**（人工介入 vs 自動跳過舊資料繼續跑）缺討論。timeout 只說「失敗」，但失敗後是整個下游 DAG 全停、還是用上次成功的版本繼續？這個營運決策完全沒提。

### §7.4 回填

**深度**：`catchup=True` 的 footgun、`backfill --max-active-runs`、`current_date()` 的時間戳問題——對我來說是高密度且直接有用的內容。dbt 的 `--vars` 外傳日期也說清楚了。

**缺口**：
- **缺「部分成功的回填怎麼繼續」**。回填 90 天，跑到第 47 天時叢集重啟，剩下 43 天要怎麼辦？Airflow `backfill` 的 `--reset-dagruns` 、dbt 場景下記已成功日期然後跳過——這個「回填的可中斷性」文字有提到原則（「可中斷續跑」），但完全沒給落地做法，cron 場景的進度記錄也只說「把成功的日期寫進一個檔」但沒有給範例。
- **缺「回填期間下游要不要暫停」的決策**。回填 3 個月歷史時，每天仍在跑的正常排程會不會讀到「還沒回填到的」分區然後產出舊資料下游？這個時序衝突在特徵庫場景非常真實。
- **`airflow dags backfill` 在 Airflow 3.x 改名為 `airflow backfill create`**——§7.1 已提到，但 §7.4 的程式碼範例沒有呼應這個警告，讀者可能照抄就撞版本問題。

### §7.5 監控與退化

**深度**：「隨資料長大退化」的說法有，Airflow SLA、retries 作為健康訊號的說法也有，`set -euo pipefail` 是我一直想找個地方跟同事說的，在這裡出現非常好。

**缺口**：
- **缺「量化退化的基準線」**。說要看 History Server 的曲線，但看什麼才算「退化到要處理」？本節只說「定期翻」，沒有給任何基準：是「比上週慢 20% 就告警」、「單次 shuffle spill 超過 X GB 就回去看 partition 設計」？對我來說這裡需要一個可操作的門檻，不是光說要看。
- **缺資料量異常（不只時間）的監控**。作業跑得比上週快，但產出的列數少了 30%——這不是時間退化，是資料品質問題假裝成正常完成。本節把「監控」完全框在「跑多久」，但對特徵庫這種「下游信賴數字對不對」的場景，**列數監控比時間監控更重要**。這個徹底缺席。（我知道第 08 章有資料品質，但列數基準線是排程監控的一部分，不是資料品質，在這裡應該有一句話。）
- **dbt `run_results.json` 的監控說法**只說「收集起來看趨勢」，沒給任何具體做法（存哪、用什麼看、多久看一次）。這個對沒有 MLOps 平台的讀者等於沒說。

### §7.6 檔案與統計維護

**深度**：external vs ACID 表的 compaction 差異說清楚了（這個經常被混淆），`ANALYZE` 和清過期分區都有落地 SQL。`DROP PARTITION` 對 external 表不清 HDFS 的警告也有。

**缺口**：
- **缺「維護排程的頻率決策」**。說「排進例行排程」，但多久做一次？compaction 要每次寫完就做、還是每週一次？`ANALYZE` 是每個分區都要做還是全表一次？對我來說最不確定的就是頻率，這裡完全沒有任何建議。
- **缺「compaction 期間的可用性」**。重寫式 compaction 是先讀再覆寫，期間如果下游正好在查那個分區，會讀到什麼？這個窗口問題對共用特徵庫（多個下游同時查）是真實的。
- **缺「分區保留政策和回填的互動」**。如果我清掉了 2023 年的分區，之後要補算那段的新特徵就做不到了。保留政策決定了回填的天花板，這個跨節的連結只靠「見第 08 章 §8.5」一句帶過，但那個決策其實在這裡就要做。

### §7.7 速查表

**深度**：格式好、覆蓋率高，把本章的踩雷都收進來了。兩列指向第 08 章的條目也有標注。

**缺口**：無重大缺口，是本章最實用的查找工具。唯一建議：「重試把問題放大」那列，建議加一個「非冪等 + retries = N 倍重複寫入」的更具體說法，目前讀起來略抽象。

### §7.8 取捨總表 / §7.9 一句話

**深度**：取捨總表乾淨，一句話收尾簡潔。

**缺口**：取捨總表缺一欄「監控頻率/維護頻率」的取捨，這是本章最沒給具體建議的地方。

## 二、「缺少端到端走查節（capstone）」的評估

### 我的立場：**應該補，且補了會讓本章從「章節集合」變成「完整劇本」**

本章目前的結構是：七個原則節 + 速查表 + 取捨表 + 一句話。每節都有通用原則和落地，但**沒有一個情境讓你感受到「這七件事是同時在一個作業上發生的」**。

「一支特徵排程從首次上線到穩定營運」這條線，在我的實際工作是這樣展開的：

1. 首次上線：選 `INSERT OVERWRITE` 指定分區，確認冪等（§7.2）
2. 把它接進 DAG：設上游相依 sensor、下游等我的 `_SUCCESS`（§7.3）
3. 補歷史：用 `backfill` 分批、丟到獨立佇列，`current_date()` 換成 `logical_date`（§7.4）
4. 上線後：設 SLA、在 Airflow 加 retries、`set -euo pipefail`、盯 History Server（§7.5）
5. 例行維護：每週 compaction + `ANALYZE`、每季清過期分區（§7.6）

這五個步驟，**現在讀完章節後我能組出來，但我需要自己串**。補一個「§7.X 端到端走查：一支特徵排程上線 30 天」的節，讓讀者跟著一個完整情境走一次，可以解決以下問題：

- 讓「cron 讀者」看清楚每一步如果用 cron 需要補哪些手工（強化工具選擇的決策感）
- 讓回填→維護→監控的時序清楚（現在這三件事感覺是平行存在的，但其實有前後關係）
- 給一個「首次上線檢查清單」的框架，這是我每次上新排程都需要但現在要自己拼的東西

### 該放哪

**放在 §7.7 速查表之前**（即現有 §7.6 之後、速查表之前），編為 §7.7，速查表順延為 §7.8。

理由：
- 走查節是「把前面五節串起來」，應該在所有原則節讀完後、進入「查找工具」（速查表）之前出現
- 放在速查表之後感覺像附錄，放在前面讓速查表成為走查的「備查索引」

### 走查節應串哪些內容

最小可行版（不需長）：

1. **情境設定**：一張 `cust_feature` 表、月底快照、下游兩個模型 DAG
2. **首次上線前的三件事**：確認 `INSERT OVERWRITE` 冪等 → 補 3 個月歷史（backfill 分批）→ 設 sensor 等上游
3. **上線後第一週**：SLA 設多少、cron 補 `set -euo pipefail`、History Server 看第一次跑的基準
4. **一個月後**：首次 compaction + `ANALYZE`、確認 _SUCCESS 給下游用的一致性
5. **30 天回顧**：哪些訊號代表要介入（時間 / 列數 / retries 次數）

這樣的走查大約 400–600 字就足夠，不需要覆蓋所有 edge case，只要讓讀者感受到「這七件事是同一個故事的不同章節」。

## 三、三級彙整

### 強項（維持現狀）

- **footgun 密度高且實用**：`INSERT OVERWRITE` vs `INSERT INTO`、`partitionOverwriteMode`、欄位錯位、`current_date()` 不可重現、`set -euo pipefail`——這些是我實際踩過且難解釋給同事的問題，本章集中說清楚是最大價值。
- **工具對比一致**：每節都有 Airflow / dbt / cron 三者的落地，比較視角讓讀者能對應自己的環境。
- **速查表是本章精華**：症狀導向的查找格式，是我最可能在出事當下翻到的頁面。

### 需補強（優先順序高 → 低）

**P0 — 補 capstone 端到端走查節（§7.7，放速查表之前）**
- 原因見上節。這是讓本章從「原則手冊」升格為「上線劇本」的關鍵缺口。

**P1 — §7.5 補列數監控**
- 時間監控只能抓「跑慢了」，抓不到「產出少了」。對共用特徵庫這個讀者關心的場景，列數基準線是更高優先的監控指標，目前完全缺席。

**P1 — §7.3 補 sensor 跨週期 execution_date_fn 說明**
- 這是 Airflow `ExternalTaskSensor` 最常見的設定錯誤（上游月底、下游每週），不提是明顯缺口。

**P2 — §7.2 補 staging 表過渡 / write-audit-read**
- 直接覆寫線上分區在共用表場景有「覆寫到一半被讀到」的窗口問題，staging 過渡是業界標準答案，本節未提。

**P2 — §7.4 補回填進度持久化範例**
- 「可中斷續跑」只說了原則，cron 場景的進度記錄缺具體做法（哪怕 5 行 shell）。

**P3 — §7.6 補維護頻率建議**
- 「排進排程」但不說多久一次，讀者還是得自己猜。至少給一個「每次寫完就 ANALYZE、每月一次 compaction」的預設建議。

### 總體結論

本章是**強章**，核心 footgun 覆蓋率高、工具對比清楚。缺口主要集中在兩個方向：（一）**深度不夠的地方**在 §7.3 sensor 跨週期、§7.5 列數監控、§7.6 維護頻率——這些都是長期營運特徵庫一定會遇到但現在沒答案的問題；（二）**capstone 走查節的缺席**讓讀者雖然知道每個原則，但缺少一個「把七件事連起來的完整劇本」。補了 capstone 後本章可以作為特徵排程上線的實際操作手冊，目前還差半步。
