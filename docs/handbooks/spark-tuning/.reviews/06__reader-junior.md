## §6.6 CDP 實務二：Impala 看不到新資料？（L152–182）

### 聚焦審稿問題：§6.6 是否太密讀不下去？

**我的判斷：沒有讀不下去。** 密度可以接受。真正的卡點是術語，不是段落密度。

- **開場場景超好**（L152）：「Spark 半夜灌了新分區，早上同事用 Impala 一查——出不來」——這正是我或同事會遇到的真實問題，有了動機才有心情看解法。
- **原因講清楚**（L154）：「Impala 把 metadata 快取在自己的 catalog 裡，外面改了它不會自動知道」——因果能接。
- `[卡]` **catalog**（L154 正文、L156 mermaid 圖）：**這節核心概念是「Impala 的 catalog（自己的快取副本）vs 共用的 Metastore（源頭）」，但 catalog 這個詞第一次出現時沒有跟 Metastore 的關係並排講**。我讀到「catalog」會想「catalog 跟 §6.1 說的 Metastore 是同一個嗎？還是不同的？」。L154 括號有補「catalog 是 Impala 私有的那份 metadata 快取；它的源頭仍是大家共用的 Hive Metastore §5.8——兩者不同：Metastore 是共用源頭、catalog 是 Impala 自己的快取副本」——這句補得很重要，也補到了，只是埋在括號，能見度偏低，建議升格為主句。
- `[卡][圖]` **HMS**（L156 圖節點「catalogd 輪詢 HMS＝Hive Metastore 的事件」）：圖的節點裡有「HMS」縮寫，**括號雖有補「HMS＝Hive Metastore」**，但是在同一個節點文字裡（節點文字很擠），讀圖時眼睛滑過容易漏掉。整體算有補，但能見度偏低。`[誤讀?]` 若讀者夠仔細可接上，屬邊界。
- `[卡-輕微]` **catalogd**（L156 圖、L172 正文）：`-d` 結尾是 daemon 的命名慣例，§6.2 有解釋 `impalad` 的 d，但沒有一句「凡是 `-d` 結尾都是 daemon」的通則，讀者要自己悟。`[誤讀?]` 有 §6.2 的 impalad 解釋在前，合理可推。

- 一個小混亂：節點 C「Impala 的 catalog 知道嗎？」拉出**兩條**邊，一條是「CDP 自動同步已開（catalogd 輪詢 HMS 事件）→ G 看得到」、另一條「不知道（手上是舊快取）→ D 看不到」。第一條的條件是「自動同步有沒有開」、第二條是「知道不知道」——**兩條邊的條件不在同一個維度**（一條是設定層、一條是結果層）。讀者得想一下才理清「自動同步開了=catalog 知道=直接到 G；沒開或沒追上=不知道=到 D」。這個邏輯通，只是圖上沒有明說「自動同步開=catalog 知道」的等價關係。

- L176「Spark 寫完接一個 REFRESH、別賭自動同步剛好追上」——**可操作、能照做**，是全章最接地氣的落地建議之一。

讀不讀得下去：**讀得下去，全節沒有密度問題**。真正的卡點是 catalog / catalogd / HMS 三個術語的能見度偏低（解釋都在括號裡），建議升格到主句或圖旁 note。

## §6.7 CDP 實務三：ACID／交易表跨引擎的限制（L186–228）

### 聚焦審稿問題：ACID 從零講起，讀不讀得懂？

**我的判斷：讀得懂，但要撐過中段的術語密集區。** 前小節（「先補基礎」）是全章最用心的一段解說；跨引擎限制那張對照表是關鍵的配套補救。

### 先補基礎小節（L190–207）

- **這節標題就很體貼**：「不預設你學過資料庫交易」——我立刻知道可以放心看，不會被甩。
- **從「普通表只能整批覆寫」切入**（L192）：「你沒辦法說把第 12345 號客戶那列金額改成 500，那一列藏在 Parquet 檔中間、檔案不是設計來就地改一格的」——這個比喻我能接，因為我確實知道我現在改資料要先把整個月份全抓出來、改完再整批塞回去，代價高得要命。從這個痛點切入，動機很強。
- **ACID 字母有拆**（L194）：「ACID 是資料庫對交易的四個保證的縮寫」，然後列出 Atomicity / Consistency+Isolation / Durability 各自的白話解釋。白話版說得清楚，「改一半的爛狀態」「帳算到一半總額兜不攏的中間態」對我很有感。
  - `[卡-輕微]` **L194「交易（transaction）」**——括號補「不是銀行業務那種一筆交易帳的意思」——這個澄清對我（銀行業務分析師）**非常非常有用**，因為我這行「交易」幾乎 100% 是指「那一筆帳」，不是資料庫裡的意思。這句澄清放在括號裡，建議升格一下，因為這是最容易讓我混淆的地方。
- **增量檔機制**（L200）：「每次 update / delete 只寫一個小的變更紀錄（delta 檔），讀的時候把 base + delta 疊起來」——機制說清楚了，結合前面 §5.8 的記憶（compaction 合併增量），能接。
- **最實用的判斷**（L207）：「只有真的要改 / 刪既有的列時（更正、合規刪除、upsert）才值得用 ACID；整批換掉一個月就根本不需要 ACID」——**這句是 §6.7 最該帶走的決策，給在對的位置**，讓我先知道「多數情況其實不需要 ACID」再去看跨引擎限制，心理負擔小很多。

### 跨引擎的限制小節（L209–225）

- **表種分類先講清楚**（L211）：「managed 表下分 full ACID / insert-only；external 表是另一類」——這個分類樹在表格之前先用一句話立起來，我讀表格時就不必在腦中自己建關係。**比舊版（把表種分類散落在 bullet 裡）好很多**。
- **對照表**（L213-217）：一張「表種 × 引擎 × 可讀 / 可寫」的矩陣，`[卡-輕微]` full ACID / insert-only / external 三列我能接，但表格標題沒有一眼看出「managed 是 full ACID 和 insert-only 的父類，external 是另一個維度」——需要結合 L211 那句才理清。不算大問題，L211 的確先鋪了，屬可接受的跨句理解。
- 表格後 L219 括號補了 HWC 全名（Hive Warehouse Connector）——好，這裡才首次給全名，前面 §6.5 用過縮寫，這裡補全，稍晚但有補。
- **設計原則**（L221–225）：「要多引擎共用、就用 external Parquet，躲開所有 ACID 麻煩；只有要真改 / 刪既有的列，才值得扛 ACID」——**把一堆限制收成一句能照做的話，是這節的救贖**。

### §6.7 小結：ACID 從零讀得懂嗎？

**大致讀得懂**，尤其「先補基礎」小節是全章最用心的解說。真正的卡點：
1. **「交易（transaction）」的語義混淆**對銀行業務讀者是最高風險點，括號澄清雖有，建議加重。
2. **表格沒有立刻顯示「managed 是 full ACID / insert-only 的父類」**，需要倒回 L211 那句才能拼圖。

## 全章三級彙整

### A. 真缺陷（建議補）

1. **YARN / 容器：解釋只有一次、埋在 L60 括號，能見度嚴重不足**。YARN 是三引擎「快慢根本原因」（Impala 常駐省掉申請 YARN 容器的開銷）的核心詞，但全章正式解釋只在 L60 一個括號，§6.2 圖（T1）裡先出現時零解釋、§6.8 再出現時也不再掛鉤。對只懂 SQL 的讀者，「申請 YARN 容器」永遠是黑箱。**建議：把 L60 括號的 YARN / 容器解釋升格為正文一句，或在圖 T1 節點旁加 note。**

2. **MPP 只在 §6.2 來源框出現、正文零出現**。清單明文要抓的術語。**建議：正文補一句「Impala 是 MPP（大規模平行處理）架構」，或來源框刪掉這個正文沒鋪過的縮寫。**

3. **HMS 在 §6.6 mermaid 圖（L156 節點）出現時，縮寫解釋夾在節點文字裡、能見度偏低**。讀者快速掃圖時容易漏掉「HMS＝Hive Metastore」的連結。清單明文要抓的術語。**建議：圖旁加一條 note 或把節點文字改成「catalogd 輪詢 Hive Metastore（HMS）事件」，縮寫括號後置。**

4. **catalog vs Metastore 的區別**：L154 括號有解釋（catalog 是 Impala 的快取副本，Metastore 是共用源頭），但括號太低調——這個區別**正是 §6.6 整節的核心**，建議升格為小標題之後的第一句正文。

5. **§6.3 決策表「查詢複雜度」列三欄不同軸**：Spark/Impala 欄講複雜度、Hive 欄講 HQL（語言），讀者無法直接比較。**建議：Hive 欄改為「中等～複雜（含現有 HQL 報表）」，把 HQL 當括號補充而非欄位主詞。**

6. **§6.3 決策表「資料量」列 Impala 欄判準讀者對不上號**：「中間結果幾十 GB」SQL-first 讀者估不出來。**建議：補一句「不確定時先試跑、看 §6.4 query profile 有沒有一直 spill」，把估不出來的情況指向可操作的工具。**

### B. 可加強（斟酌）

1. **「選引擎怎麼操作」的前置說明**（「Hue 裡換編輯器就是換引擎」）目前藏在 §6.1 末段（L29），建議挪到 §6.1 開頭或章首前提框——這是讓「選引擎」從抽象變可操作的最關鍵一句，越早出現越好。

2. **daemon 命名通則沒點破**（§6.2 L60、§6.6 L172）：impalad / catalogd 的 `-d＝daemon` 邏輯沒對讀者說一次，讀者各自死記兩個詞。補一句「凡 `-d` 結尾的程序都是 daemon（一直開著的背景程序）」，後面遇到同類詞就能自己認。

3. **§6.4 fragment 沒給「≈stage」類比**，而 vertex 有給，不一致。補 fragment≈stage 一句。

4. **§6.7「交易（transaction）」對銀行業務讀者的語義混淆**：目前括號有澄清「不是銀行業務那種一筆交易帳」，但這對我（銀行業務分析師）是最高風險點，建議升格為小標題後第一句：「注意：這裡的『交易』是資料庫術語，不是你帳務系統裡的那筆交易。」

5. **§6.9 mermaid 虛線 vs 實線無圖例**：Hive 那條虛線的語義（「不一定讀這張表」）沒說明，建議圖旁加一條 note 或統一線型。
