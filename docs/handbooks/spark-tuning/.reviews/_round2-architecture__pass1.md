### 缺口彙整（標核心程度）

- **G1（核心缺口・必補）— reverse ETL／資料送出 Hadoop 邊界回業務系統**。
  兩條主線**都**指向這個終點：初階「出名單給 PM」、進階明文「reverse ETL 把資料送回客戶行銷及管理系統」。但全書最遠只到「external Parquet 表 + `REFRESH` 讓 Impala/Hue 能查」（06 §6.5/§6.6、§6.9 貫穿例的 PM 也是「用 Impala 看這張表」）。**從 HDFS/Hive 把資料推到一個外部 OLTP/行銷系統**（JDBC sink、檔案落地交換區、對接 API、Sqoop/Impala 匯出、推送節奏與冪等、回送失敗重試、PII/權限）——**完全無覆蓋**。這不是優化題、是營運線的「最後一哩」，而 spec §1/§11 的終極能力與讀者模型都明確要求它。**建議新增專節或專章（見結構提案 C1）。**

- **G2（可加強）— 特徵庫 training–serving 一致**。
  §9.3 有點到（「snapshot-partition 模型天生對齊」+「training–serving skew」一句 gloss），方向對、但對「營運共用特徵庫供模型訓練」這條主線**偏薄**：沒談「同一段特徵計算邏輯如何在 batch 訓練表與線上/批次推論之間共用、不寫兩份」「特徵定義的單一真實來源（dbt model / 共用 SQL）」「推論時讀哪個 snapshot、新特徵上線的對齊」。snapshot 模型確實消掉了 as-of join 的需求（這點寫得好），但 training–serving 一致 ≠ 只有 as-of。**建議在 09 §9.3 後補一個 `###` 小節或在 reverse ETL 章一併談（見 C1/C2）。**

- **G3（已知暫緩）— 07 PySpark DataFrame API**。
  進階主線「複雜可重用、要單元測試的特徵邏輯」確實需要它（spec §5 自己點名「本 repo pipeline 即一例」）。**user 刻意暫緩、先寫營運線**＝合理排序，不是缺陷；但**長期營運特徵庫者最終會需要它**（純 SQL 很難測試/重用複雜特徵）。**判定：保留、但優先序排在 reverse ETL 之後、速查表之前。**

- **G4（可加強）— ad-hoc「先小樣本/先 Impala」的初階入口**。
  初階主線第一個動作常是「在 Hue 開個查詢探資料」。02 是「先量再調」的診斷章（偏向已經有慢查詢要驗屍），但「**ad-hoc 探索期該怎麼省**」（先 `LIMIT`、先 Impala 小樣本、別全表 `COUNT(DISTINCT)`）散在 03/06，沒有一個「初階入口場景」收口。**這正是 10 場景索引要做的事**——所以 G4 由 ch10 covers（見下）。
