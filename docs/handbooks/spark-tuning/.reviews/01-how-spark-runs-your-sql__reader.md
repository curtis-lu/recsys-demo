## 1.1 你的查詢，其實是一群機器一起做（L9–L36）

**[好的]** 用「自己筆電一顆 CPU 從頭讀到尾」對比，非常貼近我的經驗，立刻懂「Spark 不是這樣」。partition 用「切成很多塊、散在多台機器」+ 信用卡帳務表 3000 萬筆切幾百塊、每塊幾十萬筆——具體數字落地，這段我完全跟得上。Driver/Executor 用「指揮官 / 工人」比喻，好懂。

**[術語沒先定義 — 真缺陷] L33「CDP」「YARN」**
- 原文：「在 CDP 上，這些 executor 要用多少台、多大記憶體，是由 YARN（叢集的資源管家）分配的。」
- 「YARN」當場有括號解釋（資源管家），OK。但「**CDP**」完全沒解釋，憑空冒出。
- 我會這樣想：CDP 是什麼？是我們銀行用的某個平台嗎？是 Spark 的一個版本嗎？我不知道，這個縮寫對我沒有任何意義。我會卡一下，然後猜「大概是某個叢集平台」繼續讀，但心裡留個疙瘩。
- 建議：CDP 第一次出現要當場一句話說明（例如「CDP（我們公司跑 Spark 的那個叢集平台）」）。

**[概念圖不自明 — 可加強] L20–31 第一張 Mermaid 圖**
- 圖本身好懂：SQL → Driver → 三個 Executor，每個處理一部分 partition。箭頭方向清楚。
- 小卡點：圖裡 Driver 指向 Executor 的箭頭（D --> E1 等），我會理解成「Driver 把工作派給 Executor」。但 1.1 開頭說 Driver 還要「把結果收回來」，圖上沒有反向箭頭表示結果回傳。我會想：「結果怎麼回到 Driver 的？」這是小事，但圖只畫了單向，跟文字「收結果」對不太起來。
- 屬可加強，不影響理解主軸。

**[太抽象 → 但有救 L35「主軸」]**
- 「資料怎麼切、要不要在機器之間搬動，就決定了快慢——這是本手冊反覆出現的主軸」——這句很好，把為什麼重要講清楚了，且預告了主軸。讚。

## 1.2 Spark 不會馬上算：先攢計畫，再一次跑（L39–L51）

**[好的]** lazy evaluation 用「攢成一份待辦計畫、先不動手」解釋，action 用「真的要結果的事」+ 三個例子（存表/撈回來看/count），這段我懂。中文「延遲求值」也有附。

**[術語沒先定義 — 可加強] L48「collect」、L48「Hue」**
- 原文：「把結果撈回來看（例如 `collect`、或 Hue 裡按下執行去顯示資料）」
- 「`collect`」：我沒寫過 Spark 程式，只會 SQL。`collect` 是程式裡的一個指令嗎？對我（純 SQL 使用者）這個詞沒被解釋。我會猜「大概是某個把資料拉回來的指令」。可接受，因為旁邊有中文「撈回來看」，但嚴格說它是個沒定義的 API 名。
- 「Hue」：也是憑空出現的工具名。我會想：「Hue 是我們查 SQL 的那個網頁介面嗎？」如果是，那其實對我很親切（因為我就是用那個跑 SQL）；但作者沒說 Hue 是什麼，我得自己猜。建議一句話點明（「Hue（你平常打 SQL 的那個網頁工具）」）——這樣反而能拉近距離。

**[好的] L51「為什麼這件事重要」**
- 解釋了 lazy 的好處（Spark 看得到整份計畫才能優化，例如把 WHERE 提早、砍掉用不到的欄位）。我懂為什麼要先攢再跑了。

## 1.3 從 SQL 到一群 task：計畫怎麼變成實際工作（L55–L74）

**[概念圖不自明 — 真缺陷] L59–66 第二張 Mermaid 圖（SQL → Logical Plan → Catalyst → Physical Plan → Stage → Task）**
- 圖裡一口氣出現「**Logical Plan**」「**Physical Plan**」兩個全英文術語，圖上只有「要做什麼 / 實際執行步驟」的小字註解，但這兩個詞在**圖出現的當下還沒在正文定義過**。
- 我會這樣想：Logical Plan、Physical Plan 是什麼？「邏輯計畫 / 實體計畫」？我大概從註解猜到「要做什麼 vs 怎麼做」，但這兩個正式英文名詞我是第一次看到，圖直接丟給我。
- 正文（L68 之後）只回頭定義了 Catalyst / Stage / Task 三個詞，**唯獨 Logical Plan 和 Physical Plan 沒有在「先對上號」清單裡**。讀者會發現圖上有 5 個英文方塊，正文只解釋了 3 個。這是明顯的落差。
- 建議：要嘛在「先對上號」清單補上 Logical/Physical Plan 的一句話定義，要嘛圖上就別用英文（用「邏輯計畫」並在旁邊說清楚）。

**[術語沒先定義 — 真缺陷] L64「資料重分配」vs 後面的「shuffle」**
- 圖上 L64：「切成 Stage / 每遇到一次資料重分配切一刀」。這裡用「資料重分配」這個詞，但 1.4 才會把這件事正名為「shuffle」、1.3 正文 L71 又說「需要搬資料就切下一個 stage」。
- 我會想：「資料重分配」「搬資料」「shuffle」是同一件事嗎？三個地方用了三種講法，我不確定它們指的是不是同一個動作。對讀者來說術語不統一會懷疑「是不是有不同的東西我沒分清」。
- 建議：1.3 提到切 stage 的那一刀時，至少預告一句「這個『搬資料』之後會正名為 shuffle」，把三種講法綁成一個。

**[好的] L72 Task 定義**
- 「一個 partition 對應一個 task，100 個 partition 就是 100 個 task」——具體、好記，前後呼應 1.1 的 partition。讚。

**[鷹架洩漏？— 微調] L74「這正是下一節」**
- 原文：「所以『為什麼要切 stage』這個問題，答案就藏在『搬不搬資料』——這正是下一節。」
- 「這正是下一節」是給讀者的導覽，不算純鷹架。但寫作指引 §6 對「先說/這一節」類後設旁白很敏感。這句偏導覽、可接受，屬微調等級（若要更乾淨可改成「下一節就講搬不搬資料」）。

## 1.4 兩種運算：自己算的（便宜）vs 要交換的（貴）（L78–L104）

**[好的]** 窄依賴/寬依賴用「能不能自己算完 vs 得跟別塊交換」區分，配上 SQL 例子（WHERE/SELECT/逐列計算 是窄；GROUP BY/JOIN/DISTINCT/ORDER BY 是寬）。這些 SQL 我天天寫，能直接對上自己懂的東西，非常好。

**[術語沒先定義 — 可加強] L82「narrow」、L85「wide」、L88「key」**
- 「narrow」「wide」英文括號緊跟中文，OK。
- 「**key**」（L85「同一個 key」、L86「join key」）：對 SQL 使用者，key 大概能從上下文猜成「join 或 group by 用的那個欄位」。但它沒被當場定義。我會想：「key 就是我 GROUP BY / JOIN ON 的那個欄位吧？」——多半猜對，但嚴格說是沒定義的詞。建議首次出現點一句（「key 就是你 GROUP BY 或 JOIN ON 用的那個欄位」）。

**[概念圖不自明 — 可加強] L90–104 第三張 Mermaid 圖（窄 vs 寬/shuffle）**
- 窄依賴那半邊（partition A → 結果 A，partition B → 結果 B）一看就懂。
- 寬依賴那半邊：partition A、B 都標「客戶 1、2 的交易混在一起」→ shuffle 圓圈 → 拆成「客戶 1 的全部交易 / 客戶 2 的全部交易」。這個其實畫得不錯，能看到「混在一起 → 重新分配 → 按客戶聚齊」。
- 小卡點 1：圖裡用「客戶 1、客戶 2」，但 1.5 的例子用「cust_id」。讀者腦中要自己把「客戶 1」對上「cust_id」。不算大問題。
- 小卡點 2：shuffle 那個圓圈節點 `SH(("shuffle 重新分配"))` 用了雙圓括號（Mermaid 的圓形節點），形狀跟方塊不同。我（不懂 Mermaid）只會看到一個圓形寫著 shuffle，OK，但不知道圓形 vs 方形有沒有特別含意。其實沒有，是純樣式，不影響理解。
- 整體：這張圖比第二張自明得多，因為它配了具體的「客戶」資料。可加強而非缺陷。

## 1.5 為什麼 shuffle 是頭號敵人（L108–L128）

**[非常好]** 這節是全章最札實的。用一條真實 SQL（每位客戶這月刷多少）、3000 萬筆、散在幾百 partition、同客戶交易散落各處，然後把 shuffle 拆成 **Shuffle write（寫本機磁碟）** + **Shuffle read（跨網路拉桶）** 兩步，最後點出「幾乎每一筆都經歷一次落地磁碟 + 過網路」。這完全是寫作指引 §3/§4 要的「具體數字落地 + 貫穿範例」。我讀到這裡才真正體會到「貴」是貴在哪。

**[術語沒先定義 — 可加強] L123「分好桶」「桶」**
- 原文：「按 cust_id 分好桶，寫到本機磁碟」「再從其他機器跨網路把對應的桶拉過來」。
- 「桶」是這裡第一次出現，沒解釋。我會想：「桶是什麼？是按 cust_id 分組後的一堆小檔案嗎？」大致能從上下文猜（按 key 分出來的一份份資料），但它是個沒定義的比喻詞。可加強：第一次用「桶」時點一句它指什麼。

**[缺脈絡 — 可加強] L128「資料傾斜」**
- 原文：「最容易出問題（記憶體不夠、資料傾斜）的環節。」
- 「**資料傾斜**」憑空出現、沒解釋。我會想：「傾斜是什麼意思？資料歪掉？」這是一個之後章節大概會展開的重要概念，但在這裡只丟了名詞沒給一句話直覺。
- 建議：要嘛一句話點明（「資料傾斜＝某些 key 的資料量特別大，把某台 executor 壓垮」），要嘛標明「第 0X 章會談」。否則讀者卡在這個沒解釋的詞上。

**[好的] L128 結論句**
- 「CPU 算數很快，但寫磁碟、過網路慢得多」——把為什麼貴的根本原因講白了，誠實且具體。

## §1.4 是否「太進階以致一般讀者讀不動」——明確評價與建議

**評價：不是「太進階讀不動」，而是「進階方向對、但呈現太密、且位置打斷主線」。** 拆三點：

   - **HDFS**：第一次出現點一句它是什麼（「HDFS＝叢集存放資料的地方，§1.1 那些 partition 就放在這」），把它接回 §1.1 已建立的概念。
   - **JVM GC**：這條對本讀者最難。建議**大幅簡化或降級**——要嘛砍掉只留「記憶體開太大反而可能有額外的管理停頓（後面章節談）」，要嘛用一句完全不靠 JVM 知識的白話（「記憶體太大時，系統定期整理記憶體會卡頓一下」）。現狀「JVM 的垃圾回收會出現長停頓」對沒學過的人是三層黑盒，是整節最該動的一句。

3. **建議拆段 + 加一個小例子（針對「太密」）：**

4. **位置（節奏）問題——這條比上面更值得考慮：** §1.4 夾在 §1.3（四層）和 §1.5/1.6（shuffle）之間，但它**深度依賴 shuffle/spill 的直覺（為什麼 spill 不好＝磁碟慢，這要 §1.6 才講）**，卻排在 shuffle 之前。導致 §1.4 一直在用「之後才解釋的東西」（shuffle、spill、磁碟慢）。我讀的時候會覺得：「為什麼把最硬、又依賴後面知識的一節，插在我還沒學會 shuffle 的時候？」——建議評估**把 §1.4 整節移到 shuffle（現 §1.6）之後**，這樣 spill「為什麼不好」、shuffle「廣播為什麼省」都有了根基，§1.4 會好讀非常多。若不便移動，至少在 §1.4 開頭加一句「這節會先用到後面才細講的 shuffle/磁碟慢的直覺，看不懂的詞先記個大概，§1.6 會補」。

## 第二輪彙整（三級）

### 真缺陷（必補）

1. **§1.3 L61 stage 定義架在未解釋的「shuffle」上，且「（下一節會講）」指向錯誤**（術語沒先定義）—— stage 是最該懂的一層卻只能死記；且下一節（§1.4）講的不是 shuffle，shuffle 在 §1.5/1.6。修：指向改準＋給 shuffle 一句話直覺讓 stage 定義當場站得住。
2. **§1.3 L77–84 第二張圖與第一張圖、與「四層」的關係沒串起來**（概念圖不自明／太抽象）—— 兩張圖一個是結構、一個是流程，正文沒點明；且第二張流程圖漏了 application/job 兩層，卻接在「四層怎麼從 SQL 變出來」之後。修：圖前一句點明「結構 vs 流程」，並讓流程圖帶上或明確省略前兩層。
3. **§1.4 L115 三個沒接住的新詞：spill / HDFS 吞吐 / JVM GC**（術語沒先定義）—— 這是 §1.4 從「跟得上」掉到「跳過」的斷點，尤其 JVM GC 是三層黑盒。修：spill、HDFS 各補一句白話並接回已有概念；JVM GC 大幅簡化或降級為不靠 JVM 知識的白話。
4. **§1.4 整節太密、無落地錨點**（太密）—— 代價清單是純抽象因果，缺 §1.6 那種具體數字畫面。修：拆段（記憶體面／系統面）＋補一個「80GB 分給 20 vs 5 個 task」的迷你數字例。

### 可加強（斟酌）

5. **§1.4 位置/整章節奏：executor 取捨排在 shuffle 之前，卻依賴 shuffle/spill 的直覺**（太密/節奏）—— 評估把 §1.4 移到 shuffle（現 §1.6）之後；若不移，§1.4 開頭加一句「會先用到後面才細講的詞，先記大概」。中段連續三節高負載（1.3→1.4→1.5/1.6）缺緩坡。
6. **§1.3 L64–73 第一張圖缺圖例**（概念圖不自明）—— 虛線（shuffle 切開）vs 實線（展開成 task）含意不同卻無說明；Job 2 沒畫 stage 易誤會「有的 job 沒 stage」。修：補一句圖說、Job 2 象徵性補一個 stage 標「省略」。
7. **§1.3 L79/L81 Logical/Physical Plan 仍是「先出現在圖、後解釋在文」**（術語沒先定義）—— 比第一輪進步（L86 已補定義），但順序仍反。修：把定義移到圖前，或圖上改用中文。
8. **§1.4 L104 wave**（太密，輕微）—— 「批」對譯已夠，可選擇性加「一批 50 個、跑完換下一批」讓意象更實。現狀可接受。

### 誤讀（不改或微調）

9. **§1.4 GC 的括號中文「垃圾回收」** —— 有對譯是好的，但對譯不等於解釋（為什麼記憶體大就停頓仍沒說）；歸到上面第 3 點一起處理，不單獨改。
10. **第一張新圖的圓形/方形節點** —— 同第一輪結論，純樣式不影響理解，不改。
