# 08 章 reader 審稿（初階分析師視角）

**審稿人設定**：會 SQL、懂銀行業務資料、沒學過分散式系統；日常工具＝Hue/Impala ad-hoc + Spark 排程出名單 + 產特徵表（但沒有 ML 工程背景，feature/label/training/serving 要當場解釋才懂）。看到沒解釋術語會卡，嚴格扮演不放水。

## 一、逐節卡關點

### §8.1 本章地圖

**無明顯卡點。**

「資料產品」一詞沒有單獨定義，但段落馬上接「好幾個模型、好幾個同事長期取用的表」，我能接上。`mermaid` 圖在 Hue 上不會渲染，但章節文字已把四個面向說清楚，不影響理解。

### §8.2 資料品質驗證

**卡點 2-1｜「label」第一次出現，只有括號解釋，解釋稍短。**

> 「label〔要預測的答案，例如「會不會買」「會不會違約」〕」

這是 §8.2 正文第一次出現 label，括號內的解釋是有的，算是及格——但對完全沒碰過建模的人，「你要模型預測的那個答案」這個定義要等到 §8.3 才完整說清楚（「**label**＝你要模型預測的那個答案」）。第一次在 §8.2 見到時，括號解釋稍微站不穩：我知道「答案」，但還不知道為什麼驗 label 的值域特別重要（要等讀完才能回頭理解）。**建議移到 §8.2 初次出現時把定義說完整，或在括號後加一句「模型就是靠讀很多筆 feature+label 學會預測」。** （類型：術語落地不夠快）

**卡點 2-2｜「grain」出現，括號解釋清楚，可操作。**

> 「grain——也就是「一列代表什麼」、例如「一個客戶一個月一列」」

這個解釋時機剛好，沒有卡點。標記為通過。

**卡點 2-3｜「宣告式」的解釋邏輯通，但 dbt 本身是第一次真正操作到。**

§8.2 要讀者在 YAML 裡寫 `tests:`，但對一個只用 Hue 的人，這是第一次被要求對著一個**我不確定我有沒有裝**的工具寫設定。文中沒有提「這段是 dbt 的作法、如果你還沒有 dbt，直接跳到下面純 SQL 版」。我會本能地停下來想：「我要先安裝什麼嗎？」（第 07 章有提三層工具，但沒有說明 dbt 和純 SQL 版是**二選一**，不是都要做。）**卡關行動：猶豫我是否能跳過 dbt 段，找不到明確許可。** （類型：操作前提不清）

**卡點 2-4｜`dbt_utils.unique_combination_of_columns`：套件名稱突然出現，不知道怎麼裝。**

> 「需 `dbt_utils.unique_combination_of_columns`（dbt-utils 套件）」

正文說「得用 dbt-utils 套件或自訂 generic test」，但完全沒說怎麼啟用這個套件，只在 footer 提了一行。我的第一反應是：「這是另一包要安裝的東西嗎？還是 dbt 本來就有？」如果已在第 07 章說明，這裡應加一個前向引用。（類型：操作步驟缺失）

**卡點 2-5｜`spark-sql -S -e` 這個 shell 指令，來自哪裡？**

```bash
violations=$(spark-sql -S -e "SELECT COUNT(*) FROM ...")
```

「`spark-sql`」對我來說是陌生指令。這是 Spark 附帶的工具還是另裝的？在哪個環境有？能在 Hue 裡跑嗎？文字只說「把查詢結果安靜地印到 stdout」，沒說這個指令的前提。我的排程腳本目前是 cron，所以這段對我應該最有用，但這個指令讓我卡住。（類型：術語/工具前提未說明）

### §8.3 時間點正確性與特徵洩漏

**本節是對我最重要的一節——但也是術語最密集的一節。**

**卡點 3-1｜「特徵（feature）」在此節第一次正式定義——但 §8.2 的「label」比它早出現。**

§8.3 開頭才給出：
> 「**特徵（feature）**＝描述每個客戶的一個數字／屬性…，模型拿它來預測；**label**＝你要模型預測的那個答案」

這是本章第一次把 feature 和 label **並排完整說清楚**。這個位置其實還不錯，但它出現在 §8.3，而 §8.2 已用了「label」——序列上有一個小洞（先用再解釋）。對我這種讀者，第一次讀 §8.2 時 label 意義模糊，讀到 §8.3 才豁然開朗，但寫筆記時要翻回去。**這是可接受的小缺失，但如果 §8.1 地圖節能加一句「特徵＝描述客戶的屬性；label＝要預測的答案」，全章從一開頭就站穩。** （類型：定義出現時機稍晚）

**卡點 3-2｜「training」和「serving」這兩個詞，括號解釋清楚，可接受。**

> 「（**training**＝用歷史資料訓練模型那一步；**serving**＝上線後、真的拿模型去對每個客戶算分數那一步。）」

解釋在第一次出現的同一句，可以照著走。無卡點。

**卡點 3-3｜「entity」突然出現在 mermaid 圖說裡，沒有解釋。**

> 「**entity** 在這裡就是客戶 `cust_id`」

這句在括號裡出現，及時解釋了，可以過。但「entity」在整本手冊的語境（不只這章）是「那個被推薦/被服務的主體」，這裡只說「就是客戶」——若後面的章節把客戶換成店家、換成商品，讀者可能一時搞不清楚 entity 是通用詞還是特定欄名。建議第一次出現時加一句「這是通用名稱，在銀行例子就是客戶」。（類型：術語通用性未說清）

**卡點 3-4｜「training–serving skew」出現，沒有括號解釋。**

> 「training–serving skew，訓練與上線的特徵算法不一致」

「skew」這個詞在第 02/03 章出現過，指的是資料傾斜。這裡用 skew 指「不一致」，是不同語境的同一英文詞。括號後的中文解釋（「訓練與上線的特徵算法不一致」）足夠，但第一次讀到「training–serving skew」這個組合詞，我先被「skew」帶歪了（想到 §2 的 skew 資料），才反應過來這是新詞。**建議加一句「這是另一種洩漏的近親，跟資料傾斜的 skew 不同概念」。** （類型：同詞異義混淆）

**卡點 3-5｜「as-of join」旁註段：我讀不懂，但被要求「不熟可跳過」——跳過判斷點很好。**

旁註段明確說「進階旁註，不熟可跳過」，這個設計很好，我不會卡在這裡。唯一小問題：旁註說「Spark 3.3 沒有原生 AS OF JOIN 語法」，對我來說「原生」是新詞（是說預設沒有，還是怎麼樣都做不到？）。可以改成「Spark 3.3 的 JOIN 語法沒有 AS OF 這個選項，得自己繞道」。（類型：術語「原生」未解釋）

### §8.4 多消費者契約

**整節卡點較少，是本章可操作性最高的一節。**

**卡點 4-1｜「fan-out」術語：括號說明「扇出（fan-out）給 N 個模型」——清楚。**

「扇出」有中文在前，無卡點。

**卡點 4-2｜「破壞性變更」在「真要改語意，用新增＋過渡＋棄用」段落中——我沒看到具體步驟。**

段落說：
> 「開一個新欄 `feat_x_v2`、和舊欄並存一段過渡期、通知下游遷移、確認沒人用了再刪舊欄」

這是一個完整的四步流程，夠清楚。但「通知下游遷移」這一步——怎麼通知？誰是下游？對我（特徵產出人）來說，「下游」是別人的模型，我不一定知道有誰在用。文章沒有說如何找出下游使用者（只說有這個責任，沒給找的方法）。這在我的實際工作中是一個摩擦點。（類型：操作步驟不完整——這是特徵產出者的視角盲點，對其他章節不一定適用，這裡點出即可）

**卡點 4-3｜「就緒訊號應該是品質閘通過後才落」——但我不知道誰來落。**

> 「理想的就緒訊號（§7.3 的 `_SUCCESS`／sensor）應該是**品質閘通過後才落**」

§7.3 在前一章，這裡有前向引用。但「品質閘通過後才落」這個「落」誰來做？是我手動建一個空檔？是 dbt 自動生？是 Airflow task 結束時？讀者視角看到這句不知道要新增哪一行 code 或哪個設定。（類型：操作步驟不完整）

### §8.5 資料版本與可重現性

**招式一到三的邏輯很清楚，是本章結構最好的一節。**

**卡點 5-1｜「可重現」一詞在標題就出現，但到段末才說清楚它的用途。**

「可重現（reproducibility）」對我來說最自然的理解是「跑兩次結果一樣」（重現性）。文章實際說的是「三個月後能重新算出當初那份特徵長什麼樣」，這是不同的概念（歷史重現，不是結果穩定性）。我在前三行裡沒有立刻抓到這個定義，要讀到「當有人問『當初訓練那個模型、用的 5/31 特徵到底長什麼樣？』」這句才豁然開朗。**建議在「可重現」第一次出現時，直接加括號說「可重現＝三個月後能重算出當初那份特徵」**。（類型：術語定義時機稍晚）

**卡點 5-2｜「Iceberg、Delta Lake」提到「內建 time-travel」——但我不知道我們用的是哪個。**

> 「有些較新的表格式（Iceberg、Delta Lake）內建 time-travel…但本書環境是**純 Hive ＋ Parquet/ORC，沒有 time-travel**」

這句把選擇說清楚了（我們沒有），所以不影響操作。無卡點，通過。

**卡點 5-3｜招式二的 `audit 帳本`：`INSERT INTO` vs `INSERT OVERWRITE`——footnote 有說，正文也有說，但兩者混在一起讀有點費力。**

正文說明 `INSERT INTO`（append）和前章 `INSERT OVERWRITE`（覆寫）的差異，解釋得到位。唯一小問題：`built_by STRING -- 哪支作業（如 airflow:build_cust_feature）`，我不用 Airflow，這個欄位我填什麼？可加「沒有 Airflow 則填作業腳本名稱即可」。（類型：工具前提下的操作模糊）

**卡點 5-4｜招式三的 current-build view 用了 `ROW_NUMBER`——這是新語法嗎？**

view 程式碼裡用到 `JOIN ... ON f.snapshot_date = latest.snapshot_date`，這我能讀懂（就是一般 join）。但招式三到 §8.6 的 current view 換成了 `ROW_NUMBER() OVER (PARTITION BY ...)`——對我來說這是第 02/07 章沒有出現過的 window function。我是知道 window function 的（Hue 上用過），但這裡第一次在版本管理的脈絡看到它，沒有任何說明（只有程式碼，沒有一行「這用了 window function，原理見…」）。**如果讀者的 window function 基礎來自前面的章節，這裡有個教學落差。** （類型：語法前提未說明）

### §8.6 進階：設計模式與歷史回補

**本節有進階警語「只做當期產表的人，§8.5 三招就夠了」——這個分流設計非常好，我不會強迫自己讀完。**

對我（會碰到回補、但還不是「長期經營特徵庫」等級）的卡點主要集中在新術語。

**卡點 6-1｜「bitemporal（雙時間軸）」——術語馬上解釋，清楚。**

> 「在資料倉儲裡叫 **bitemporal（雙時間軸）**——雙層分區就是它最樸素的實作」

中文在後、英文在前，馬上解釋，無卡點。

**卡點 6-2｜「active-version 指標」與「current view 改指向」——我不明白「指標」在這裡是什麼具體形態。**

> 「**active-version 指標**」

「指標」對我第一反應是 KPI（metrics）或 SQL 的 measure。文章實際說的是「一張記錄『目前對外用哪個版本』的表」，但這個理解要讀到後面 `feature_version_promotion` 的 DDL 才建立。在標題和第一段先出現「active-version 指標」，我會卡一下。（類型：術語在定義前出現）

**卡點 6-3｜「blue/green 部署」——這是軟體工程術語，對我完全陌生。**

> 「這其實就是資料版的 **blue/green 部署**」

這個比喻對有 DevOps 背景的人很直覺，但對我完全沒有意義（我連「部署」這個詞在銀行 IT 語境也不常用）。比喻反而讓我更困惑：什麼是 blue、什麼是 green？文章沒有解釋這個比喻的基礎含義，直接用它類比。**建議把比喻換成或補充「就像新版程式碼先在測試機備好、確認沒問題才切到正式機、出問題就切回去」。** （類型：比喻前提缺失）

**卡點 6-4｜「Write-Audit-Publish（WAP）」——縮寫在此首次出現，解釋在旁邊，清楚。**

> 「**這個模式有個業界名字：Write-Audit-Publish（WAP）**」

接著馬上給了對應表，三步對應本章，清楚。無卡點。

**卡點 6-5｜「SCD Type 2」——術語沒有任何解釋，直接給結論「比分區級 build_version 細」。**

> 「**列級版本化（SCD Type 2）**：在每一列加 `effective_from`／`effective_to`／`is_current`」

「SCD」是 Slowly Changing Dimension 的縮寫，但這裡沒有展開。若有 DW 背景知道 SCD，這沒問題。但對只用 Hue 做 ad-hoc 的人，這個縮寫是未知術語。文章給了足夠的功能描述（每一列加生效區間），技術上能理解這是什麼，但不知道為什麼叫 SCD Type 2（Type 1 是什麼？）。**我可以略過縮寫的來源，只接受功能描述。建議在縮寫後加括號「（緩變維度，一種維表設計法）」。** （類型：術語縮寫未展開）

## 三、三級彙整

### 🔴 一級：我完全卡住、沒辦法繼續操作

1. **§8.6「blue/green 部署」比喻**（卡點 6-3）：這個比喻對我沒有意義，而它被當作 §8.6 設計模式的類比核心，沒有補充說明讀者視角的人無法把這個類比對應到「產出/發佈分離」的實際步驟。

### 🟡 二級：可以繼續讀，但這個詞/步驟留下問號

2. **`spark-sql -S -e` 指令的前提**（卡點 2-5）：不知道這個指令在我的環境有沒有，也不知道能不能在 cron 裡用。
3. **dbt vs 純 SQL 是「二選一」而非「都要做」**（卡點 2-3）：我需要一句「如果沒有 dbt，直接跳到下面純 SQL 版」才知道我能跳過。
4. **`label` 在 §8.2 先用、在 §8.3 才完整定義**（卡點 3-1／2-1）：讀 §8.2 時定義站不穩，要靠記憶補全。
5. **「active-version 指標」在具體定義之前出現**（卡點 6-2）：術語在說清楚之前已被用了，要讀到後面才能回頭理解。
6. **`training–serving skew` 中的「skew」易與資料傾斜混淆**（卡點 3-4）：需要一句「這跟資料傾斜的 skew 是不同概念」。
7. **「可重現」的定義時機**（卡點 5-1）：對我最自然的理解是「跑兩次一樣」，而非「能重現歷史版本」，定義要讀到後面才清楚。

### 🟢 三級：小字眼問題，不影響操作，潤飾用

8. **`entity` 的通用性未說清**（卡點 3-3）：「在這裡就是客戶」夠用，但若讀者看到其他章的 entity 指不同對象會困惑。
9. **`SCD Type 2` 縮寫未展開**（卡點 6-5）：功能描述足夠，縮寫來源未說，讀者可忽略縮寫本身。
10. **`ROW_NUMBER() OVER (...)` 在 §8.6 未加說明**（卡點 5-4）：window function 對會 Hue 的人不陌生，但這裡沒有任何銜接（「這用了 window function」），是一個小教學空白。
11. **「原生（native）」未解釋**（卡點 3-5）：可改成「Spark 3.3 的 JOIN 語法沒有 AS OF 這個選項」。
12. **§8.2 的「通知下游遷移」步驟未說如何找出下游使用者**（卡點 4-2）：實務摩擦點，但不影響理解這個原則。
