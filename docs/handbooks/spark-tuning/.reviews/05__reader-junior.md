### 5.5 管好檔案大小：小碎檔問題

**這是本章最長的一節，也是審稿重點。**

#### 5.5 主體：成因 / 徵兆 / 解法

- **成因**：流程圖清楚列四條路，每條成因在圖下方展開說明。「openCostInBytes」那個算術例子（1 萬個 1MB 檔虛胖成 400 task vs 理想 77 task）是具體數字，讀起來很有感。
- **術語卡關**：
  - **窄依賴（narrow dependency）**（T）：說「只有 `SELECT`／`WHERE`、沒有會 shuffle 的 `JOIN`／`GROUP BY`」——括號給了口語解釋，讀得過去，但「窄依賴」這個詞第一次出現。[T，已部分解決]
  - **`spark.sql.files.openCostInBytes`**（T）：提到這個參數，但只說「預設約 4MB」，沒解釋它是做什麼用的、在哪設定。我知道這個值影響打包帳，但若我自己的環境想驗證，不知道去哪找。[T]
- **「先澄清一個常見誤會」方塊**：解釋了「檔案數 ≠ executor 數」，這個釐清很必要，但它插在「成因」和「徵兆」之間，打斷了節奏。我讀到這裡先跳開、再跳回來，有點失去脈絡。如果把這個方塊移到「解法」之前（因為它是在幫讀者理解「為什麼用分區數而不是資源大小來控制檔數」），可能更順。 [O]
- **徵兆**：「最直接：列出表的目錄看檔案大小」這條很好，是我馬上能做的事。第二條「讀過第 02 章再看 Spark UI task 數」給了讀過前章的引導，OK。

#### 進一步：`REPARTITION` 還是 `COALESCE`？放在哪？

這個子節是我覺得**最容易過載的地方**。

問題一：**內容層次在一段內跳了三個維度**：
1. `COALESCE` vs `REPARTITION` 的基本差別（表格）
2. `COALESCE` 的「往上游傳染平行度」的副作用（比表格更深一層）
3. `REPARTITION(n)` vs `REPARTITION(n, col)` 又是另一個維度的差別

這三個是不同層次的問題，讀者理解一個之後馬上要接另一個，中間沒有喘息。  [L]

問題二：`COALESCE` 的「上游傳染」說明很重要，但措辭比較技術：「n 就是那段的天花板」「叫一個 task 扛全部上游」。對不熟悉 task execution 的讀者，需要一個具體場景（比如「你有 240 個 task 在算 GROUP BY，後面接 COALESCE(1) 就等於把那 240 個 task 全壓成 1 個 task 一個一個跑」）。 [O]

問題三：**`REBALANCE` 出現的位置**。它藏在 `REPARTITION(n, col)` 的說明括號裡，後面有「（懶得自己猜 n 時，這個最省心）」——對我這種初階讀者，這是很有用的懶人選項，但它藏得太深，讀到這裡我的注意力已經在 `REPARTITION(n, col)` 身上，很容易錯過這個關鍵提示。 [O]

#### 兩個實戰問題

**問題一（分區表要帶分區欄位）**：有流程說明「動態分區 + 裸 REPARTITION(n) → 每個分區目錄 n 個檔」，這個雷我以前沒意識到，很有用。說明邏輯清楚。

**問題二（多個 CTE，hint 放最外層 SELECT）**：清楚，沒卡關。

#### §5.5 小結：是否過載的明確判斷

**判斷：過載。建議拆 `###` 子節。**

理由：

1. 成因 / 徵兆 / 解法 = 一個完整的「診斷 → 修法」迴圈，可以獨立成 `### 小碎檔的成因、徵兆、解法`。
2. `REPARTITION` vs `COALESCE` 的比較（包含三個子維度）= 一個純工具說明，可獨立成 `### REPARTITION vs COALESCE：怎麼選`。
3. 兩個實戰問題 = 進一步細節，可獨立成 `### 兩個常見陷阱：分區表與多層 CTE`。

拆開後每個子節有一個明確的讀者問題（「我的表有小檔嗎？」→「我應該用哪個 hint？」→「我會不會踩哪個雷？」），讀者可以按需求定點讀，不必一口氣吞完。

## 四、三級彙整

### A 級（影響讀者理解或操作的核心問題）

1. **§5.5 資訊過載，需拆 `###` 子節**：成因/徵兆/解法、REPARTITION vs COALESCE（三個維度）、兩個實戰問題，塞在同一個 `##` 節下連續閱讀容易失焦。建議拆三個 `###`，讓讀者知道自己在哪個問題上。
2. **「我要 REPARTITION 幾塊」缺計算步驟**：「資料量 ÷ 128MB ≈ 塊數」這個估算公式隱含在文中但沒有明說。§5.9 給了「4GB → 32 塊」的例子，但沒說怎麼算來的。補一句計算公式，讓讀者第一次照做時有依據。
3. **`COALESCE` 的「上游傳染平行度」缺具體場景**：技術說明「n 就是那段的天花板」對初階讀者不夠直覺，需要一個具體例子（「240 個 task 的 GROUP BY 後面接 COALESCE(1)，就等於把那段壓成 1 個 task」）。
4. **ANALYZE 的分區版語法延後到 §5.9 才出現**：排程分區表的場景最常用 `ANALYZE … PARTITION(…)`，但 §5.6 語法範例沒有這個版本，需查到 §5.9 才找到。建議在 §5.6 補上或加引導。

### B 級（讀起來有點卡，但不影響能否照做）

5. **`REBALANCE` 藏太深**：藏在 `REPARTITION(n, col)` 說明的括號裡，但它是最省心的懶人選項，應提高能見度（例如在 COALESCE vs REPARTITION 表格下面補一行「不確定 n？用 REBALANCE」）。
6. **§5.1 地圖沒說清楚 bucketing 的閱讀模式**：bucketing 列在主軸但本文結論是「先不用管」，地圖沒有預先告知，讀者在 §5.1 可能把 bucketing 列為重點準備認真學。
7. **「常見誤會」方塊的位置**（executor vs 分區數的澄清）：插在成因和徵兆之間，打斷節奏，移到解法前更自然（因為它是在解釋「為什麼控制分區數，不是調資源」）。
8. **§5.2 進階子節的行動指引被稀釋**：「把最常用來過濾的欄排好再寫出」是具體可做的事，但淹在一大段原理說明中。若用加粗或獨立成一句「**實作要點：**」，可讀性更好。
9. **compaction 的因果**（§5.8）：「增量檔累積 → 需定期 compaction」的因果在同段內，但沒有明顯連接，讀者可能只記得「要做 compaction」但不知道為什麼。

### C 級（細節或不影響操作的術語問題，低優先）

10. **`spark.sql.files.openCostInBytes`**：提到這個參數的預設值但沒說在哪查/改。不影響行動，但好奇心強的讀者可能想知道。
11. **「成本式（Cost-Based）」的「成本」**：CBO 說「成本式優化器」，但「成本」是什麼沒說（是 CPU？IO？shuffle？）。對初階讀者，「成本」可以加一句「這裡的成本指估算的 IO 與計算量」。
12. **`spark.locality.wait` 的控制對象**（§5.4 補充）：說「官方說預設多半夠用」，但沒說它控制什麼。這節的結論是「不用動」，所以不影響行動，但有讀者可能想知道它是等待幾秒的意思。
