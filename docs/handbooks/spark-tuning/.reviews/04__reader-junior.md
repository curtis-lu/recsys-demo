> 人設：會 SQL、懂銀行業務資料、日常 Hue/Impala ad-hoc + Spark 排程出名單 + 產特徵表；沒學過分散式系統；看到沒解釋術語會卡。

## §4.1 心法：先別急著轉旋鈕——AQE-first

### 讀得懂的部分
- 「SQL 層旋鈕 vs 資源層旋鈕」的兩類區分非常清楚，用「低風險可回退」vs「高風險啟動時定死」解釋，直觀。
- 「OOM（記憶體爆掉）或佔住資源餓死同事」——這個比喻立刻讓我覺得資源層是真的有後果的事。
- 整節路徑圖（§4.1→§4.2–4.3→§4.4→§4.5–4.7→§4.8）在這裡先說，讀起來有方向感。

### 卡關點

**[術語-未解釋]** `spark.sql.shuffle.partitions` 和 `spark.executor.memory` 在此節首次出現時只是舉例、沒有說「這串字是什麼東西」。我知道這是「設定值」，但它寫在哪裡？Hue 裡可以看到嗎？還是某個設定檔？這一節沒有交代，要到 §4.3 才知道可以用 `SET` 查。如果我跳著讀（這是很多人的習慣），會不知道這串英文我能做什麼。
> 我會這樣問：「`spark.sql.shuffle.partitions` 這串字我要在哪裡找到？是我打查詢的地方嗎？」

**[概念-需橋接]** 「OOM（記憶體爆掉）」——括號解釋是好的，但對我來說 OOM 是「電腦當掉」的印象。Spark 的 OOM 是什麼現象？查詢直接報錯？還是自動重試？後面 §4.5 有詳細講，但這裡提了卻沒給短句指路。
> 我會這樣問：「OOM 到底會發生什麼？我的查詢就直接噴錯？」

## §4.2 AQE 自動幫你做的三件事

### 讀得懂的部分
- 三件事用 mermaid 流程圖呈現，視覺化後很好理解。
- 前兩件事（①合併小分區、②SortMergeJoin→BroadcastHashJoin）都有前章節錨點（§1.6、§3.5），是很好的橋接。
- 「在跑之前估計」vs「執行途中真正量到」的對比解釋了 AQE 為什麼比傳統優化器好，我能理解。

### 卡關點

**[術語-第一次出現未解釋]** **`advisoryPartitionSizeInBytes`**——括號裡說「預設 64MB」，但這個名字的「advisory」是什麼意思？它只是「建議」大小、AQE 不一定嚴格遵守嗎？對我來說看到這個長名字會停頓，但花了兩秒又繼續讀，因為「每塊約 64MB」已經說清楚了，這個是次要問題。
> 我的反應：「這個字讓我多想了一下，但還是讀得下去。」

**[術語-第一次出現未解釋]** **`SortMergeJoin`** 和 **`BroadcastHashJoin`** 在此以縮寫形式出現，第 03 章有討論過 broadcast，但 SortMergeJoin 是第一次出現這個完整術語名稱。雖然能從字義猜「要先排序才能合併的 join 方式」，但對沒學過分散式系統的我，不確定猜得對不對。
> 我會這樣問：「SortMergeJoin 跟我平常寫的 JOIN 有什麼不一樣？是說 Spark 內部做 JOIN 的某種方式嗎？」

**[概念-需橋接]** 「把它切成幾小塊並行處理，化解那個把整個 stage 拖住的肥 task」——`stage` 這個詞假設我記得第 01 章，但若我是初次讀到這裡，不確定 stage 是什麼。括號說「§3.10 講過」，但沒說 stage 是什麼概念。建議在此節某處加一句「（第 01 章講的 stage／task 分層）」當輕量提醒。
> 我的反應：「stage 這個字出現了但沒解釋，我要回去翻第 01 章嗎？」

## §4.3 確認 AQE 開著、怎麼用 `SET` 改設定

### 讀得懂的部分
- **這是全章對初階讀者最有操作價值的一節。** `SET` 語法說明、確認 AQE 的兩個地方（SQL 頁籤 + Environment 頁籤）、表格區分 SQL 層 vs 資源層——我覺得這三件事講得非常清楚。
- 「徵兆」那段（你 `SET spark.executor.memory=16g` 之後去 Executors 頁籤一看，記憶體還是原本數字）很實用，因為這真的會讓我以為設定失敗了、實際上是 SET 對資源層無效。
- 表格裡的 ✅ / ❌ 讓我一眼抓到重點。

### 卡關點

**[術語-第一次出現未解釋]** **`Livy`**——「在 CDP 上，這個工作階段常透過一個叫 **Livy** 的服務…來開」。這是什麼？是我在 Hue 裡操作的東西嗎？我每天用 Hue 但沒看過 Livy 這個字出現在界面上（可能在後台？）。對我來說這個字造成輕微的「啊？我沒用過這個」感。
> 我會這樣問：「Livy 是什麼？我需要知道它嗎？還是它是平台幫我管好的，我不用管？」

**[概念-需橋接]** `static SQL config` 的例外說明（`spark.sql.warehouse.dir` 會報錯）——這個補充是好的，但對初階讀者，「warehouse.dir」不一定有直覺感（雖然在銀行 Hive 環境用 Hue 的人可能知道 warehouse 是什麼）。這個小括號算是補充知識，不影響主線閱讀。算可接受。

**[可讀性-結構]** 這一節最後兩段（`SET` 對資源層無效 → Livy → 請平台管理者）資訊量很高，但沒有小標或分段，有點密。初階讀者讀到這裡可能要重讀一遍。建議：「Livy」那段可以加一個小提示框，例如「實務上不用理解 Livy 是什麼；重點是：資源層的設定不在你的查詢框裡調，要在開啟工作階段的地方給」。

## §4.4 AQE 之後，還值得手動懂的少數 SQL 旋鈕

### 讀得懂的部分
- 三個旋鈕用表格整理，格式清晰。
- 「三個都記同一個原則：§03 改寫法、§05 補統計都做完之後才輪到的微調」——這個錨點讓我知道優先序。
- `shuffle.partitions` 在 AQE 下「角色變了」的解釋（只併不拆）讓我理解為什麼不必每次都手動調。

### 卡關點

**[術語-第一次出現未解釋]** **`spill`** 在表格「取捨/風險」欄出現「每塊太大狂 spill」，這是第一次在 §4.4 正文裡以動詞使用。雖然前章有提，但在這個表格裡沒有括號解釋，不查前章的人可能卡住。
> 我會這樣問：「spill 是什麼？是查詢報錯嗎？還是變慢？」

**[術語-第一次出現未解釋]** **`files.maxPartitionBytes`** 的「讀檔時每個 partition 多大（§1.2）」——`partition` 在第 01 章有解釋，這裡指的是「讀資料時切的塊」，對我來說能理解，但「大檔想要更高讀檔平行度」這個句子裡的**平行度**是第一次在正文裡出現這個詞（前章可能提過，但沒有括號解釋）。
> 我會這樣問：「讀檔平行度是什麼意思？切成更多塊有什麼好處？」

**[概念-可加一行]** 表格「`shuffle.partitions`」的「AQE 開時多半不用動」——如果我是新手，我可能會問「那我要怎麼知道什麼時候需要動？」正文有說「資料超大時 200 起點太少」，但沒給「超大」的量感（例如「幾十億筆」）。對初階讀者，加一個粗略量感會有幫助。不過這不算強烈缺陷。

## §4.5 一個 executor 的記憶體裡裝了什麼：execution／storage／overhead

### 讀得懂的部分
- mermaid 圖把記憶體層層拆解，視覺化後有助理解。
- spill 的救法優先序（先減量→提高平行度→最後才加記憶體）是操作性強的結論。
- 「官方明說建議維持預設」——這句讓我放心不用去動 `memory.fraction` 和 `storageFraction`。

### 卡關點

**[標籤-進階節未明確標示]** §4.1 說 §4.5–§4.8 偏進階，但章節本身沒有「進階/可跳」的提示。我讀到 §4.5 時，不確定這一節是「我應該讀」還是「可以跳」。對初階讀者，建議在節標題後明確加一行，例如「> 進階閱讀 — 如果你目前只做 ad-hoc 查詢，可先跳到 §4.8 的全章串連，有需要時再回來」。

**[術語-累積負擔]** §4.5 在短短一節裡出現了：`heap`、`JVM`、`overhead`、`unified memory`、`execution memory`、`storage memory`、`M`（大池）、`R`（保留區）——這些術語雖然在圖裡都有，但密度很高。`JVM`（Java 虛擬機）有括號解釋，是好的；`heap` 沒有括號解釋，只說「JVM 拿來放資料物件和運算暫存的主記憶體區」——這樣解釋還可以，但對完全沒程式概念的人仍是新詞。
> 我會這樣問：「heap 是我可以用的空間的名字嗎？跟我 `SET executor.memory=8g` 設的那個是同一件事嗎？」（答案在正文有，但要多讀一遍才找到。）

**[概念-公式記法]** `M ＝（heap − 300MB）× spark.memory.fraction` 這個算式對 SQL-first 的人是偏技術的表達。正文有說「切 0.6 給 M」，這樣反而更直白。公式出現的必要性有點微妙——如果這節是「進階可跳」，那公式放著沒問題；但如果初階讀者被引導看這節，公式會讓他們有點卡。

**[術語-第一次出現未解釋]** `cache()` / `persist()` 在「storage memory：你用 cache()／persist()…存起來的資料放這」——這是第一次在本章出現，說「第 10 章」，對初階讀者：我知道有個 cache 的概念，但我日常 Hue ad-hoc 不太用 cache，不確定這段是不是我現在要管的事。

## §4.6 給 executor 配多少：core、記憶體、台數的實際設定

### 讀得懂的部分
- 「先提醒一個常見誤判：`executor.memory` 開源預設 `1g`，但你的 CDP 環境幾乎一定被改過」——這個接地氣，防止我拿錯預設值去做計算。
- 工作範例（100 core / 400 GB → 20 台 × 5 core × 18g）把 §1.7 的概念翻譯成實際指令，很有用。
- bash 指令範例（`--num-executors 20 --executor-cores 5 --executor-memory 18g`）直接可抄。

### 卡關點

**[標籤-進階節未明確標示]** 同 §4.5，沒有「進階/可跳」的明確提示。

**[術語-第一次出現未解釋]** `--num-executors` 在 bash 範例裡出現，但前面設定表格裡寫的是 `spark.executor.instances`。這兩個是同一件事嗎？初階讀者看到「instances 在這裡、num-executors 在 bash」可能會不確定對應關係。
> 我會這樣問：「`--num-executors` 和 `spark.executor.instances` 是同一個設定嗎？」

**[概念-公式]** 「從總額反推 heap：20 ÷ 1.1 ≈ 18 GB（不是 ×0.9）」——括號強調「不是 ×0.9」是有道理的（很多人會犯這個錯），這個細節對進階讀者有用。對初階讀者，這個算術本身不會難，但突然出現「÷1.1」的公式推導需要花點精神跟上。

**[可讀性]** 取捨段落（core 調多、台數調多、記憶體調大）三條的內容很扎實，但連續三個 bullet 都有「→」且都連到前章節，初次讀者需要在腦中同時追蹤三個方向。這對初階讀者偏密；如果這節本是「進階」，那沒問題。若要讓初階讀者也讀，可加一句總結「這三個方向都有副作用，所以在資源層動之前，先確認 §4.5 的減量做過了」。

## §4.7 dynamic allocation 與多租戶：別佔著資源餓死同事

### 讀得懂的部分
- 「多租戶」有括號解釋（多個使用者／作業共用同一批機器）——清楚。
- mermaid 流程圖（backlog 多→多要 executor；閒置→還回去）很直觀，是我能理解的日常資源管理邏輯。
- 「CDP 預設是開的」這個環境差異的提醒非常重要，放在段落開頭很好。
- SLA 有括號解釋。

### 卡關點

**[標籤-進階節未明確標示]** 同上。這一節雖然是進階，但「別餓死同事」的觀念對初階讀者其實有實際意義（我就是會被餓死的那個人）。可以考慮讓初階讀者讀第一段（「dynamic allocation 是什麼、為什麼開」）、後面的 minExecutors / streaming 衝突標為「進階細節」。

**[術語-第一次出現未解釋]** **`external shuffle service`**——「靠的是一個獨立保管這些中間檔的常駐服務〔即 external shuffle service〕、或等價的 shuffle 追蹤機制——CDP 已經幫你設好，你不必管」。括號有解釋，且說「不必管」，這樣處理很好。但「shuffle 中間檔」的概念——是第 01 章講過的 shuffle write 那些暫存檔嗎？對初階讀者可能需要一個小確認：「（第 01 章 §1.6 那些 shuffle write 的暫存檔）」。

**[術語-第一次出現未解釋]** **`streaming 作業`**——「別對 streaming 作業開 dynamic allocation」。`streaming` 沒有括號解釋。對我來說，我的工作是批次排程出名單，不是 streaming，所以這段不是我現在的事；但我看到這個詞會停一下想「streaming 是什麼」。加一句「（即資料持續流入、即時處理的作業；批次排程不在此列）」就夠了。

## 三級彙整

### 【A 級：強烈建議修】——影響初階讀者理解主線

1. **§4.5 / §4.6 / §4.7 缺少「進階/可跳」標示**：§4.1 說了這些節偏進階，但節標題和正文都沒有醒目的「> 進階 — 可先跳到 §4.8」提示框。初階讀者不知道自己可以跳，硬撐讀下去會在術語堆裡卡很久。這是**結構問題**，影響整個進階節的可讀性定位。

2. **`Livy` 未解釋（§4.3）**：在說明「資源層要在啟動時給」的段落，突然出現「一個叫 Livy 的服務」。初階讀者日常用 Hue，可能從來沒有意識到 Livy 的存在；這個詞造成「我需要去學一個新東西嗎」的焦慮。建議加一句「（Hue 背後幫你開 Spark session 的服務，你不用直接操作它；重點是資源設定在這一層給）」即可。

3. **`spill` 在 §4.4 表格首次以動詞出現、無括號**：雖然 §4.5 有完整解釋，但在 §4.4 的表格「每塊太大狂 spill」出現時沒有括號，是小的術語遺漏。考慮在 §4.4 表格裡加「（溢寫到磁碟，見 §4.5）」。

### 【B 級：建議補強】——不影響讀懂主線、但會讓初階讀者停頓

4. **`SortMergeJoin` 第一次出現無解釋（§4.2）**：雖然從字義能猜，但對沒有分散式背景的讀者，這是 Spark 內部的 join 實現方式，值得一句話解釋，如「（Spark 預設的 join 執行方式，需要先把兩邊資料 shuffle 後排序再合併）」。

5. **`streaming 作業` 無括號（§4.7）**：這段對批次讀者不影響，但加一句「（即持續流入的即時資料流，批次排程不在此列）」能幫初階讀者快速確認「這不是我的事、繼續讀」。

6. **`--num-executors` vs `spark.executor.instances` 未說明對應（§4.6）**：bash 範例用 `--num-executors`、前面設定說 `spark.executor.instances`，初階讀者可能不確定這兩個是不是同一件事。

7. **§4.8 第 3 步 Summary Metrics 的錨點提醒**：加一句短說明（在 Spark UI 的哪裡看）會讓這節更獨立可讀。

### 【C 級：細節打磨】——不影響理解、僅影響閱讀流暢度

8. **§4.1 `OOM` 的現象描述**：括號解釋了 OOM 是「記憶體爆掉」，但沒說查詢會怎樣（報錯？重試？）。建議加一句，如「（查詢會噴 `OutOfMemoryError` 並失敗）」。

9. **§4.9 那段一句話過長**：建議拆成兩句，提升口頭複誦的可能性。

10. **`advisoryPartitionSizeInBytes` 名稱的 advisory 意涵**：這是次要問題，不影響讀者理解 64MB 的概念；但若有餘裕，可以加「（『advisory』意思是建議值，AQE 會盡量往這個目標湊，不嚴格保證每塊剛好 64MB）」。

## 整體評估

- **§4.1–§4.4 初階可讀部分**：整體讀得懂，邏輯線清楚。主要問題是 §4.3 裡 `Livy` 術語出現、§4.4 表格裡 `spill` 無括號，以及各「進階節」缺少明確的跳讀提示。
- **§4.5–§4.7 偏進階部分**：技術術語密度高，初階讀者若沒有「可跳」提示會硬撐。mermaid 圖有幫助，但 `heap`、`JVM`、`M`、`R` 等內部記法對初階讀者仍有認知負擔。
- **§4.8 串連節**：是全章最適合初階讀者的一節，情境貼近日常工作，步驟清楚，可操作。
- **最大結構問題**：章節標示說初階只需讀 §4.1–§4.4，但章節本身沒有對應的視覺提示（如「進階/可跳」框），容易讓初階讀者繼續讀進去然後被術語淹沒。
