## §6.6 CDP 實務二：Impala 看不到新資料？`REFRESH` 與 `INVALIDATE METADATA`（L144–176）

這節是清單點名「會不會太密、太重、讀不下去」的重點節。我的真實感受：**它沒有讀不下去——反而是全章組織最好的一節之一**，因為它有一個我能代入的場景在前面領路。

- **開場場景超好**（L146）：「Spark 半夜灌了新分區，早上同事用 Impala 一查——資料出不來、或說沒這張表」——這正是我（或我同事）真的會遇到的事，一句話就讓我「想知道答案」。動機先於指令，符合「動機先於公式」。
- **原因講得清楚**（L148）：「Impala 為了快，把『有哪些表、哪些分區、哪些檔案』快取在自己的 catalog 裡，外面改了它不會自動知道」——這個因果我完全懂，而且呼應了 §6.1「三引擎各自把 metadata 快取在自己手裡」的伏筆。
- `[卡]` **L148、L156 第一次出現「catalog」**（Impala 的 catalog）、L156 出現 **catalogd**、**HMS**。
  - **catalog**：上下文勉強能懂是「Impala 自己那份 metadata 快取／目錄」，但這個詞跟 §6.1 的「Metastore（中繼目錄）」很像，我會混淆「catalog vs Metastore 是同一個嗎？」。其實 catalog 是 Impala 端的快取、Metastore 是共用的源頭——這個關鍵區別正是本節核心，但「catalog」這個詞第一次用時沒把它跟 Metastore 的關係講明（要到圖 L156 才隱約看到 catalogd↔HMS 的箭頭）。**我會問**：Impala 的 catalog 和大家共用的 Metastore，是兩個不同的東西嗎？
  - **catalogd**（L156 圖、L172 正文）：又一個 `-d` 結尾的常駐程序（catalog daemon），跟 impalad 同套命名。L172 正文補「Impala 的 catalog 服務（catalogd）」算有交代，但「daemon」這個共通的命名邏輯（`impalad`/`catalogd` 的 d 都是 daemon）從沒對讀者點破，我只能各自死記。
  - **HMS**（L156 圖「catalogd 輪詢 HMS 事件」、L176/L278 來源框）：**HMS 第一次出現在 mermaid 圖裡，圖內零解釋**。HMS＝Hive Metastore Service，但圖上只給縮寫。我前面好不容易把「Hive Metastore」記成「中繼目錄」，這裡又冒出「HMS」這個縮寫，我**不會自動把 HMS 跟 Hive Metastore 連起來**。`[圖][卡]`：圖內第一次出現的縮寫沒展開，是清單明文要抓的（HMS）。**我會問**：HMS 是不是就是前面那個 Hive Metastore？（是，但圖沒講。）
- `[圖]` **L152–162 的 mermaid 流程圖**：這張比 §6.2 那張好——它是真的「流程」（寫資料→HDFS 更新→catalog 知道嗎？→分兩條路→各自用 REFRESH / INVALIDATE→看得到了）。箭頭有因果、看得懂。**但有兩個小卡點**：
  - 分歧節點 C「Impala 的 catalog 知道嗎？」拉出**三條**邊：一條「CDP 自動同步已開（catalogd 輪詢 HMS 事件）」直接到 G、一條「不知道（舊快取）」到 D。三條邊的條件不是同一個維度（一條講「自動同步開了沒」、一條講「知不知道」），我得想一下才理清「喔，自動同步開著就直接看得到、沒開／沒追上才要手動」。
  - 圖裡 D 再分兩條（「只是加了檔／分區→REFRESH」vs「新建表／改 schema→INVALIDATE」）——這個分流是本節最該帶走的決策，圖上呈現得清楚，好。
- **REFRESH vs INVALIDATE 兩段（L164、L166）會不會太密？** 我的判斷：**密度可以接受**，因為每段都用「輕量／很重」「優先用這個／少用」「資料變了結構沒變／結構層級變了」成對對比，而且各給了一句「用在什麼情況」。我讀完能總結成一句「資料變了用 REFRESH、表結構變了用 INVALIDATE」。`[擠-輕微]`：L164 一段內同時塞了「增量重載」「去 Metastore 拿表資訊」「從 HDFS 增量更新檔案與資料塊清單」「可只刷一個分區（給了語法）」「是同步的」——五個點擠在一段，對我這種讀者，「資料塊（block）」又是個沒解釋的詞（HDFS 把檔案切成 block），但這個細節其實不影響我下決定，可以更精簡。
- L168 ⚠️ 框「INVALIDATE METADATA 不帶表名＝全部失效、非常貴、絕不要這樣下」——**這個警告很值，是會害死人的雷，單獨拉出來醒目，好**。
- L170「能用 REFRESH 就別用 INVALIDATE」+ 官方原話——一句話原則，好記。
- L172「知道有這回事就好——CDP 多半會自動同步」這段是**體貼的收尾**：先教你手動指令、再說「其實多半自動」「但仍要懂，因為①自動有延遲②可能被關」。這個層次我覺得拿捏得好，不會讓我學完才發現「白學了反正自動」。
- L174「落地建議：Spark 寫完接一個 REFRESH，別賭自動同步剛好追上」——**可操作、能照做**，讚。
- **本節整體**：清單擔心的「太密讀不下去」我**沒有發生**——場景帶路 + 成對對比 + 一句話原則，反而是好讀的一節。真正的卡點是**術語**（catalog vs Metastore 沒辨異、HMS 圖內沒展開、catalogd/daemon 命名沒點破、block 沒解釋），不是密度。

## §6.7 CDP 實務三：ACID／交易表跨引擎的限制（L180–195）

這節是清單點名「ACID 限制會不會太密太重」的另一個重點節。我的真實感受：**比 §6.6 更吃力，因為術語更硬、且 ACID 到這裡才算解釋，但解釋得不夠正面**。

- `[卡]` **ACID 從 §6.1（L26）就出現，一路到這節（L182）才終於展開——但展開方式是「支援 insert／update／delete、用 ORC 存、背後靠增量檔累積變更」**。問題：
  - **「ACID」這四個字母本身到全章結束都沒拆**（Atomicity/Consistency/Isolation/Durability）。本章把 ACID 當成「能 update／delete 的交易表」的代名詞在用，沒解釋為什麼叫 ACID。對沒學過資料庫交易理論的我，ACID 始終是個不透明的標籤。`[誤讀?]`：05 若拆過字母可放行；但本章作為「第一次認真用 ACID 下決策」的章節，至少該一句「ACID＝支援交易（可安全地 update／delete）的表，名字來自資料庫四大保證」。**我會問**：ACID 是什麼的縮寫？為什麼「能改能刪」要叫這麼專業的名字？
  - **「增量檔累積變更」**（L182）+ §6.7 多次提的 **compaction**（L186「§5.8 提的 compaction 合併增量檔」、L191）——compaction 當場說是「合併增量檔」並指路 05，05 讀過能接。`[卡-輕微]`：但「為什麼 update/delete 會產生『增量檔』、為什麼要 compaction」這個機制本章沒講（05 講過就放行），不懂機制的話這裡只能死記「ACID 表有增量檔要合併」。
- `[卡]` **L184「Impala 能讀 full ACID 交易表，但不能建／寫」**——**full ACID** 第一次出現，和 **insert-only／micromanaged**（L185）、**managed**（L187）形成一組我分不清的近義詞：
  - full ACID vs insert-only vs micromanaged vs managed——L184–185 一口氣丟出四個「表的種類」，我得很努力才能拼出關係：managed（05 學過，Hive 預設建的）裡面又分 full ACID（可 update/delete）和 insert-only（只新增）。本節**沒有先用一句話把這個分類樹立起來**，就直接講「哪種 Impala 能讀、哪種能寫」，我會在「到底有幾種表、誰是誰的子集」這層先卡住。**我會問**：full ACID、insert-only、managed、external，這幾種表是什麼包含關係？能不能先給我一張「表的種類」的小地圖再講限制？
  - `[擠]` L184–187 四個 bullet 每一條都在講「哪種表、哪個引擎、能讀還是能寫」的排列組合（Impala 讀 full ACID 可、寫不可；insert-only 讀寫皆可；full ACID 寫歸 Hive；Spark 寫 managed 要 HWC）——這是**四個引擎×表種×讀/寫的矩陣**被攤成散文 bullet，我讀的時候要在腦中自己建表格。**這節最該有一張「引擎 × 表種 × 可讀/可寫」的小表，反而沒有**（§6.2/§6.3/§6.6 都有圖，這個最需要矩陣的地方卻只有 bullet）。`[範疇]`：圖示資源分配失衡——較好懂的 §6.2 給了大圖，最繞的 ACID 矩陣反而靠純文字。
- **L189–193 的設計原則「能用 external Parquet 就用，躲開所有 ACID 跨引擎麻煩」**——**這段是本節的救贖**，把前面一堆限制收成一句我能照做的話：「要多引擎共用又不想煩，就用 external Parquet；真的要 update/delete 才用 ACID，否則整批 partition 覆寫就夠」。這個「先問你到底需不需要 update/delete」的判斷我能照著做，讚。L193「如果需求其實是整批換掉一個月，就 INSERT OVERWRITE PARTITION + external Parquet，根本不需要 ACID」——具體、解脫感強。
- `[卡-輕微]` L187「Spark 要存取 managed／ACID 表通常要走 HWC（Hive Warehouse Connector）」——HWC 這裡才第一次給全名（前面 §6.5 L133 已用過縮寫但只說「§5.8 的橋接」）。給全名是好的，只是順序上縮寫先出現、全名後出現。
- **本節整體**：清單擔心的「太重」我認為**有部分成立**——不是因為篇幅，而是因為**術語密集（full ACID / insert-only / micromanaged / managed / ORC / 增量檔 / compaction / HWC 全擠在 15 行內）且缺一張該有的對照表**。救贖在最後的「external Parquet 一把收掉」原則。如果讀者撐到 L189 就豁然開朗；撐不到（被 L184–187 的表種矩陣繞暈）就可能放棄。**建議的不是刪內容，是補一張表種×引擎×讀寫的小矩陣 + 一句 ACID 字母解釋。**

## 全章彙整

### 一、主旨一致性

- **主旨清楚且全程一致**：從前提框（「上一層的決定：這件工作交給哪個引擎」）→ §6.1（選引擎不是選資料存哪、選錯只是慢不是錯）→ §6.3 決策樹 → §6.8 取捨 → §6.10 boxed 句，一條線拉到底，沒有跑題、沒有自相矛盾。
- **結論誠實、不過度宣稱**：反覆強調「樹是傾向不是法律」「界線分明只有一條（互動走 Impala、容錯重批次別走 Impala）」「選錯只是慢不是算不出來」——這些結論都對得起內文證據，符合寫作指引「寧可承認看情況」。沒有「其實很簡單調一下就好」式漂亮話。

### 二、範疇／篇幅平衡

- **核心（§6.2 跑法、§6.3 決策、§6.8 取捨、§6.9 串起來）佔比恰當**，是全章重心，對。
- `[範疇]` **唯一的失衡是「圖示資源分配」**：§6.2（已較好懂的「跑法」）給了一張大 mermaid，§6.9 也給了圖，但**最繞、最需要視覺化的 §6.7 ACID 表種矩陣（full ACID / insert-only / managed / external × 三引擎 × 讀/寫）反而只有散文 bullet**。該配圖的地方沒配、不太需要圖的地方配了。建議把一張「表種×引擎×可讀可寫」小表挪給 §6.7。
- §6.4（診斷工具）對「選引擎」主旨而言是**支線**（它談的是選定後怎麼診斷），佔一節篇幅可接受，但讀者要意識到它不是「怎麼選」的一環。不算失衡。

### 三、三級彙整

#### A. 真缺陷（建議補）

1. **YARN／容器 全章零解釋**（§6.2 L57–58 圖、L64、§6.8 L210）。這是本章被反覆使用（「申請 YARN 容器的啟動開銷」是 Hive/Spark「不夠快」的根本原因）卻從未定義的核心詞。對只會 SQL 的讀者，不知道 YARN 是 Hadoop 的資源調度器、「容器」是分到的運算資源，就無法真正理解「為什麼常駐的 Impala 快」。**至少一句**：「YARN＝Hadoop 的資源調度員，每次起 Spark/Hive 工作都要先跟它『申請一塊運算資源（容器）』，這申請就是那幾秒到幾十秒的開銷。」
2. **MPP 只在 §6.2 來源框（L68）出現、正文零解釋**。清單明文要抓。正文補一句「Impala 是 MPP＝大規模平行處理」，或來源框別用正文沒鋪過的縮寫。
3. **HMS 第一次出現在 §6.6 mermaid 圖內（L156）、圖內零展開**，且讀者不會自動把 HMS 連回前面記住的「Hive Metastore（中繼目錄）」。清單明文要抓。圖內或圖後補「HMS＝就是前面說的 Hive Metastore」。
4. **§6.7 缺一張「表種×引擎×可讀/可寫」對照表**，導致 full ACID／insert-only／micromanaged／managed 四種表 + 三引擎 + 讀/寫的矩陣被攤成散文 bullet（L184–187），讀者得自己在腦中建表。這是全章最該配圖卻沒配的地方。
5. **ACID 四字母全章未拆、未給一句「為什麼叫 ACID」**（§6.1 出現、§6.7 才用機制帶過）。至少一句「ACID＝支援交易（可安全地 update／delete）的表，名字來自資料庫四大保證」。
6. **§6.9 L249「本節沒有新東西」是後設旁白外洩**（寫作指引 §6 明文要抓的「對讀者講寫作決定」），建議刪。

#### B. 可加強（斟酌）

1. **daemon 沒明點「＝常駐程序」**，且 `impalad`／`catalogd` 共通的 `-d＝daemon` 命名邏輯沒對讀者點破（§6.2 L39/L60、§6.6 L156/L172）。點破一次，後面 `-d` 結尾的程序讀者就能自己認。
2. **§6.6 catalog vs Metastore 沒辨異**（L148）：Impala 的 catalog（自己的快取）和共用的 Metastore（源頭）的區別正是本節核心，卻沒在 catalog 首次出現時講明關係。補一句「catalog 是 Impala 自己手上那份 metadata 快取，源頭是大家共用的 Metastore」。
3. **§6.3 決策表兩個判準我對不上號**：(a)「資料量」列 Impala 欄「塞不塞得進記憶體」——SQL-first 讀者估不出來，建議給個粗界線；(b)「查詢複雜度」列三欄不同軸（Spark/Impala 講複雜度、Hive 講 HQL 語言），建議三欄統一講複雜度。
4. **§6.4 fragment 沒給「≈stage」類比**，而同表 vertex 有給（L111–113），不一致。補上 fragment≈stage。
5. **HQL 第一次出現（§6.3 L79）沒解釋**＝Hive 的 SQL 方言。SQL-first 讀者只認識「SQL」，不知道為什麼叫 HQL、跟我寫的 SQL 差在哪。一句帶過即可。
6. **章首缺「我平常怎麼指定用哪個引擎」的前置**（開頭 L31 標）：通篇假設「你能選引擎」，但沒講只會在 Hue 寫 SQL 的人實際要在哪裡、按什麼才會「用 Impala 而非 Hive 跑」。補半句場景（在 Hue 切換／在排程腳本指定）會讓「選引擎」這件事從抽象變可操作。
7. **§6.7 缺「表的種類小地圖」**：在丟「哪種能讀哪種能寫」之前，先一句話立起 managed⊃(full ACID + insert-only)、external 是另一類 的分類樹。
8. **§6.2 mermaid 的三個 subgraph 沒加值**（L63）：框內無箭頭、是「三張並排條列卡」，且是六個生詞的集中傾倒處。可考慮拆解或把生詞先在圖外鋪一句。
9. **§6.9 mermaid 虛線 vs 實線無圖例**（L239–245）：Hive 那條虛線的語意（「不一定讀這張」）沒說明，建議加一句或統一線型。
10. **OOM／block／C++ 等零星英文**第一次出現未翻（§6.2 L62 OOM、L60 C++、§6.6 L164 資料塊/block）。都是可略的細節詞，順手翻一下更好。

#### C. 誤讀（不改或微調）

1. **external Parquet／monthly_cust_txn／HWC／ANALYZE／compaction／spill** 等——都是 05（或 01）教過、本章當場有指路（「§5.8 的…」），對讀過前置章的我能接上，不算本章缺陷。屬正常的跨章呼應。
2. **ad-hoc**——業界常用詞，雖未翻成中文，一般分析師大致懂「臨時隨手查」。可降級，不必硬翻。
3. **Tez／DAG**——Tez 在 §6.2 L64 有較完整解釋（DAG 引擎、stage 間不落地、容器可重用）；DAG 嚴格說本章沒展開，但 01 章心智模型接近，屬邊界，列在 A4/B 的相關處即可，不單獨記為缺陷。
4. **章末「資料來源與精確度說明」「自動工具」字眼**——縮在免責區、不夾正文，對閱讀無礙，屬可接受的鷹架殘留。

### 結語（給作者的一句話）

**「能不能照著選」這個核心訴求——答得到。** §6.3 決策樹 + §6.8 取捨 + §6.10 boxed 句構成一條我能照走的路，最後一句「界線分明只有一條」尤其乾脆誠實。**真正擋路的不是邏輯、是術語**：YARN／容器／MPP／HMS／daemon／full ACID 這幾組基礎設施詞，在「沒學過分散式系統」的我面前反覆現身卻沒被定義，讓本來很好的敘事每隔幾段就絆我一下。補齊這幾個詞的「一句話定義」+ 給 §6.7 一張表種矩陣，這章對目標讀者就幾乎無懈可擊。
