# Spark 優化參考手冊

寫給數據部門同事的實用手冊：讓你能自己**讀懂查詢為什麼慢、改寫 SQL、調整 Spark 設定、選對引擎、設計合理的資料儲存，最終把資料可靠地交付出去**，把資料處理變快、變省、變可信。

「效率」在本手冊指三件事，遇到兩難時會明講取捨：

1. **運算時間**（查詢跑多久）
2. **記憶體使用**（會不會 OOM、會不會 spill 到磁碟）
3. **資料儲存**（檔案佔多少空間、好不好讀）

---

## 預設讀者與前提

**讀者**：會寫 SQL、熟悉自己的業務資料的分析師、資料科學家。**不需要**資料工程或分散式系統背景——partition、shuffle、executor 這些詞第一次出現時都會解釋。

**環境**（本手冊所有建議都對齊這個環境）：

| 項目 | 內容 |
|---|---|
| 計算引擎 | Spark 3.3.x（AQE 預設開啟）、Hive 3.1.3、Impala，跑在 CDP（Cloudera）平台上 |
| 底層 | YARN + HDFS |
| 你怎麼下指令 | 主要是 SQL（Spark SQL、Hive/Hue、Impala）；範例以 SQL 為主，DataFrame API 集中在第 10 章 |

> 資料量級參考（本手冊範例會用到）：客戶數 ~1000 萬、信用卡帳務 ~3000 萬筆/月、App ~1000 萬筆/天。

---

## 如何使用本手冊

不必從頭讀到尾。依你的情況切入：

- **想先建立直覺、看懂後面在講什麼** → 先讀第 01、02 章，這是其他章的基礎。
- **手上有個查詢很慢、想知道慢在哪** → 第 02 章（用 Spark UI 找瓶頸）→ 依症狀翻第 03/04/05 章。
- **想改 SQL 把查詢寫快** → 第 03 章。
- **想知道該不該、該怎麼調 Spark 參數** → 第 04 章。
- **要建表/產表，想讓它省空間又好讀** → 第 05 章。
- **同一個分析該用 Spark、Hive 還是 Impala** → 第 06 章。
- **要把排程跑得可靠、可重跑（冪等、回填、相依、監控、維護）** → 第 07 章。
- **要讓產出的資料產品可信（品質驗證、時間點正確性／特徵洩漏、特徵庫契約、資料版本）** → 第 08 章。
- **要把名單/特徵送回業務系統（reverse ETL 回 CRM/行銷平台）** → [第 09 章](09-reverse-etl.md)。
- **SQL 已經不夠用、想改用 PySpark** → 第 10 章（撰寫中）。
- **想看自己這類工作（ad-hoc / 排程產表 / 特徵運算）照哪些章** → 第 11 章（撰寫中）。
- **想快速查一個參數預設值、或某個名詞是什麼** → 第 12 章（撰寫中）。

> **兩條主線**：
>
> - **初階資料分析師**：01–06 建立優化能力，07–08 讓排程與資料產品可信賴，最終把名單交給 PM → **第 09 章**送出去。
> - **進階 analytics engineer**：在初階基礎上，第 10 章掌握 DataFrame API 工程能力，第 11 章按場景對號入座，第 09 章把資料 reverse ETL 回業務系統（CRM/行銷平台）。
>
> **把資料送出去（初階出名單給 PM、進階 reverse ETL 回業務系統）是兩條線的共同終點**。讀完 07–08 之後，都指向第 09 章。
