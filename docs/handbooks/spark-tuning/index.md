# Spark 優化參考手冊

寫給數據部門同事的實用手冊：讓你能自己**讀懂查詢為什麼慢、改寫 SQL、調整 Spark 設定、選對引擎、設計合理的資料儲存，最終把資料可靠地交付出去**，把資料處理變快、變省、變可信。

「效率」在本手冊指三件事，遇到兩難時會明講取捨：

1. **運算時間**（查詢跑多久）
2. **記憶體使用**（會不會 OOM、會不會 spill 到磁碟）
3. **資料儲存**（檔案佔多少空間、好不好讀）

---

## 預設讀者與前提

**讀者**：會寫 SQL、熟悉自己的業務資料的分析師、資料科學家。**不需要**資料工程或分散式系統背景——partition、shuffle、executor 這些詞第一次出現時都會解釋。

**環境**（本手冊所有建議都對齊這個環境）：

| 項目 | 內容 |
|---|---|
| 計算引擎 | Spark 3.3.x（AQE 預設開啟）、Hive 3.1.3、Impala，跑在 CDP（Cloudera）平台上 |
| 底層 | YARN + HDFS |
| 你怎麼下指令 | 主要是 SQL（Spark SQL、Hive/Hue、Impala）；範例以 SQL 為主，DataFrame API 集中在第 10 章 |

> 資料量級參考（本手冊範例會用到）：客戶數 ~1000 萬、信用卡帳務 ~3000 萬筆/月、App ~1000 萬筆/天。

不必從頭讀到尾。手冊有兩種入口：**依你手上的具體動作**（下面清單），或**依你工作的整體型態對號入座**（再往下的〔場景對應〕）。

- **想改 SQL 把查詢寫快** → [第 03 章](03-sql-tuning.md)。
- **想知道該不該、該怎麼調 Spark 參數** → [第 04 章](04-spark-config.md)。
- **要建表/產表，想讓它省空間又好讀** → [第 05 章](05-storage-efficiency.md)。
- **同一個分析該用 Spark、Hive 還是 Impala** → [第 06 章](06-engine-selection.md)。
- **要把排程跑得可靠、可重跑（冪等、回填、相依、監控、維護）** → [第 07 章](07-operating-pipelines.md)。
- **要讓產出的資料產品可信（品質驗證、時間點正確性／特徵洩漏、特徵庫契約、資料版本）** → [第 08 章](08-data-product-correctness.md)。
- **要把名單/特徵送回業務系統（reverse ETL 回 CRM/行銷平台）** → [第 09 章](09-reverse-etl.md)。
- **SQL 已經不夠用、想改用 PySpark** → [第 10 章](10-pyspark-dataframe-api.md)。
- **想快速查一個參數預設值、或某個名詞是什麼** → [第 11 章 速查與名詞表](11-cheatsheet-and-glossary.md)。

> **兩條主線**：

## 場景對應：先認出你在做哪種工作

數據部門的工作多半落在三種情境裡。先找到你現在在做哪一種，再順著它的「典型流程 → 該看哪幾章 → 最常踩的雷」走——不必把前面各章全記住，這裡是一張地圖，幫你對號入座、快速翻到該看的章。

| 你在做什麼 | 對應情境 | 主要服務 |
|---|---|---|
| 在 Hue 開個查詢、探資料、做一次性分析 | 情境① ad-hoc 探索 | 初階 |
| 把一段查詢變成每天/每週自動跑、產出表或行銷名單 | 情境② 排程產表/名單 | 初階 → 進階 |
| 經營多人共用的特徵庫、供模型訓練、把結果送回業務系統 | 情境③ 特徵庫 | 進階 |

### 情境①：ad-hoc 探索分析

**你是誰**：在 Hue 裡用 Impala/Spark SQL 探資料、回答一個臨時的業務問題、做一次性的分析。最在意「快點跑出來、別等半天」。

**典型流程**：開 Hue → 寫 SQL → 嫌慢 → 想辦法弄快。

**該看哪幾章**：

1. 先建立直覺：[第 01 章](01-how-spark-runs-your-sql.md)（為什麼有的查詢慢——shuffle 最貴）。
2. 嫌慢就先量：[第 02 章](02-diagnose-with-spark-ui.md)（用 Spark UI／`EXPLAIN` 看慢在哪，§2.10 三張 checklist 直接對著畫面打勾）。
3. 改寫法弄快：[第 03 章](03-sql-tuning.md)（裁分區、別 `SELECT *`、join 策略、`approx_count_distinct`）。
4. 選對引擎：[第 06 章](06-engine-selection.md)（**秒級互動的探索查詢，常常該用 Impala 而不是 Spark**——§6.3 決策表）。

**這個情境最常踩的雷**：

- **整張表全掃**：`WHERE` 沒帶分區欄、或把分區欄包在函式裡 → partition 裁剪失效（§3.2／§3.4）。探索期先 `LIMIT`、先挑一個月。
- **`SELECT *` 撈所有欄**：欄式存的 Parquet 只讀需要的欄才省，`*` 等於放棄這個優勢（§3.3／§5.2）。
- **對大欄 `COUNT(DISTINCT)`**：很貴；探索期能接受近似就用 `approx_count_distinct`（§3.8）。
- **走錯引擎**：拿 Spark 去跑一個秒級就該回的小互動查詢，光啟動就比 Impala 慢（§6.2／§6.3）。

### 情境②：定期排程產表 / 出行銷名單

**你是誰**：把一段算好的查詢，變成**每天/每週自動跑**、產出一張表或一份行銷名單交給 PM。最在意「每次都跑對、不會半夜爆、重跑不會壞」。

**典型流程**：算（SQL）→ 排程自動跑 → 產出表/名單 →（交給 PM 或推進系統）。

**該看哪幾章**：

1. 把查詢本身弄對弄快：[第 03](03-sql-tuning.md)、[第 04](04-spark-config.md)（AQE-first、資源配置）、[第 05 章](05-storage-efficiency.md)（產出表的 partition 設計、別產一堆小檔）。
2. 把排程跑得可靠：[第 07 章](07-operating-pipelines.md)（**這是這個情境的主場**——冪等可重跑、排程相依、回填、監控、維護；§7.7 有一支排程從 0 到穩定的端到端走查）。
3. 確保產出可信：[第 08 章](08-data-product-correctness.md)（出名單前先過品質閘、別出一份壞名單）。
4. 把名單送出去：[第 09 章](09-reverse-etl.md)（要 reverse ETL 推進行銷系統時）。
5. 排程作業退化排查：[第 02 章](02-diagnose-with-spark-ui.md) §2.11「漸進退化」（每週都比上週慢時）。

**這個情境最常踩的雷**：

- **重跑會變兩份**：用 `INSERT INTO`（append）當每日更新、或 positional `INSERT … SELECT` 欄位錯位 → 重跑/補跑就多一份或寫錯欄（§7.2）。用 `INSERT OVERWRITE` 指定分區求冪等。
- **`current_date()` 進排程 SQL**：回填過去某天時它取的是「今天」，不可重現（§7.4）。日期要當參數傳。
- **cron 靜默失敗**：腳本中途錯了卻回傳成功、`_SUCCESS` 照樣落 → 你以為跑好了其實是空的（§7.5，`set -euo pipefail`）。
- **產出一堆小檔**：shuffle 分區太多、或寫分區表沒先 `REPARTITION` → 下游讀很慢、NameNode 也累（§5.5）。
- **沒驗就出名單**：名單筆數異常（0 筆、或暴增）、含重複客戶卻直接交出去（§8.2 品質閘）。

### 情境③：特徵運算與共用特徵庫

**你是誰**：經營一張或多張**多人共用的特徵表/特徵庫**，供模型訓練用，可能還要把結果 reverse ETL 回業務系統。最在意「資料正確、可長期維運、不會悄悄出錯」。

**典型流程**：算特徵 → 驗 → 發佈版本 →（供訓練讀 / 推論讀 / 推回業務系統）。

**該看哪幾章**：

1. 儲存與共用：[第 05 章](05-storage-efficiency.md)（分區設計、`ANALYZE` 統計、external 表給多引擎共用）、[第 06 章](06-engine-selection.md)（多引擎讀同一份特徵的同步與 ACID 陷阱）。
2. 排程可靠：[第 07 章](07-operating-pipelines.md)（特徵排程的冪等/相依/回填/監控/維護）。
3. **資料產品正確（這是主場）**：[第 08 章](08-data-product-correctness.md)（品質驗證、**時間點正確性／特徵洩漏**、training–serving 一致、多消費者契約、資料版本與發佈）。
4. 送回業務系統：[第 09 章](09-reverse-etl.md)（reverse ETL、就緒閘、PII/稽核）。
5. 複雜可重用、要測試的特徵邏輯：[第 10 章](10-pyspark-dataframe-api.md)（何時從 SQL 換 DataFrame API）。

**這個情境最常踩的雷**：

- **特徵洩漏**：算某個時間點的特徵時用了「那當下還拿不到的未來資料」→ 離線評估虛高、一上線就崩（§8.3，本情境**頭號**風險）。
- **training–serving drift**：訓練一套算法、推論另寫一套，遲早不一致（§8.3 單一真實來源；§10.5 包成共用函式）。
- **改壞共用表**：把同名欄的語意改掉、刪欄、改型別 → 一次打爛 N 個下游（§8.4 只加不改契約）。
- **統計過期**：每週 append 新快照卻忘了 `ANALYZE` → 優化器用舊統計選錯計畫、查詢突然變慢（§5.6）。
- **reverse ETL 推半成品/非冪等**：把沒驗過或壞回補的資料推進業務系統、或重推變兩份（§9.3／§9.4）。
- **Spark 直讀 ACID 表**：不走 HWC 直讀 full ACID 表，可能讀進已刪除/舊版列、髒資料靜默進特徵（§6.7）。

### 三情境總表

| | ① ad-hoc 探索 | ② 排程產表/名單 | ③ 特徵庫 |
|---|---|---|---|
| **主場章** | 02→03→06 | 07（＋03/04/05） | 08（＋05/06/07/09） |
| **頭號雷** | 全表掃／走錯引擎 | 重跑變兩份／cron 靜默失敗 | 特徵洩漏／改壞共用表 |
| **求快 or 求穩** | 求快 | 求穩、可重跑優先 | 求正確、可維運優先 |
| **主要主線** | 初階 | 初階 → 進階 | 進階 |

跨情境共通：**先量再調（第 02 章）** 是所有「嫌慢」的起點；**冪等與品質（07、08 章）** 是所有「要長期跑」的底線。
