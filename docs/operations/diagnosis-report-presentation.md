# 診斷報表呈現原則

> **目標**：新診斷報表一次就貼近需求，減少來回。立場鐵則（不下結論／不設門檻／每數字自帶範圍說明）管「呈現什麼」；這份管下一層「怎麼編排，一個只有這頁的讀者才真的讀得懂、並靠清楚的邏輯架構自己下判斷」。
>
> 來源：evaluation 診斷層重構期間，與使用者數輪覆盤收斂出的原則。

## 零、最高原則：每段資訊都要扣回這份報表的目的

一份報表為了一個**明確目的**而存在——幫讀者做出某個特定判斷／回答某個特定問題。**每一個區塊、每一個數字、每一張圖、每一句話，都要因為它服務這個目的而存在，而且它與目的的關聯要讓讀者看得見。**

- 放任何東西之前先問：「這對讀者要用這份報表做的事，有什麼幫助？」答不出來就砍掉，或把它的相關性明講出來。
- 一個讀者無法連回自己任務的數字，就是雜訊——即使它正確。
- 這條決定了下面每一節：概覽把「讀者的問題」對應到「哪一區」；區塊順序從「最貼近他要做的決定」排到「最原子」；每一句話都該經得起「這幫他回答目的裡的哪個問題」的檢驗。

## 一、整份報表怎麼編排（冷啟動讀者的閱讀動線）

動線由目的推導：先讓讀者知道報表在幹嘛、往哪找 → 給地基概念 → 由粗到細下鑽 → 交代邊界。順序依據是「他要先知道什麼，才看得懂下一段」。

**1. 概覽（定向）——最前面。** 這份診斷回答什麼問題（一句話）、規模、幾個能一眼判斷「嚴不嚴重」的關鍵數字、以及「想回答什麼問題→看哪一區」的導覽（＝把目的拆成的子問題對應到區塊）。把**嚴重度訊號**與**規模／歸一化分母（非好壞）**分開標——分母常數混進關鍵數字表會被誤讀成嚴重度。

**2. 核心概念（地基）——概覽後、細節前。** 多數診斷都建立在**一個原子量**上，其他數字都是它的加總或衍生。講清楚一次：定義 ＋ 用**一個具體數字**走一遍怎麼算（不只給抽象公式）＋ 它的總量／分母（後面所有比例都除以它）＋「下面每區＝它加總到什麼粒度」的地圖。講清這一個，後面全是「同一個數換個切法」。**不要各區重複這條定義**（會漂移，也是「挖東牆補西牆」的根源）。

**3. 由粗到細的下鑽區——主體。** 從「最貼近讀者要做的決定」排到「最原子」（粗彙總 → 成對／矩陣 → 原始單筆）。每區都是「地基量加總到這個粒度」，並自帶：
- 該區**新出現數字**的定義（貼在表旁；地基量只回指核心概念）；
- 用**最能凸顯重點的形式**呈現資料（見第二節）；
- 一兩句「怎麼讀、最容易誤讀哪裡」；
- 明說這區「看不到什麼」；
- **一句它幫讀者回答目的裡的哪個問題**。

**4. 完整性檢查——永遠最後。** 本次執行的事實（指標參數、總量、抽樣規模）＋「什麼情況會讓數字看起來正常、其實沒量到或不完整」。

**貫穿全篇：數字要能互相接得起來、能自行驗算。** 一區的數字要能追回另一區（某欄＝某矩陣列的最大格；某比例＝某加總 ÷ 開頭的總量），並給可手算核對的路徑。**能手算核對的報表才可信；追不出來的數字，讀者會當它是 bug。**

## 二、呈現形式：選最能凸顯重點的媒介

表格與圖都只是媒介，沒有誰優先——**針對每一段資訊，選最能把它的重點凸顯出來的形式**：要查精確值、逐列比對、按列讀 → 表格；要一眼看形狀、分布、關聯 → 圖。不預設用哪一種，也不要用圖代替讀得出數字的表、或用表埋掉一眼可見的趨勢。

- **形式配合量的結構**：逐列自成一個分布（每列加總＝1）的矩陣，用「按列讀的數字表」，不用全域色階熱圖（誘導跨列比色而誤讀）。
- **排序對齊讀者的任務**：「該優先看哪個」的表按「每單位嚴重度」排，不按「量體加權的總量」排（大宗但其實不差的會頂到最前）；表上明說排序鍵是什麼、代表什麼。
- 要加一個新數字或新排序前，**先算它是不是與既有欄冗餘**（常是既有量的縮放）——冗餘就用既有欄，別加一個讀不出意義的新數。

## 三、定義

- 定義**貼在數字出現的那一區**（不放頁首總表要人滾回去查）；唯一例外＝地基量，開頭講一次、各區回指。
- **寫定義前，拿一個具體數字追一次 code path 驗算**。憑記憶或直覺寫的定義，一問就破。
- 同一個名字在不同粒度出現（單列值 vs 加總值，量級差很多）要在**現場**標清楚是哪一種。

## 四、克制（判斷留給讀者）

- **克制不是少幫忙**——報表的目的就是幫讀者判斷，方式是把角度理清楚攤給他，不是替他選。
- 只呈現數字、對照點、範圍說明。**禁**替讀者詮釋或評級的字：severity／verdict、「該先查誰」、「可能是／不是…造成的」、「偏高／偏低／不足／異常」。
- 導覽（問題→區塊）與「這張表回答什麼」是允許的——那是幫助判斷的邏輯架構，不是下結論。
- 給多個角度（例：總量 vs 每單位嚴重度），明說各是什麼，讓讀者選、不替他選。

## 五、流程

- 產出報表後，**派一個「讀者 subagent」通讀再修**：資料科學背景、對專案有基礎認知、假設報表是他自己產的、與你**零共用 context**。要他挑「哪裡沒定義清楚／邏輯怪／圖難讀／哪裡在替我下判斷」，至少數個具體問題；依反饋修，改完可再派一輪確認。
- 但**讀者 subagent 補不了定義的正確性**——他讀得出「沒說清楚」，讀不出「定義跟程式實際算的對不上」。地基量與關鍵公式要自己拿數字對著 code 驗。
- 有既有版本可參考時，**取其優點、補其缺點，不照抄**：編排值得學，但定義清晰度與圖表選擇不一定最好，逐項自己判斷。

## 六、機械

- 報表文字（bullets／定義／表格）經 HTML escape、**不轉譯 markdown**：不要用 `**粗體**`（會顯示成字面星號）。強調靠句子結構或 ⚠ 符號。
