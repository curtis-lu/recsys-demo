程式行為圖解

這個框架幾件「跟直覺不一樣、踩過才知道」的行為。每節：反直覺點 → 圖 → 為什麼。對照 pipelines/dataset.md、training.md 與 design-principles.md。

1. 抽樣是「決定性 hash」，不是隨機亂數

直覺以為抽樣 = 每次隨機抽。實際是對每一列算 hash、和門檻比大小——同設定永遠抽到同一批。

每列：keep ⟺ crc32( identity | site | seed ) % BUCKETS < ratio · BUCKETS │ │ │ (time,entity,item) │ 固定 seed 抽樣站點名 train → site = "sample_keys" ┐ 同 seed、不同 site calib → site = "calibration_keys" ┘ → 抽到獨立的桶（不重疊）
為什麼：可重現（重跑得到同樣 split）、可增量（ratio 調大只會「多納入」不會重洗）。site 讓 train 與 calibration 即使共用 seed 也不會抽到同一批客戶。（dataset/helpers_spark.py）

2. 前處理在 train 期 fit、對整張表 transform 一次

直覺以為每個 split 各自編碼。實際是「只用 train 期學字典、再套用到整張 feature_table」。

fit（只看 train 期） ──→ preprocessor（編碼字典） │ feature_table（整張） ──transform 一次──→ preprocessed_feature_table │ train / train_dev / val / test / calibration 全部共用這一份
為什麼：(1) 防洩漏——字典不看 val / test / 未來資料；(2) 一致——所有 split 用同一套編碼；(3) 省算——只編一次。所以 preprocessor 屬 base 層、與抽樣無關。（preprocessing/_spark.py）

3. 「丟掉無正例的組」只發生在 val / test

直覺以為過濾對全部 split 一視同仁。實際只有 val / test 丟，train / train_dev / calibration 全留。

split無正例的 query group為什麼
val / test丟掉（filter_groups_with_positives）排序指標（mAP/NDCG）對「整組沒正例」沒有意義（AP 分母為 0、回 None）
train / train_dev / calibration全留pointwise / lambdarank 的 loss 會用到每一列（含負例組）
連動：這正是 metrics.html 裡「zero-positive group 回 None、不計入 mAP」的另一面——評估端本來就會排除，所以資料端先丟掉以省儲存與記憶體。

4. label 是「稀疏」的，join 不到就是負例

直覺以為 label_table 每個 (entity,item) 都有一列。實際 label 只記「有事件」的列，沒對到的在組裝時補 0。

keys（候選母體，dense：每個候選都有） │ LEFT JOIN label_table（sparse：只有發生事件的列） on (time,entity,item) ▼ join 不到 → label = coalesce(label, 0) # 視為負例 │ LEFT JOIN preprocessed_feature_table on (time,entity) ▼ *_model_input
為什麼：候選母體由 sample_pool 界定（dense），label 只需記正事件（sparse）→ 省空間。但要小心：sample_pool 的候選範圍決定了「排名母體」，漏掉候選 = 改了母體。（preprocessing/_spark.py build_model_input）

5. 校準要「兩個開關」都打開

直覺以為一個 flag 控制校準。實際 dataset 端與 training 端各一個，缺一不可。

dataset 端 dataset.enable_calibration: true → 產出 calibration_model_input（校準資料） training 端 training.calibration.enabled: true → calibrate_model 實際做校準（如 sigmoid） 模型輸出兩欄： score（校準後） · score_uncalibrated（原始；關閉校準時 score＝它）
為什麼要校準：LTR 的 score 是排序用相對分、不是機率；即使 binary 目標，原始輸出也未必是校準過的機率。只在要把 score 當機率解讀（算期望值、跨期比較）時才需要。純看排名不用。（README §3 Q4、training.md）

6. 改設定 bust 哪層版本 ≠ 直覺

直覺以為「改任何設定都要整批重算」。實際依設定屬哪層，只重算那層之後。

改什麼？ ├─ schema / 前處理 / carry_columns / feature_table 欄位 → bust base_dataset_version → 幾乎全部重算 ├─ train 抽樣（sample_ratio / overrides / group_keys / train_dev_ratio）→ bust train_variant_id → 只 train 系列 + 之後 ├─ calibration 抽樣 → bust calibration_variant_id ├─ objective / 其他 model-defining training 設定 → bust model_version（不動 dataset） └─ 純 logging / threading 的 algorithm_params 鍵 → 不翻任何版本
最容易繞錯：只想調 sample_weights 的值（維度欄已 carry）→ 不 bust dataset、只 bust model_version；但若新權重維度欄還沒 carry → 得先加 carry_columns（這就 bust base 了！）。逐情境 SOP 見 change-guide.md。
